# 0016: Stop sending orderLinkId to Bybit (match bbu2 approach)

## Context

The bot generates deterministic `orderLinkId` values (SHA256 of symbol+side+price+direction) and sends them to Bybit with every order. Bybit remembers used `orderLinkId`s even after orders are cancelled/filled, causing error 110072 ("OrderLinkedID is duplicate") on every restart. This creates an infinite retry loop.

bbu2 does not send `orderLinkId` to Bybit at all. Instead it relies on:
- `_is_good_to_place()` for dedup (exact match check + reduce-only qty check)
- Bybit's `orderId` for order tracking and cancellation

## Changes

### 1. Remove `order_link_id` from executor's `place_order` call
**File:** `apps/gridbot/src/gridbot/executor.py:178`

Remove the `order_link_id=intent.client_order_id` parameter from `self._client.place_order()`. The `client_order_id` continues to exist on `PlaceLimitIntent` for internal in-session dedup (`_tracked_orders` dict in runner), it just won't be sent to Bybit.

### 2. Add exact-duplicate check to `_is_good_to_place`
**File:** `apps/gridbot/src/gridbot/runner.py`, method `_is_good_to_place()`

bbu2's `_is_good_to_place` (bybit_api_usdt.py:296-300) has an exact duplicate check before the reduce-only qty check: if an order with the same price, qty, side, and reduceOnly already exists in placed orders, return False. Our current implementation only has the reduce-only qty check. Add the exact-duplicate check to match bbu2.

Check against `_tracked_orders` where `status == "placed"`:

### 4. Update `inject_open_orders` to not depend on `orderLinkId`
**File:** `apps/gridbot/src/gridbot/runner.py`, method `inject_open_orders()` (line ~688)

Currently skips orders without `orderLinkId`. Since new orders won't have one, reconciled orders from the exchange may still have it (from old sessions) or may not. Ensure orders are tracked by `orderId` as primary key instead.
