# 0018 — Instrument WebSocket reconnect elapsed time

## Description

Add lightweight timing instrumentation around `disconnect()`/`connect()` pairs
in `Orchestrator._health_check_once`, mirroring the existing REST-fetch
instrumentation (`_POSITION_FETCH_SLOW_THRESHOLD`). Purpose: collect empirical
data on how long WS reconnects take in practice and surface cases where they
block the main polling loop. **No behaviour change — only logs.** This provides
evidence for a future decision on whether a harder fix (background reconnect,
pybit-level timeouts) is warranted.

---

## Motivation

Review report flagged that `ws.disconnect()` / `ws.connect()` in
`_health_check_once` can block the main polling loop, because pybit uses
`websocket-client` and TCP/TLS handshake has no explicit per-op timeout. On a
degraded network a reconnect could hang for tens of seconds before OS-level
TCP timeouts kick in.

The report's proposed fix (wrap in `signal.alarm(5)` or thread-based timeout)
is rejected — signals are unsafe in a multi-threaded Python app with pybit's
background threads, and thread-based cancellation leaves dangling workers.

Before committing to a real fix (pybit timeout plumbing or fully-async
reconnect), we need evidence that the problem actually happens in production
and at what magnitude. Lightweight timing + threshold-warning is zero-risk
and will surface real incidents.

## Reference pattern (to mirror exactly)

The existing REST-fetch instrumentation at `orchestrator.py:980–987`:

```
finally:
    elapsed = time.monotonic() - start
    if elapsed > _POSITION_FETCH_SLOW_THRESHOLD:
        logger.warning(
            "Position fetch for %s took %.1fs (threshold=%.1fs) — "
            "blocking REST stalled the main polling loop",
            account_name, elapsed, _POSITION_FETCH_SLOW_THRESHOLD,
        )
```

WS instrumentation uses the identical shape, swapping:
- `start` → `reconnect_start`
- `_POSITION_FETCH_SLOW_THRESHOLD` → `_WS_RECONNECT_SLOW_THRESHOLD`
- Message → `"Public WS reconnect for %s took %.1fs (threshold=%.1fs) — blocking main polling loop"` (and `"Private WS reconnect …"` for the private branch)

---

## Algorithm (per WS, public and private symmetric)

For each account inside `_health_check_once`:

1. Read `pub_ws = self._public_ws.get(account_name)`.
2. If `pub_ws and not pub_ws.is_connected()`:
   a. Fire existing disconnect alert via `self._notifier.alert(...)`.
   b. `reconnect_start = time.monotonic()`.
   c. Enter inner try/except:
      - `pub_ws.disconnect(); pub_ws.connect()`; on success,
        `logger.info(f"Public WS reconnected for {account_name}")`.
      - On `Exception`: existing `self._notifier.alert_exception(...)`.
   d. `finally`:
      - `elapsed = time.monotonic() - reconnect_start`.
      - If `elapsed > _WS_RECONNECT_SLOW_THRESHOLD`: emit the warning (format
        per "Reference pattern" above).
3. Repeat steps 1–2 for `priv_ws = self._private_ws.get(account_name)` with
   the private-WS message variant.

The outer `try/except` around `_health_check_once` (current line 996 → 1050)
is unchanged.

## Threshold rationale and escalation trigger

**Starting value:** `_WS_RECONNECT_SLOW_THRESHOLD = 5.0`. Typical reconnect
is ~1–2 s (TCP handshake + TLS + pybit subscribe-ack). 5 s is a meaningful
excess without being extreme and won't spam on normal network jitter.

**Decision framework (so we don't accumulate logs forever without acting):**
after at least **7 days of steady-state live operation**, escalate to a real
fix (feature 0019: pybit timeout plumbing or background reconnect) if **any**
of the following holds:

| Signal | Threshold |
| --- | --- |
| Count of `"WS reconnect … took"` warnings per account | > 20 / week |
| Single observed elapsed | > 15 s (any occurrence) |
| Warnings clustered within a short window | > 3 within 60 s (suggests flap) |

If none of these trigger, the instrumentation stays and the constant may be
tightened (e.g. 3 s) to collect finer data. If the problem never surfaces, no
further action needed — the existing behaviour is acceptable.

Count-based triggers are chosen over p95/percentiles because we do not assume
a metric pipeline; the thresholds can be checked with `grep` + `wc -l` on
production logs.

## Operational workflow

### Prerequisite: run with JSON log file

Logging setup lives in `apps/gridbot/src/gridbot/main.py:27–69`. Console
(stdout) is always on at INFO; a JSON-per-line file handler is enabled only
when `--log-file=<path>` is passed (DEBUG level). **For the data-collection
phase this flag is mandatory** — without it, warnings vanish on every
restart and the whole exercise is pointless. Deploy instruction: launch the
bot with `--log-file=logs/gridbot.json` (or the project's standard log path).

### Review cadence

| Moment | Action |
| --- | --- |
| T+0 (post-deploy) | Synthetic smoke in test env: force-disconnect a WS (or call `pub_ws.disconnect()` manually via a debug hook) and confirm the WARNING appears in the JSON log file. Unit tests already cover the code path — this check is just to verify logging plumbing end-to-end. |
| T+7 days | First production checkpoint. Run the queries below; compare counts / max / cluster density to the Escalation trigger table. |
| T+14, T+30 days | Repeat checkpoint if T+7 did not trigger. |
| > 30 days clean | Leave instrumentation in place as a permanent regression guard; optionally tighten threshold to 3 s to collect finer data. |

### Review procedure (grep + jq)

```bash
# Every WS-reconnect warning with timestamp
jq -r 'select(.message | test("WS reconnect for .* took")) | [.timestamp, .message] | @tsv' \
    logs/gridbot.json > ws_reconnects.tsv

# (a) Count over the period:
wc -l ws_reconnects.tsv

# (b) Max observed elapsed:
grep -oE 'took [0-9.]+s' ws_reconnects.tsv | sort -rn | head -5

# (c) Cluster density (>3 in any 60s window):
cut -f1 ws_reconnects.tsv | sort | \
    awk -F'[T:.]' '{print $1, $2, $3, $4}' | uniq -c | awk '$1 > 3'
```

Compare to the Escalation trigger table (above). Any row tripped → create
`0019_PLAN.md`.

### Decision matrix for 0019 (if triggered)

Pick approach based on what the logs show:

| Log signal | Likely root cause | Approach |
| --- | --- | --- |
| Elapsed consistently 10–15 s, narrow variance | OS-level TCP/TLS handshake timeout | **Option A** — plumb `timeout=` through pybit's `WebSocket(...)` init in `packages/bybit_adapter/src/bybit_adapter/ws_client.py:113, 342`. Cheaper, bounded-risk. |
| Wide elapsed variance, clusters, occasional > 30 s | pybit/internals blocking, possibly lock contention | **Option B** — background reconnect: health check only flags WS as needing reconnect; per-WS worker thread performs disconnect/connect with an in-flight lock to prevent parallel attempts. Higher complexity but addresses real blocking. |

### Lifecycle after 0019 ships (if it does)

- **Keep the instrumentation.** It becomes a permanent regression guard — if
  a future refactor reintroduces blocking, warnings will surface again.
- **Cost is negligible** (one `time.monotonic()` call + compare per health
  check), and keeping it avoids the cycle of "remove instrumentation → miss
  regression → re-add instrumentation".

## Non-goals / out-of-scope

- **No behaviour change.** Reconnect still blocks the main loop; this PR is
  diagnosis, not the fix.
- **No pybit-level timeout plumbing.** Deferred until logs show the problem
  is real.
- **No background-thread reconnect.** Deferred until logs show the problem
  is real.
- **No change to public WS client code** (`packages/bybit_adapter/**`).
- **No dedupe/throttle on the new warning, by design.** Network flap could
  cause clustered warnings in logs, but for a *diagnostic* PR that cluster
  is exactly the signal we want to capture (see "clustered warnings" row in
  Escalation trigger). Telegram-side noise is already suppressed by the
  existing `self._notifier.alert(..., error_key=...)` throttle, so clustered
  `logger.warning` lines land only in file/stdout. If log volume becomes a
  real operational problem, dedupe is a follow-up concern — not this PR.
