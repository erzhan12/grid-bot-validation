# Feature 0083 — Execution-identity idempotency guard (Issue #202)

## Context

A private-execution WebSocket resync/redelivery replays the **same**
`ExecutionEvent` many times in one tight burst. Each redelivery reaches
`StrategyRunner.on_execution` (`apps/gridbot/src/gridbot/runner.py:756`),
is **not** dropped by the Feature 0031 phantom filter, and produces one
`tracked.mark_filled()` + one `"Order filled:"` INFO log + one
`self._engine.on_event(event)` call (runner.py:817-825).

Evidence (mainnet_live, 2026-06-30 18:15:59): 48 identical
`solusdt_test: Order filled: SOLUSDT Sell qty=0.1 price=72.46` in one
second, SOLUSDT position flat throughout. 24h aggregate fills=5512 /
placements=466 (~11:1). gridbot-health Table 4 showed SAME ORDER=0 and
`dropping phantom ExecutionEvent`=0 in the window — the 0031 filter did
not engage.

**Root cause.** The Feature 0031 SAME-ORDER detector fires only on a pair
of **different** exchange `order_id`s arriving close in time
(`_check_same_orders`, runner.py:798; `_same_order_dedup_cache`,
runner.py:483). A single-`order_id` redelivery burst is not a
different-oid pair, so the detector never fires and
`_drop_phantom_event_for_current_call` stays `False`. There is **no
execution-identity idempotency guard** (dedup by `exec_id`) upstream of
`mark_filled` / log / `on_event`.

**Fix.** Add a bounded **FIFO cache** of processed `exec_id`s and
early-return in `on_execution` on a repeat, **independent of** and
**complementary to** the 0031 SAME-ORDER pair logic (different-oid
pairs). Economically benign (position unchanged, `cum_realized` is
exchange-sourced via `cumRealisedPnl`); the guard removes the
fill-counter inflation, the log spam, and the redundant
`engine.on_event` re-derivation. The guard is placed at the **top** of
`on_execution` (before `_check_same_orders`) so a redelivery burst also
cannot corrupt the SAME-ORDER detector's buffers or clear a latched
error (see §Ordering rationale).

## Design decisions (from review findings P1/P2)

- **No TTL, no clock — FIFO cap only (P1).** A Bybit `exec_id` is a
  globally unique execution identity: a genuine distinct fill *never*
  reuses an `exec_id`. Therefore there is **no correctness reason to
  ever expire an entry by time** — expiring would let a late WS replay
  (arriving after the TTL) be reprocessed, reintroducing exactly the bug
  this feature fixes. Memory is bounded instead by a FIFO entry cap. This
  removes the injectable-clock dependency entirely (supersedes the
  earlier clock-idiom discussion; `self._clock()` is **not** used by this
  feature).
- **Guard runs before `_check_same_orders` (P2).** See §Ordering
  rationale — this is load-bearing, not a placement preference.

## Open questions — resolved by code research

**Q1 — Is `engine.on_event` idempotent for a redelivered fill of an
already-`mark_filled` order?** Partially. `mark_filled()`
(runner.py:147) is idempotent (`self.status = "filled"` unconditionally).
But `GridEngine._handle_execution_event`
(`packages/gridcore/src/gridcore/engine.py:196`) is **not** a no-op: it
unconditionally sets `self.last_filled_price = float(event.price)` and,
when `last_close is not None`, calls `self.grid.update_grid(...)`
(engine.py:213-217). A redelivered fill re-runs `update_grid` with the
same price. Position stays flat (the redelivered price equals the
already-applied price, so grid re-derivation converges to the same slot
state) — this is why the exchange position never moved — but the
re-invocation is wasteful and semantically wrong (the fill already
happened once). Guarding upstream in `on_execution` removes the
redundant re-derivation entirely. **No engine change is needed.**

**Q2 — Do redelivered events leak into other per-fill side-effects
without idempotency (event_saver, recorder, GridStateWriter)?**
- **event_saver / recorder DB path is already idempotent.** The
  `PrivateExecutionRepository` upsert uses
  `on_conflict_do_nothing(index_elements=["exec_id"])`
  (`shared/db/src/grid_db/repositories/execution.py:130-133`) and tracks
  `duplicates_skipped`. Redelivered `exec_id`s are dropped at insert.
  This is a **separate process** from gridbot (recording is not a gridbot
  flag) — no change here.
- **GridStateWriter** is fingerprint-deduped: `save()` short-circuits when
  the `(side, price)` tuple + grid_step + grid_count fingerprint equals
  the last-enqueued payload (RULES.md §Grid State). The upstream guard
  makes this moot regardless. No change here.

**Q3 — Should the analyzer's fills counter dedup by exec-identity?** No
change required. The analyzer (`.claude/skills/gridbot-health/analyze.py`)
counts fills purely by matching `"Order filled"` log substrings
(analyze.py:1098, 1720; `_activity` / Table 2 / Table 14). Because the
guard early-returns **before** the `"Order filled:"` INFO emit,
redelivered fills never produce the log line, so the analyzer counter is
correct at source. No `exec_id` dedup is added to the analyzer (it has no
access to `exec_id` in the log line anyway).

## Files & functions to change

### 1. `apps/gridbot/src/gridbot/runner.py` — the guard (primary change)

- **New module-private constant** near the existing 0031 constants
  (runner.py:83-104), following the `_UPPER_SNAKE` convention:
  - `_EXEC_DEDUP_MAX_ENTRIES` — hard cap on cache size; evict oldest
    (FIFO) on overflow. This is the **only** bound (no TTL). Size it so
    the retained horizon, expressed in *count of distinct fills*, far
    exceeds any plausible WS reconnect/resync replay horizon. At the
    observed rate (~5512 fills / 24 h ≈ 230/h), a cap of `4096` retains
    ≈ 18 h of distinct `exec_id`s — orders of magnitude beyond the
    seconds-to-minutes window in which Bybit replays executions after a
    reconnect. State the chosen value verbatim in the implementation and
    note it may be raised if fill rate grows (memory is ~40 bytes/entry;
    4096 ≈ 160 KB per strat).

- **New instance state** in `__init__` (near runner.py:483 where
  `_same_order_dedup_cache` is declared):
  - `self._processed_exec_ids: "OrderedDict[str, None]" = OrderedDict()`
    — an insertion-ordered set of processed `exec_id`s (value unused).
    Use `collections.OrderedDict` (add `from collections import
    OrderedDict` alongside the existing `from collections import deque`,
    runner.py:14). OrderedDict gives O(1) FIFO eviction via
    `popitem(last=False)`.

- **New private method** `_seen_exec_id(self, exec_id: str) -> bool`
  (place near `_check_same_orders`, ~runner.py:2700+). Algorithm — no
  clock, no expiry:
  1. If `exec_id` is falsy (`""`) → return `False` (cannot dedup a
     missing identity; fall through to normal processing — matches how
     `exec_id=exec_data.get("execId", "")` defaults empty in the
     normalizer, `packages/bybit_adapter/src/bybit_adapter/normalizer.py:244`).
     Empty `exec_id` is not expected on real live fills but must not
     crash or wrongly dedup.
  2. If `exec_id in self._processed_exec_ids` → **repeat**. Return
     `True`. Do **not** move the entry to the end / refresh it — pure
     insertion-order FIFO keeps eviction deterministic.
  3. Otherwise record `self._processed_exec_ids[exec_id] = None`; if
     `len(...) > _EXEC_DEDUP_MAX_ENTRIES`, `popitem(last=False)` until
     within the cap. Return `False`.

- **Wire the early-return at the TOP of `on_execution`**
  (runner.py:756-835). Placement: **immediately after** the per-event
  phantom-drop-flag reset (`self._drop_phantom_event_for_current_call =
  False`, runner.py:777) and **before** the tracked-order lookup
  (runner.py:787), `_check_same_orders` (runner.py:798), the phantom-drop
  gate (runner.py:808), and the `mark_filled` + `"Order filled:"` INFO +
  `engine.on_event` block (runner.py:817-825):

  ```
  if self._seen_exec_id(event.exec_id):
      logger.debug( "<strat>: dropping redelivered ExecutionEvent "
                    "exec_id=... order_id=... (WS resync)" )
      return []
  ```

  (Illustrative placement only — no code in the plan.)

#### Ordering rationale (critical — this is the P2 fix)

The guard sits **upstream of `_check_same_orders`** (runner.py:798), not
between the phantom gate and `mark_filled`. This is load-bearing:

- **A full-fill redelivery must not reach `_check_same_orders`.** That
  method, on every `leaves_qty == 0` execution, `appendleft`s the record
  into the `maxlen=2` `_recent_executions_{long,short}` buffers
  (runner.py:2778) and **resets `self._same_order_error = False`**
  (runner.py:2790-2791) before re-evaluating. A redelivery burst would
  (a) evict the genuine different-oid pair out of the maxlen=2 buffer and
  (b) reset the latched flag, and re-evaluation of the redelivered record
  (same `order_id`, no different-oid pair) would **not** re-establish it —
  spuriously **clearing an already-latched SAME-ORDER error** and firing a
  false recovery-INFO via the `was_set and not _same_order_error` branch
  (runner.py:2796-2797). Dropping redeliveries **before** line 798 leaves
  the buffers and the flag untouched. This is a pre-existing latent bug
  that the guard's placement now also closes.

- **Composition with 0031 SAME-ORDER detection is preserved.** Both legs
  of a genuine different-oid pair have **distinct** `exec_id`s and are
  first-sightings, so both pass the guard (`_seen_exec_id` returns `False`
  and records them) and reach `_check_same_orders`. Only exact
  redeliveries (2nd..N-th sighting of one `exec_id`) are dropped, and
  those carry **no new** SAME-ORDER signal — the first occurrence already
  fed the buffer.

- **Composition with the 0031 phantom-drop gate is preserved.** A
  WS_GLITCH phantom leg is a first-occurrence: it passes the guard
  (recording its `exec_id`) and is then dropped by the phantom gate
  (runner.py:808). Recording the phantom's `exec_id` is harmless — a
  redelivery of that exact phantom `exec_id` *should* be dropped anyway.
  A later **genuine** fill for that resting order arrives with a **new**
  `exec_id` → passes the guard → processes normally.

- **The guard is upstream of `mark_filled`, the `"Order filled:"` log,
  and `engine.on_event`** — this removes the counter inflation (Table
  2/14), the log spam, and the redundant grid re-derivation (Q1) in one
  place.

- **`_recent_executions_*` buffers (runner.py:466-467):** with the guard
  at the top, redeliveries never reach `_check_same_orders`, so they
  never touch these buffers. First occurrences append normally. No change
  to `_check_same_orders` or `_check_same_orders_side` is needed.

- **`exec_id` re-exposure:** none needed. `ExecutionEvent.exec_id`
  already exists (`packages/gridcore/src/gridcore/events.py:89`) and is
  populated by the adapter normalizer from `execId` (normalizer.py:244).
  No dataclass or adapter change.

- **Partial fills (N3):** `_check_same_orders` skips partial fills
  (`leaves_qty != 0`, runner.py:2743), but the `exec_id` guard runs
  **before** that filter and applies to every execution, partial or
  final. This is correct: a redelivered *partial* fill shares the same
  `exec_id` as its original and is deduped; a genuine partial-then-final
  sequence carries **distinct** `exec_id`s and both process (covered by
  `test_distinct_exec_ids_not_deduped`). No `leaves_qty` special-casing
  in the guard.

- **Record-before-complete note (N1):** `_seen_exec_id` records on first
  sight (step 3) **before** `mark_filled` / `engine.on_event` run. If
  processing were to throw *after* recording, the event is dropped by the
  orchestrator (`on_execution` re-raises, no retry queue for executions),
  and a subsequent WS redelivery of the same `exec_id` would be deduped
  rather than reprocessed. For the reported bug — a **successful** first
  delivery followed by burst redeliveries — this is exactly correct. Add
  a one-line implementation comment stating the record-first ordering is
  intentional; it is not a plan blocker.

## Scope (N5)

Live `StrategyRunner.on_execution` only. Backtest and replay use
different fill paths (deterministic event replay, no live WS resync), so
they cannot produce the redelivery burst this guard targets. No change to
backtest/replay unless WS-redelivery is later observed elsewhere.

## Concurrency / threading

No background thread and no clock. Eviction is O(1) on insert
(`popitem(last=False)` when over `_EXEC_DEDUP_MAX_ENTRIES`), honoring the
project rule "no bg threads for TTL". `on_execution` runs on the single
main event loop (dispatched from `orchestrator._on_execution` →
`runner.on_execution`); the cache needs no lock.

## Error-handling discipline

`on_execution` already wraps its body in `try/except` that logs ERROR
with `exc_info=True` and **re-raises** (runner.py:833-835). The new guard
sits inside that body, so it inherits the re-raise discipline. The guard
itself raises nothing (pure OrderedDict ops); it must not swallow.

## Reproducing test(s) — TDD, add first (RED)

New tests in `apps/gridbot/tests/test_runner.py` (existing home for
`on_execution` / SAME-ORDER coverage). No injected clock is needed (the
guard has no time dependency).

1. **`test_redelivered_execution_deduped_by_exec_id`** (the reproducing
   test): build a runner with a tracked resting order; feed the **same**
   `ExecutionEvent` (identical `exec_id`, `order_id`, `price`, `qty`) N
   times (e.g. 5) via `on_execution`. Assert:
   - `mark_filled` runs / `"Order filled:"` INFO is emitted **exactly
     once** (assert via `caplog` count of the `"Order filled:"` substring
     == 1).
   - `engine.on_event` is invoked exactly once for the fill (spy/mock the
     engine or assert `last_filled_price` set once / grid unchanged after
     first).
   - occurrences 2..N return `[]`.
   - `_same_order_error` stays `False` (redelivery burst raises no
     spurious SAME-ORDER latch — and, per test 6, does not clear one).

2. **`test_distinct_exec_ids_not_deduped`**: two events, same `order_id`
   + `price` but **different `exec_id`** (a real partial-then-final or two
   genuine fills) → both process (two `"Order filled:"` lines, engine
   called twice). Guards against over-dedup; also covers the partial-fill
   case (N3).

3. **`test_exec_dedup_fifo_eviction`** (replaces the earlier TTL-expiry
   test; incorporates N2): feed `_EXEC_DEDUP_MAX_ENTRIES + 1` **distinct**
   `exec_id`s, assert the cache stays capped at `_EXEC_DEDUP_MAX_ENTRIES`
   and the **oldest** `exec_id` was evicted — re-feeding that oldest
   `exec_id` now **processes** (returns non-`[]` / emits a fill) rather
   than being deduped, while a still-resident `exec_id` re-fed is still
   deduped. Asserts eviction mechanics, not time-based duplicate
   acceptance. (For a fast test, the implementation may allow the cap to
   be constructor-injectable, or the test may patch
   `_EXEC_DEDUP_MAX_ENTRIES` to a small value.)

4. **`test_empty_exec_id_not_deduped`**: two events with `exec_id=""` →
   both process (missing identity is never deduped; must not crash).

5. **`test_exec_dedup_composes_with_0031_phantom_drop`** (regression):
   a WS_GLITCH_SUSPECTED phantom (different-oid pair, REST-adjudicated) is
   still dropped by the 0031 gate; the phantom leg's first-occurrence
   `exec_id` is recorded by the guard (harmless), and a later **genuine**
   fill for the resting order — carrying a **new** `exec_id` — still
   processes. Confirms the two mechanisms compose and neither masks the
   other.

6. **`test_redelivery_does_not_clear_latched_same_order_error`** (P2
   regression, the key placement test): latch `_same_order_error = True`
   via a genuine different-oid SAME-ORDER pair (two full fills, different
   `order_id`, same price, within `_SAME_ORDER_TIME_WINDOW_SEC`). Then
   feed a **full-fill redelivery burst** (repeat one of those executions,
   same `exec_id`, several times). Assert:
   - `_same_order_error` stays **`True`** (the redelivery is dropped by
     the guard before `_check_same_orders`, so the latch and the maxlen=2
     buffers are untouched).
   - **no** recovery-INFO / `reset_same_order_error` fires from the
     redelivery burst.
   If this test were run against the rejected after-`_check_same_orders`
   placement, it would FAIL — which is the whole reason the guard sits at
   the top.

## RULES.md note (Step 6)

Add a short bullet under the **SAME ORDER detection** section (RULES.md

redelivery bursts are deduped by `_processed_exec_ids` (bounded FIFO cache,

path is independently `exec_id`-idempotent via `on_conflict_do_nothing`, so
no writer change was needed.
