# Feature 0043 — Hedge credit in backtest liq_price for UTA hedge-mode

## Goal

Feature 0034 surfaced a buy-side `liq_price` divergence: live reports
`liqPrice = 0` for hedged longs, backtest computes `bt_liq ≈ 9 USDT`.
Feature 0042 corrected the balance input (`totalAvailableBalance`)
which fixed the short-side delta but left buy-side untouched, because
`BacktestRunner._estimate_liquidation_price` treats each direction in
isolation — it ignores the opposite leg.

Bybit's UTA hedge mode applies a **hedge credit**: when both long and
short are open the opposing exposures offset each other and Bybit
reports `liqPrice = 0` (or near-zero) for the side that is covered by
the opposite leg. Feature 0043 makes our liq formula hedge-aware so
the snapshot CSV converges with live within ≤ 1 USDT for both sides.

## Confirmed Context

- `BacktestRunner._estimate_liquidation_price`
  (`apps/backtest/src/backtest/runner.py:685-733`) is per-direction and
  consumes only `(entry_price, direction, position_value,
  wallet_balance)`. No opposite-leg awareness.
- Three call sites pass per-direction state:
  - `runner.py:527` — `_process_fill` (liq_ratio log).
  - `runner.py:612` — `_emit_position_snapshot` (parity output).
  - `runner.py:669` — `_build_position_state` (risk multipliers).
- `Position.create_linked_pair()` already wires long ↔ short on the
  gridcore side (`packages/gridcore/src/gridcore/position.py:117-128,
  431-459`), so an `opposite_state: PositionState` parameter has a
  natural source at every call site.
- 0034 production validation CSV is no longer on disk; fresh recorder
  data is required to derive and validate the formula.

## Non-Goals

- Do not touch IM / MM math (`calc_initial_margin`,
  `calc_maintenance_margin`). Hedge credit is a balance-side concern,
  not a margin-tier change.
- Do not refactor `calculate_amount_multiplier` — its hedge logic is
  correct and shipping.
- Do not preemptively model isolated margin. Gridbot runs UTA cross.
- Do not commit, push, or open a PR without explicit operator
  instruction.

## Phased Plan

### Phase 1 — Data collection (operator-driven, blocks Phase 2)

1. Run recorder on mainnet against a strategy that exercises **all
   four hedge configurations** in a single recorder run:
   - Long-only (short size = 0).
   - Short-only (long size = 0).
   - Net-long (long > short, both > 0).
   - Fully hedged (long ≈ short, both > 0).
2. Acceptance for Phase 1: `position_snapshots` table contains at
   least one row per configuration, with `liq_price` populated from
   live REST (already present via 0034 telemetry writer).
3. Operator dumps a CSV via the comparator's pair-and-compare flow
   so we have aligned `(long_size, long_entry, long_liq_live,
   short_size, short_entry, short_liq_live, total_available_balance,
   mark_price)` rows.

### Phase 2 — Formula derivation (analysis)

#### Method

Two recorder runs on mainnet LTCUSDT (run_ids `2fa15fa2` and
`e0723754`), `scripts/_phase2_formula_check.py`. Total **13 distinct
paired (L, S) states** with non-NULL `liqPrice` from Bybit covering both
directions: 8 net_short and 5 net_long. Quantitative net_long points
were obtained by manual long-side intervention to push `q_net > 1.78`
(threshold above which net_long produces a non-NULL Bybit liq at the
gridbot's current scale).

All comparisons use synchronous data: Buy snapshot and wallet snapshot
within ≤5 ms of each other (Bybit emits both in the same WS burst on
fill events). Sell snapshot can be older — its state is forward-stable
when there are no Sell-side executions, which is exactly Bybit's
internal view at the moment it computes the live liqPrice.

#### Final formula

```
q_net = long_size - short_size
mm_total = calc_maintenance_margin(
    L_position_value + S_position_value,
    symbol, tiers,
)                                        # full tier-MMR on summed notional
if q_net > 0:                            # net long
    liq_long  = max(0, L_entry - (total_equity - mm_total) / q_net)
    liq_short = 0
elif q_net < 0:                          # net short
    liq_short = S_entry + (total_equity - mm_total) / abs(q_net)
    liq_long  = 0
else:                                    # fully hedged
    liq_long = liq_short = 0
```

Two non-obvious choices in the formula, derived empirically:

1. **Balance input is `total_equity`, not `total_available_balance`.**
   `net_eq` tracks live; `net_avail` is off by 30-45 USDT consistently.
   This contradicts the natural reading of 0042 (which plumbs
   `total_available_balance` into `BacktestSession.current_balance`),
   so 0043 introduces a separate `total_equity` plumb path and leaves
   `current_balance` semantics intact for executor / qty / risk-mult.

2. **MM term uses full tier-MMR on combined notional, NOT the sum of
   Bybit's published `positionMM` per leg.** Bybit publishes
   `positionMM` for the smaller (hedged) leg with a hedge-mode discount
   (e.g. for a Sell of `2.5 @ 57.68` the published MM is 0.174 instead
   of 0.721 = 2.5 × 57.68 × 0.005). Internally for liq computation
   Bybit reverts to the non-discounted full mmr on both legs. Using
   summed published MM gives 4× larger residual delta than full-mmr.

#### Residual bias

After full-mmr correction the bias signs split by direction:

- **net_short**: +0.36 to +0.60 USDT (formula slightly above live).
- **net_long**: -0.04 to -0.17 USDT (formula slightly below live).

Likely sources:

- Funding / fee accrual not reflected in `total_equity` snapshot —
  Bybit may use `breakEvenPrice` (entry-adjusted for accumulated
  funding) instead of raw `entryPrice` in its internal calc; the
  asymmetric sign matches an unrealised-funding effect.
- Tier boundary deduction near the liq price (small for tier 1
  LTCUSDT).

All 13 quantitative points pass acceptance ≤1 USDT with margin. Max
|Δ| = 0.60 USDT << 1.0 acceptance bound. Refinement of the residual is
deferred (Open Decisions).

#### Timing-not-an-artifact verification

Repeated the L=4.3 comparison with:

- Wallet @ 4 ms after Buy (synchronous): equity = 105.40120609.
- Wallet @ 2 s before Buy (stale): equity = 105.40334825.

Resulting Δ identical to 3 decimal places (-0.481 vs -0.482). The
~0.5 USDT residual is **structural, not a timing/staleness artifact**.

### Phase 3 — Implementation

Phase 2 produced three implementation-affecting facts:

- Liq is pair-shaped (single input pair → both legs' liq).
- Balance input is `total_equity`, not `total_available_balance`.
- MM term is `calc_maintenance_margin(L_pv + S_pv, symbol, tiers)` —
  full tier-MMR on combined notional, NOT the sum of per-leg
  `position_mm` (which carries Bybit's hedge-mode discount on the
  smaller leg).

Refactor:

1. **Replace `_estimate_liquidation_price` per-leg with a pair
   function** (`apps/backtest/src/backtest/runner.py:685-733`):
   ```
   _estimate_pair_liq_prices(
       long_state: PositionState,
       short_state: PositionState,
       total_equity: Decimal,
   ) -> tuple[Decimal, Decimal]   # (liq_long, liq_short)
   ```
   Body matches the formula in Phase 2 verbatim. Reuses
   `calc_maintenance_margin` from `gridcore.pnl` with `self._mm_tiers`.
   Pair-aware is the only mode post-0043; no opposite-state optional
   parameter to footgun on.
2. **Plumb `total_equity` from `WalletSeed` to the runner**. 0042 lands
   `total_equity` in `WalletSnapshot` and `WalletSeed` but only
   `total_available_balance` forwards to `BacktestSession.initial_balance`.
   Add `BacktestSession.total_equity` (or surface via existing
   `session.equity_value` which tracks `current_balance + unrealized_pnl`
   — same semantic as live `totalEquity`). Keep `current_balance`
   plumbing untouched: executor margin gating, qty calculator, and
   risk multipliers still consume `total_available_balance`.
3. **Three call sites** at `runner.py:527, 612, 669` migrate from
   per-leg to the pair result. Each site has both trackers in scope:
   - `_process_fill` (line 527, liq_ratio log) — show the just-mutated
     direction's liq.
   - `_emit_position_snapshot` (line 612, parity output) — emit both
     legs' liq into the snapshot for the just-mutated direction.
   - `_build_position_state` (line 669, risk multipliers) — single
     liq formula for the codebase.
4. **Safe caps**. Keep `max(liq, 0)` for the dominant leg's edge case
   (formula returns negative when equity covers position fully — Bybit
   returns 0/NULL in the same case). The over-hedged leg returns 0 by
   construction.
5. **Update docstring + `RULES.md`**: point at this PLAN, document the
   three non-obvious choices (pair shape, `total_equity` not
   `total_available_balance`, full tier-MMR not summed `positionMM`) so
   future readers do not re-discover them via debugging.

## Open Decisions

- Whether to gate hedge credit behind a config flag for
  backwards-compat of pre-0043 backtest configs. Default lean: no
  flag — hedge mode is the only mode gridbot uses.
- Whether to chase the residual ±0.6 USDT direction-asymmetric bias
  (net_long: -0.04..-0.17, net_short: +0.36..+0.60). Suspected source
  is funding/fee accrual in `breakEvenPrice` vs raw `entryPrice`;
  swapping `entry_price` for `break_even_price` in the formula would
  likely close it. Requires capturing `breakEvenPrice` in
  `PositionSnapshot` first. Default lean: ship full-mmr formula,
  revisit only if 0034 comparator still shows > 1 USDT max abs delta
  on a multi-config validation set.
