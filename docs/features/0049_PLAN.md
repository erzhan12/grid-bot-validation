# Feature 0049 - Surgical recorder wipe for shared Phase 4 DB

Feature 0049 implements issue
[#113](https://github.com/erzhan12/grid-bot-validation/issues/113). The issue
title still says `feat(0048): surgical wipe in start_recorder.sh - preserve
gridbot data in shared DB`, but `0048` is already used by the grid recenter
feature; keep this work under `0049` in plan, PR, and runbook references.

Feature 0047 made `grid_state_snapshots` part of the Phase 4 replay seed path:
live gridbot writes the snapshots and replay consumes them. That only works
reliably when gridbot and recorder share one SQLite DB. Today the tooling
contradicts that architecture: gridbot usually reads `DATABASE_URL` from
`.env`, the recorder config points at `data/recorder_ltcusdt_phase4.db`, and
`scripts/phase4/start_recorder.sh` deletes the whole DB file on each start.
When both processes are aligned to the same DB, that file wipe removes
gridbot-owned history, including `grid_state_snapshots`, live `runs`, and
shared setup rows such as `bybit_accounts` and `strategies`.

The change is intentionally narrow: document shared DB as the Phase 4 default,
replace the recorder-start file deletion with a table-level wipe of only
recorder-owned data, and update Phase 4 helper/runbook queries that become
ambiguous once live and recording runs coexist in the same DB.

## Files and Current Behavior

### `scripts/phase4/start_recorder.sh`

Current relevant flow:

1. Resolves `REPO_ROOT` and `CONFIG`.
2. Parses `database_url:` from the recorder YAML into `DB_URL`.
3. Converts `sqlite:///...` or `sqlite:////...` into `DB_PATH` using
   `DB_PATH="${DB_URL#sqlite:///}"`.
4. Stops any existing recorder process for the same config.
5. Runs `rm -f "$DB_PATH" "$DB_PATH-wal" "$DB_PATH-shm" "$LOG_FILE"`.
6. Starts `uv run recorder --config "$CONFIG"` in the background.

The nuclear wipe is the only code path that must change. The process stop,
startup, log polling, and status output should remain scoped to the existing
script.

### `scripts/phase4/stop_recorder.sh`

Current relevant behavior:

1. Parses `database_url:` from the same recorder YAML as `start_recorder.sh`.
2. Stops the recorder process.
3. Prints final table counters.
4. Prints:
   - `RUN_ID=$(sqlite3 "$DB_PATH" "SELECT run_id FROM runs ORDER BY start_ts DESC LIMIT 1;")`
   - `ACCOUNT_ID=$(sqlite3 "$DB_PATH" "SELECT account_id FROM bybit_accounts LIMIT 1;")`

Both queries are ambiguous in a shared DB. After gridbot starts, the latest
row in `runs` can be a live run, not the latest recorder run. `bybit_accounts
LIMIT 1` can also select the wrong account row when shared setup data contains
multiple accounts.

## Surgical Wipe Algorithm

Replace the DB-file deletion in `scripts/phase4/start_recorder.sh` with this
table-level algorithm:

1. After the prior recorder process has stopped, check whether `DB_PATH`
   exists.
2. If the DB file does not exist, skip the SQL wipe and continue to remove only
   `/tmp/recorder.log`; the recorder will create tables on startup through
   `db.create_tables()`.
3. If the DB file exists, run a single `sqlite3 "$DB_PATH"` operation with
   foreign key enforcement enabled.
4. Use SQLite's `.bail on` so a SQL error aborts the batch instead of
   continuing to later statements.
5. Enable `PRAGMA foreign_keys = ON` before opening the transaction. Do not
   put this PRAGMA inside the transaction; SQLite ignores foreign key mode
   changes while a transaction is active.
6. Open the transaction with `BEGIN IMMEDIATE` so the wipe gets a short
   explicit write lock instead of failing halfway through competing writes.
7. Delete recorder-owned rows in this order:
   - `private_executions`
   - `orders`
   - `wallet_snapshots`
   - `position_snapshots WHERE source = 'live'`
   - `ticker_snapshots`
   - `runs WHERE run_type = 'recording'`
8. Commit the transaction.
9. Keep the operation transactional so a failed wipe does not silently leave a
   partially reset DB before recorder startup.
10. Remove `/tmp/recorder.log` after the SQL wipe path, but do not remove
   `"$DB_PATH"`, `"$DB_PATH-wal"`, or `"$DB_PATH-shm"`.
11. Keep `set -euo pipefail` behavior: if `sqlite3` fails, the script should
   stop before launching a new recorder against an uncleared DB.

The `position_snapshots WHERE source = 'live'` delete is intentionally copied
from issue #113. It wipes live-source recorder telemetry and preserves
`position_snapshots WHERE source='backtest'` until recording runs are deleted.
The final `DELETE FROM runs WHERE run_type='recording'` cascades through
`position_snapshots.run_id` and removes backtest position snapshots tied to
those recording runs. This is deliberate: replay artifacts for the old
recording session are reset along with the old recording session metadata.

Use this SQL shape in the shell implementation:

```sql
.bail on
PRAGMA foreign_keys = ON;
BEGIN IMMEDIATE;
DELETE FROM private_executions;
DELETE FROM orders;
DELETE FROM wallet_snapshots;
DELETE FROM position_snapshots WHERE source = 'live';
DELETE FROM ticker_snapshots;
DELETE FROM runs WHERE run_type = 'recording';
COMMIT;
```

The implementation may embed this via a quoted heredoc such as
`sqlite3 "$DB_PATH" <<'SQL'` so shell expansion cannot alter SQL text.

## Data Ownership Rules

Wipe on recorder start:

| Table or rows | Reason |
|---|---|
| `private_executions` | recorder-owned private execution ground truth |
| `orders` | recorder-owned order state stream |
| `wallet_snapshots` | recorder-owned wallet seed snapshots |
| `position_snapshots WHERE source='live'` | recorder-owned live position seed snapshots |
| `ticker_snapshots` | recorder-owned public ticker stream |
| `runs WHERE run_type='recording'` | recorder run metadata |
| `position_snapshots WHERE source='backtest'` tied to deleted recording runs | replay artifact for the old recording session; removed by FK cascade |

Preserve on recorder start:

| Table or rows | Reason |
|---|---|
| `grid_state_snapshots` | gridbot-owned Feature 0047 seed data |
| `runs WHERE run_type='live'` | gridbot-owned live run metadata |
| `runs WHERE run_type='backtest'` and `runs WHERE run_type='shadow'` | not recorder-owned |
| `bybit_accounts` | shared config/setup data |
| `strategies` | shared config/setup data |
| `users` | shared config/setup data |

Do not add `public_trades` to the wipe in this feature. The Phase 4 LTCUSDT
config uses `capture_public_trades: false`, and issue #113 does not list
`public_trades` as part of the surgical wipe.

## Recording Run Selection

Update `scripts/phase4/stop_recorder.sh` so the printed identifiers are scoped
to the latest recording run, not the latest row in the shared `runs` table.

Required query shape:

1. Select `RUN_ID` from `runs WHERE run_type='recording' ORDER BY start_ts DESC
   LIMIT 1`.
2. Select `ACCOUNT_ID` from that same latest recording run row, not from
   `bybit_accounts LIMIT 1`.
3. If no recording run exists after recorder shutdown, print a clear warning
   instead of emitting blank identifiers.

Make the same query change in `docs/features/0029_RUNBOOK.md` Step 7. The
runbook examples must not use unfiltered `ORDER BY start_ts DESC LIMIT 1` in a
shared DB.

## Script Changes

### Header comments

Update the script header so it no longer says "DB wipe" in a way that implies
file deletion. The comment should describe "recorder-owned data wipe" or
"surgical DB reset".

### DB URL parsing comments

Keep the current `sqlite:///` parsing path. Update the nearby comment so it
mentions both supported forms:

- relative: `sqlite:///data/recorder_ltcusdt_phase4.db`
- absolute: `sqlite:////<abs-path>/data/recorder_ltcusdt_phase4.db`

The existing parameter expansion is compatible with both forms because
removing `sqlite:///` from an absolute four-slash URL leaves a leading `/`.

### Wipe block

Replace the current `echo "==> Wiping prior recording artifacts..."` block and
the `rm -f "$DB_PATH" "$DB_PATH-wal" "$DB_PATH-shm" "$LOG_FILE"` command with
a conditional SQL wipe.

Required operator-facing messages:

- Existing DB: state that recorder-owned data is being wiped while preserving
  gridbot history and shared config.
- Existing DB: state that `grid_state_snapshots`, live runs, accounts,
  strategies, and users are kept.
- Missing DB: state that no DB exists yet and recorder startup will create it.
- Log cleanup: keep removing `/tmp/recorder.log`.

Do not remove WAL/SHM files. Issue #113 explicitly says: "Don't change
`WAL/SHM` handling - surgical DELETE leaves them intact".

### `stop_recorder.sh` identifier block

Replace the post-stop identifier queries with latest-recording-run queries.
Keep the existing final counters, but do not rely on those counters to infer
which run id should be pasted into replay config.

### Grid seed run id caveat

Document the existing Feature 0047 tension without changing replay in this
feature:

- Replay wallet, position, order, and execution loaders use the recording
  `run_id`.
- `grid_state_snapshots` rows are written by gridbot under the live gridbot
  run id.
- Current replay DB grid-state loading calls
  `load_grid_state_from_snapshots(..., run_id, ...)` with the resolved
  recording `run_id`.

Until a follow-up changes replay to resolve grid state by live run id or another
stable association, operators must either provide the correct file fallback
through `seed.grid_state_path` or explicitly understand that DB grid seed may
fall back to file/fresh build when no snapshot exists for the recording run.
This caveat is outside the surgical wipe implementation, but the runbook should
not imply that sharing the DB alone fixes the live-run vs recording-run lookup.
