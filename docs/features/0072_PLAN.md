# Feature 0072 — Event-driven follower fill mode (issue #168)

## Context

Add a new replay fill mode, `event_follower`, where **fills come from the
recorded live's `private_executions`** instead of from the `last_cross` /
`book_touch` / `strict_cross` simulator. The strategy still generates the same
**intents** (place/cancel) per tick; whether each open intent becomes a fill is
decided by replaying the live execution stream rather than by simulator
inference against the ticker.

Goal: make long-window A-vs-B hypothesis comparisons predictive. The simulator
over-fills (queue-priority blindness, memory `project-replay-queue-priority-blindness`),
accumulating `backtest_only` fills that drain the wallet differently for A vs B
and eventually hit `wallet_balance is zero with long position value …`
(`runner.py:746`). Sourcing fills from the real execution stream eliminates
that noise by construction: matched fills carry Bybit's exact price/qty/fee/PnL,
so the wallet evolves exactly as live's did for any window the live bot
survived.

Trade-offs (document, do not "fix"): the fill **sequence** is taken from live,
not synthesized from market state. The mode answers "given live's exact market
interactions, how would strategy A vs B have *dispatched* them" — it cannot model
fills the strategy never placed an order for, cannot simulate a new market
regime, and cannot test latency/placement-timing. For risk-mgmt activation
timing and parameter sweeps it is strictly better than the simulator; for
latency arb / new-regime exploration the simulator modes (and issue #118
`aggressor_crossed`) remain the right tool. Live recorder is untouched — this is
replay-side only, runtime-zero impact.

## Design

`event_follower` is a **pre-tick fill injection** that replaces the
simulator-driven `check_fills` call for the window `(prev_tick_ts,
current_tick_ts]`. The intent-generation half of the tick loop is unchanged.

The recorded executions stream is loaded once at run start (ordered by
`(exchange_ts, exec_id)`) and consumed by an `exchange_ts`-advancing cursor. On
each tick, all executions with `prev_tick_ts < exec.exchange_ts <=
tick.exchange_ts` are drained and dispatched.

**Within-tick open→close ordering (critical).** A grid level often closes the
same tick window it opened: live's recorder shows an OPEN fill then, a few ms
later, the CLOSE fill of the order the strategy placed *reactively* after the
open. In replay, that close order does not exist in `active_orders` when
`process_fills` first runs — the engine only emits and the executor only
places it in the later `execute_tick` half of the *same* tick. A single
forward pass over the window would therefore fail to match the close
(`live_only`), breaking acceptance #2 even with an identical strategy.

The follower drain is consequently **iterative within the tick window**, not a
single pass:

1. Drain the window's executions into a per-tick buffer in repository
   `(exchange_ts, exec_id)` order — **no re-sort in engine or follower**
   (see Files affected: repository is the single sort site).
2. Match-and-apply every buffered execution whose key is in `active_orders`
   right now; remove them from the buffer. For each applied fill:
   `order_manager.apply_recorded_fill(...)` →
   `_apply_recorded_fill_row(fill_event)` (tracker + pending wallet; see
   partial-fill strategy below) → `_maybe_flush_aggregated_trade` (trigger
   **1** only — `is_fully_filled`; see "Intra-loop vs post-fixpoint") →
   `self._engine.on_event(fill_event)` **for grid mutation only**
   (`GridEngine._handle_execution_event` at `engine.py:196-223` updates
   `last_filled_price` / `update_grid` and **always returns `[]`** — it never
   emits `PlaceLimitIntent`).
3. After each applied fill, derive placement intents from the **ticker path**:
   build a synthetic `TickerEvent` at the fill's `exchange_ts` (reuse
   `event.last_price` / `event.mark_price` from the current tick) and call
   `self._engine.on_event(ticker_event, order_manager.get_limit_orders())`
   — the same `_handle_ticker_event` → `_check_and_place` path
   `execute_tick` uses (`runner.py:398-399`). Execute the returned intents
   through the **same intent-dispatch block as `execute_tick`**
   (`runner.py:407-428`): for each `PlaceLimitIntent`, gate with
   `_should_place_close` (which calls `_resolve_intent_qty`), then
   `executor.execute_place(..., wallet_balance=self._session.current_balance,
   timestamp=fill_event.exchange_ts)`; for each `CancelIntent`,
   `executor.execute_cancel(..., timestamp=fill_event.exchange_ts)`. Use
   **`fill_event.exchange_ts`** (not the tick's `exchange_ts`) so the reactive
   close order enters `active_orders` before the next buffered execution is
   matched. This is the **only** intent execution done inside the fill phase.
   **`execute_tick` dedup:** orders placed here use the same deterministic
   `client_order_id` the engine would emit again at tick end; the existing
   `_client_order_ids` dedup in `order_manager.place_order` rejects the
   duplicate, so `execute_tick` need not special-case already-placed intents.
4. Repeat 2–3 until a full pass matches nothing new. Anything still in the
   buffer after the fixpoint had no active order to match and stays
   `live_only`.

This reuses the existing equity-timing contract: `process_fills` still returns
the fill-derived intents (now already placed), `update_equity` still runs
between the two phases on the post-fill state, and `execute_tick` still runs
the *non-fill* per-tick intents afterward. The reactive close placement is
moved earlier (into the fill phase) only for event_follower; the simulator
modes are untouched. Because the close order is placed at `event.exchange_ts`
and its matching recorded execution carries `exchange_ts ≤ tick.exchange_ts`,
the pair both fall in the same window and both match on the second iteration.

For each recorded execution:

1. **Match key-faithfully.** Compute the live join key
   `key = extract_client_order_prefix(exec.order_link_id)`
   (`gridcore/intents.py:16`) — the deterministic grid identity, identical
   to the prefix the live recorder stores AND the prefix the post-run
   `LiveTradeLoader` uses as its match key (`loader.py:122`). Replay's own
   `BacktestOrderManager.active_orders` are keyed by the SAME deterministic
   `client_order_id` (the strategy emits it via `PlaceLimitIntent.create`;
   the wire suffix is replay-side absent / stripped). The follower selects
   the active order **whose `client_order_id == key`**:
   - If exactly one active order has that `client_order_id`, that is the
     match (side and price are asserted as a sanity check — same side, and
     Buy: `order.price >= exec.exec_price` / Sell: `order.price <=
     exec.exec_price`; a violation logs a WARN and is still matched, since
     the deterministic id is authoritative).
   - If `key` is `None` (execution had no `order_link_id` — the pre-#1
     fallback rows the loader joins by `order_id`), fall back to matching by
     `order_id` against `order.order_id`, then by side/closest-price as a
     last resort. These rows are rare and pre-date the orderLinkId hotfix;
     log a counter so a window full of them is visible.
   - **Matcher key (for `BacktestTrade` stamping):** `matcher_key = key or
     exec.order_id` — mirrors `LiveTradeLoader` (`loader.py:122-123`:
     `client_id = link_prefix or ex.order_id`). The post-run matcher keys on
     `(client_order_id, occurrence)`; fallback rows MUST stamp
     `BacktestTrade.client_order_id = exec.order_id`, not the replay order's
     deterministic `client_order_id`.
   - Price/closest-price is therefore a fallback/sanity layer, NOT the
     primary key. This is the change that makes `backtest_only = 0` true
     **by construction** (see below), not merely by test.
2. On match: `apply_recorded_fill` → `_apply_recorded_fill_row` (per partial;
   tracker + pending wallet; **no** `record_trade`) →
   `_maybe_flush_aggregated_trade` (trigger **1** only after each applied row;
   no-op on partials — `record_trade` only when `is_fully_filled`) — see
   "Partial-fill wallet strategy".
   **Never**
   call simulator `_process_fill` on follower rows.
   **Aggregation parity with live:** `LiveTradeLoader` aggregates multiple
   `PrivateExecution` rows sharing `(extract_client_order_prefix(order_link_id),
   order_id)` into one `NormalizedTrade` (`loader.py:108-136`; test
   `test_partial_fills_same_order_id_still_aggregated`), while
   `BacktestTradeLoader.load_from_session` emits one row per `BacktestTrade`
   with no aggregation (`loader.py:205-231`). The follower must mirror live:
   one `BacktestTrade` per `(matcher_key, recorded_order_id)` order lifecycle
   (VWAP price, summed **applied** `exec_qty`/`exec_fee`/`closed_pnl`, latest
   `exchange_ts`), stamped with `matcher_key` (`key or exec.order_id` above).
   **`recorded_order_id`** is always `exec.order_id` from `RecordedExecution`
   (Bybit id, used by `LiveTradeLoader` grouping). **`replay_order_id`** is
   the matched `SimulatedOrder.order_id` in `active_orders` (`sim_*` today —
   `order_manager.py:104`) passed to `apply_recorded_fill` for order lookup.
   Never conflate the two in rollup buffers or trigger-2 tail checks.
   Because the produced backtest trade carries exactly the live normalization
   key and one trade per order lifecycle, the post-run matcher
   (`matcher.py:50-62`, keyed on `(client_order_id, occurrence)`) pairs it
   with the corresponding live trade.

**Placed-qty invariant (no overfill).** Matching by `client_order_id` is
identity-only; fill application must never exceed what replay actually placed.
Before applying each row, compute `apply_qty = min(exec_qty, order.qty)` (the
order's **remaining** qty after prior partials in this lifecycle). If
`exec_qty > order.qty`, log WARN (`qty_excess_divergence`), increment a runner
counter, and apply only `apply_qty` — **do not** pass the excess to
`tracker.process_fill` or pending accumulators. Pro-rate `exec_fee` and
`closed_pnl` for the applied slice:
`apply_fee = exec_fee * (apply_qty / exec_qty)`,
`apply_pnl = closed_pnl * (apply_qty / exec_qty)` (when `exec_qty > 0`).
The excess live qty is intent-set / sizing divergence (A/B acceptance #3–4);
it surfaces as a qty mismatch on the paired trade, not as an overfilled
backtest position. When `apply_qty == 0` (order already fully consumed),
skip application and treat the row as unmatched → `live_only`.

**Partial-fill wallet strategy (single contract).** `session.refresh_balances`
and `session.update_equity` both derive balance from `total_realized_pnl` and
`total_commission`, which today are updated **only** inside `record_trade`
(`session.py:223-224`). Deferring `record_trade` until full fill therefore
requires an interim wallet path visible to **both** helpers — the replay engine
calls `update_equity` immediately after `process_fills` (`engine.py:495-502`),
which would otherwise drop in-flight partial pnl/fees before `execute_tick`.
Use **pending accumulators** on the runner (keyed by
`(matcher_key, recorded_order_id)`):

- **Per partial row** (`_apply_recorded_fill_row`): call
  `tracker.process_fill` with `exec_price` / **`apply_qty`** (discard returned
  realized); bump the order-key's pending `Σ apply_pnl` / `Σ apply_fee`; call
  `session.set_pending_wallet(Σ global pending closed_pnl, Σ global pending
  exec_fee)` **immediately before** `session.refresh_balances(unrealized)` so
  intra-drain reactive `execute_place` (same tick window) sees the correct
  `current_balance`; both `refresh_balances` and
  `update_equity` (event_follower branch only) fold global pending sums into
  `pnl_delta` alongside `total_realized_pnl` / `total_commission` via a
  shared `_pnl_delta(unrealized)` helper; run pair-liq, `_update_risk_multipliers`,
  snapshot — same downstream as `_process_fill` but **no** `record_trade`. This
  keeps `current_balance` correct for reactive `execute_place` inside the drain
  loop **and** for `execute_tick` after `update_equity` on the same tick.
- **End of `process_fills`:** runner calls
  `session.set_pending_wallet(Σ pending closed_pnl, Σ pending exec_fee)` across
  all in-flight order keys so the engine's `update_equity` sees pending without
  an engine.py change.
- **On flush** (`_flush_aggregated_trade`): emit one `BacktestTrade` with
  VWAP price and summed **applied** qty/fee/pnl, `client_order_id=matcher_key`,
  call `record_trade` once, clear that order-key's pending accumulators (values
  are now in `total_*`), refresh `set_pending_wallet`, then `refresh_balances`
  again.

**Flush triggers (aggregation parity with live).** `LiveTradeLoader`
aggregates *all* `PrivateExecution` rows per `(client_id, order_id)` into one
`NormalizedTrade` — it does **not** require the exchange order to be fully
filled. The follower must mirror that: each `(matcher_key, recorded_order_id)`
rollup buffer accumulates partial rows until **exactly one** of triggers 1–4
fires (see below) — **not** on every partial row. A single partial with
`is_fully_filled=False` does **not** flush; only trigger 1 (`is_fully_filled`),
trigger 2 (last-in-stream, post-fixpoint), trigger 3 (cancel), or trigger 4
(end-of-replay sweep) may call `_flush_aggregated_trade`.

**Intra-loop vs post-fixpoint (F-3-1).** Trigger 2 (last-in-stream) must
**never** run inside the within-tick drain loop while same-order partials may
still sit in the per-tick buffer — doing so would split one live order
lifecycle into two `BacktestTrade`s. Split evaluation accordingly:

- **After each applied fill** (`_maybe_flush_aggregated_trade`): trigger **1**
  only (`is_fully_filled`).
- **Before each cancel** (`_dispatch_intents` → `_maybe_flush_aggregated_trade`):
  trigger **3** only (cancel-with-partial flush).
- **Per-tick post-fixpoint** (after the drain loop's match pass completes for
  the tick): trigger **2** only (`_flush_post_fixpoint`).
- **End-of-replay** (after the final tick): trigger **4** only via
  `runner.finalize_event_follower()` called from `engine.py` after the tick
  loop — **not** bundled into per-tick `_flush_post_fixpoint` (would flush
  in-flight partials every tick mid-run).

Triggers:

1. **`is_fully_filled`** — replay placed qty exhausted (existing path).
   **Intra-loop.**
2. **Last execution in stream** — **post-fixpoint only** (per-tick
   `_flush_post_fixpoint`; not end-of-replay — trigger 4 covers that). After the
   drain fixpoint for a tick window, for each
   `(matcher_key, recorded_order_id)` with a non-empty rollup buffer, if no
   further `RecordedExecution` with that **`recorded_order_id`** remains in
   the follower's unconsumed tail (`cursor` → end of list) **and** no row with
   that `(matcher_key, recorded_order_id)` remains in the per-tick buffer,
   flush now. Live
   already emitted one aggregated trade for that order lifecycle; leaving the
   buffer pending would produce `live_only` with no backtest counterpart.
3. **Order leaves `active_orders` without full fill** — on cancel, flush that
   `(matcher_key, recorded_order_id)` rollup before removal. Hook is **only**
   in `runner._dispatch_intents`: call `_maybe_flush_aggregated_trade`
   (trigger 3) immediately before `executor.execute_cancel` (not in
   `order_manager.cancel_order`, which owns no rollup state). The recorded
   partials are the complete live lifecycle for that placement even though
   replay still had remaining placed qty. **Intra-loop** via `_dispatch_intents`.
4. **End-of-replay sweep** — after the final tick, flush every remaining
   non-empty rollup buffer (safety net for trigger 2 edge cases).
   **End-of-replay only** via `runner.finalize_event_follower()` from
   `engine.py` after the tick loop — **not** in per-tick `_flush_post_fixpoint`.

When partial recorded qty sums to less than replay placed qty (intent-set /
sizing divergence), the flushed `BacktestTrade` carries the **applied** partial
sums (VWAP, summed fee/pnl) — same as live's aggregated trade. Remaining
placed qty on the (now-cancelled or stream-ended) order is not filled; the qty
gap surfaces as a trade qty mismatch, not as a missing backtest trade.

Do **not** call the existing simulator `_process_fill` wholesale on partial
rows — it always `record_trade`s. Split into the two helpers above; the
simulator path stays byte-for-byte unchanged when `self._event_follower is None`.
**Unit test (required):** two partial rows for the same `order_id` must update
`current_balance`/fees after each row *and* emit one matcher-facing
`BacktestTrade`.
3. On no match (no active order with `client_order_id == key`): **do not**
   synthesize any backtest position change. This live fill becomes
   `live_only` in the post-run matcher. Intent generation is allowed to
   drift.

### Fill application (correctness invariant)

The new path differs only in **fill source**, not in **how** the fill mutates
position/wallet. The recorded `(exec_price, exec_qty, exec_fee, closed_pnl)`
are used as-is (issue out-of-scope: do NOT recompute Bybit's `realized_pnl`).

**Exact wallet arithmetic (no double-count).** `session.refresh_balances`
(`session.py:309-321`) computes both balance baselines from the identity:

```
balance = initial_balance
        + total_realized_pnl           # Σ BacktestTrade.realized_pnl (record_trade, session.py:223)
        + unrealized_pnl               # long.calc_unrealized(mark) + short.calc_unrealized(mark)
        - total_commission             # Σ BacktestTrade.commission
        + total_funding
```

So `total_realized_pnl` and `unrealized_pnl` are the two PnL terms that must
not double-count. They are fed from two *different* sources:

- `total_realized_pnl` ← we set `BacktestTrade.realized_pnl = recorded
  closed_pnl` (NOT `tracker.process_fill`'s return). This makes the realized
  term equal live's exact realized stream.
- `unrealized_pnl` ← `tracker.calculate_unrealized_pnl(mark)` =
  `size × (mark − avg_entry_price)` (long) using the tracker's *own*
  size/entry, which is still maintained by calling `tracker.process_fill` for
  its size/entry side-effects.

These two terms are consistent **iff** the tracker's `avg_entry_price`
evolution reproduces the entry basis Bybit used to compute each `closed_pnl`.
The grid bot's fills are plain limit fills at the recorded `exec_price`, and
`tracker.process_fill` updates `avg_entry_price` from exactly that
`exec_price` (weighted-average on opens, basis-preserving on reduces —
`position_tracker.py:219-` `_add_to_position` / `_reduce_position`). So on an
**open** fill realized is 0 on both sides and only entry/size move (no
conflict). On a **close** fill, the tracker's own derived realized
(`reduce_qty × (exec_price − avg_entry_price)`) and Bybit's `closed_pnl` are
computed from the *same* `exec_price` and the *same* running average entry —
they agree up to Bybit's funding/fee inclusion and rounding. We use the
recorded `closed_pnl` for the realized term (authoritative) and let the
tracker keep its entry/size; the unrealized term is then computed off that
same entry on the *remaining* size, so the wallet identity holds with no
double-count: the closed portion contributes via `closed_pnl`, the open
remainder via `unrealized`, and they partition the position exactly once.

**Tolerance guard.** Add a debug assertion in the event_follower branch:
`abs(recorded_closed_pnl − tracker_derived_realized) ≤ EPS` on closing fills
(EPS ~ a few cents, the funding/rounding band documented in memory
`project-pos-value-mark-jitter`). A breach means the tracker's entry diverged
from Bybit's basis (e.g. a missed open fill the strategy never placed) — log
WARN with both values; do NOT abort (the recorded value still wins for the
wallet). This surfaces basis drift instead of silently letting
`equity = realized + unrealized` drift.

**One open→close cycle, worked:** open Buy 1 @ 100 → tracker size=1,
entry=100, realized=0, recorded closed_pnl=0; `balance = init + 0 + 1×(mark−100)
− fee`. Close Sell 1 @ 110 → recorded closed_pnl=+10; we set
`total_realized_pnl += 10`; tracker reduces size→0 so `unrealized=0`;
`balance = init + 10 + 0 − fees`. Live's wallet after the same two fills is
`init + 10 − fees`. Match. No double-count because once size is 0 the
unrealized term is structurally 0 and the entire gain lives in the realized
term.

Approach:

- Build an `ExecutionEvent` whose `price = exec_price`, `qty = exec_qty`,
  `fee = exec_fee`, and carry the recorded `closed_pnl` (reuse the existing
  `ExecutionEvent.closed_pnl` field, which `check_fills` currently sets to 0).
- In `_apply_recorded_fill_row`: call `tracker.process_fill` for size/entry,
  **discard its return**, bump pending accumulators from recorded
  `closed_pnl`/`exec_fee`. In `_flush_aggregated_trade`: set
  `BacktestTrade.realized_pnl` / `commission` from aggregated recorded sums
  (not tracker-derived). Run the EPS assertion on flush.
- Fee likewise comes from `exec_fee` (recorded), not from
  `qty * price * commission_rate`.

Everything after the trade record is unchanged: `refresh_balances`, pair-liq,
`_update_risk_multipliers` (`calculate_amount_multiplier` now reads now-true
position state, so risk-mgmt rules fire at the right wallet/equity points), and
`_emit_position_snapshot`.

### `backtest_only` semantics

In event_follower mode the backtest never invents a fill. Every produced
backtest trade is created by step 2 above from one or more recorded execution
rows for the same `(matcher_key, recorded_order_id)` order lifecycle,
aggregated to match `LiveTradeLoader._aggregate_fills`, and its
`client_order_id` is stamped to `matcher_key` (`extract_client_order_prefix(
order_link_id) or recorded_order_id` — same rule as `LiveTradeLoader`
`loader.py:122-123`). Therefore every backtest match key is, by construction,
a member of the live key set, so the post-run matcher's
`backtest_keys - live_keys` is empty and `backtest_only_count = 0` **provided
partial-fill aggregation parity holds** (one `BacktestTrade` per
`(matcher_key, recorded_order_id)`, not one per `PrivateExecution` row).

The one residual that could still produce `backtest_only > 0` is the
`(client_order_id, occurrence)` composite: if a single grid id is reused
within the window (live closes + re-opens the same level) and the backtest
dispatches a *different number* of **aggregated order lifecycles** for that id
than live did, the occurrence indices won't line up. This is exactly the
"intent-set divergence" the equal-strategy round-trip is supposed to make zero
(acceptance #2) and the differ-strategy run is supposed to surface (acceptance
#3). State explicitly: in the equal-strategy case the per-id lifecycle count
matches live by construction (same strategy, same recorded fills, same
active-order set, same partial-fill aggregation), so occurrences line up and
`backtest_only = 0`; in the differ-strategy case a divergent id *can* yield a
small `backtest_only` (the backtest dispatched a re-open that live's occurrence
sequence no longer pairs with), and that is reported as divergence, not hidden.
The acceptance test asserts `backtest_only = 0` for the equal-strategy
round-trip and **must include a partial-fill fixture** (two execution rows,
same `order_id`, aggregated to one trade on both sides); if `backtest_only`
surfaces nonzero, the failure is a real occurrence/aggregation bug to fix, not
a tolerance to widen.

`live_only_count` retains meaning but shifts: it now measures "live fills the
backtest had no open order to match" = where strategy A's kept-open order set
diverged from live's. Reported as such (see Phase 5). No mechanical change to
`MatchResult` / `calculate_metrics`.

### Phase 2 — Recorded-fill source

New plain frozen dataclass `RecordedExecution` + class `EventFollower` in
`apps/backtest/src/backtest/fill_simulator.py` (co-located with the simulator;
keeps `apps/backtest` as the single fill home):

- `RecordedExecution` (frozen dataclass): `exec_id: str`, `order_link_id: str
  | None`, `order_id: str`, `side: SideType`, `exec_price: Decimal`,
  `exec_qty: Decimal`, `exec_fee: Decimal`, `closed_pnl: Decimal`,

  `PrivateExecution` ORM rows.** The engine (Phase 3) loads the ORM rows

### Phase 4 — Metrics / no-`backtest_only`

holds **by construction**, not by inference: every produced backtest trade's
`client_order_id` is stamped to `matcher_key` (`extract_client_order_prefix(

`loader.py:122-123`), and partial fills are aggregated per
`(matcher_key, recorded_order_id)` to match `LiveTradeLoader` (Design step 2),

live_keys = ∅` on the `client_order_id` axis). The only way `backtest_only > 0`
can appear is an `occurrence`-index mismatch on a within-window-reused grid id
where the backtest dispatched a different **aggregated lifecycle** count than
live — which is genuine intent-set divergence (acceptance #3), not a phantom to
suppress. The equal-strategy round-trip (acceptance #2) asserts
`backtest_only = 0` (including partial-fill fixture); if it fails there, fix
the occurrence/aggregation bug rather than masking it.

## Acceptance (from issue #168)

1. `mode = event_follower` runs a 7-day window without the `wallet_balance is
   zero` error (same data + strategy as live).
2. Strategy == live → `live_only_count = 0`, `backtest_only_count = 0`,
   `cumulative_pnl_delta ≤ $0.01`, `pos_value_final_delta = 0`.
3. Strategy differs (e.g. `min_total_margin: 3 → 1`) → `live_only_count > 0`,
   `total_backtest_pnl ≠ total_live_pnl`, divergence attributable to
   rule-firing timing.
4. A/B sweep on `min_total_margin` (0.5 vs 3.0) → clean
   `(A.total_backtest_pnl − B.total_backtest_pnl)`, sign monotonic across sweep.
5. Existing `last_cross` mode unaffected (defaults preserved; regression tests
   pass) — the event_follower branches in `process_fills` /
   `_apply_recorded_fill_row` / `session.set_pending_wallet` /
   `session._pnl_delta` pending fold-in are guarded by mode.
6. Live recorder untouched.

## Risks (from issue + verification)

- **Non-matching intents** → all live fills become `live_only`, backtest
  produces no fills (null-result run). Feature, not bug — surface in reporter.
- **Matching ambiguity** → resolved by key-faithful match: the primary key is
  `extract_client_order_prefix(exec.order_link_id)` against each active order's
  deterministic `client_order_id`, with `order_id` then side/closest-price as
  documented fallbacks. The produced `BacktestTrade.client_order_id` is stamped
  to `matcher_key` (`key or exec.order_id`, mirroring `LiveTradeLoader`) so the
  post-run matcher pairs it (makes `backtest_only = 0` true by construction —
  see Design step 2 / Phase 4). **Qty excess** on a matched id (replay placed
  less than live filled) is capped at placed qty — surfaces as trade qty
  mismatch / `live_only` attribution, not backtest overfill (Design
  "Placed-qty invariant").
- **Within-tick open→close ordering** → reactive close orders the strategy
  only places after an opening fill are matched via the iterative drain (Design
  "Within-tick open→close ordering"); a same-window fixture in the round-trip
  test guards it.
- **Detached ORM rows** → executions are materialized to `RecordedExecution`
  dataclasses inside the DB session before the follower receives them (avoids
  `DetachedInstanceError`; cf. feature 0038).
- **Determinism** → executions sorted `(exchange_ts, exec_id)`; cursor is
  forward-only and rejects non-monotonic tick ts.
- **No `closed_size` column** — issue assumed it exists; only `closed_pnl`
  (signed cycle PnL for the fill) and `exec_qty` are available. Position
  size/entry is maintained by the tracker; the recorded `closed_pnl` is used
  for the trade's realized PnL and wallet evolution.

## Out of scope (from issue)

- Streaming event_follower live (replay-only).
- Mixing event_follower + simulator mid-window (one mode per run).
- Recomputing live `realized_pnl` (use Bybit's recorded `closed_pnl` as-is).
- Capturing new WS event types (uses existing `private_executions`).
