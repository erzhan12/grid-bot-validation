# Feature 0071 — Expose RiskConfig fields in replay config (issue #162)

## Context

`ReplayStrategyConfig` (`apps/replay/src/replay/config.py`) does not surface
five operator-tunable risk-management thresholds. The replay engine
(`apps/replay/src/replay/engine.py`) builds a `BacktestStrategyConfig` from
`ReplayStrategyConfig` and passes only 9 of its fields, so
`min_liq_ratio`, `max_liq_ratio`, `min_total_margin`,
`increase_same_position_on_low_margin`, and `leverage` silently fall back to
`BacktestStrategyConfig` defaults on every replay run.

The most impactful gap is `min_total_margin`: the backtest default is `0.15`,
while the live per-strategy value is much higher (operator reports `3` for LTC
/ `2.5` for SOL — see "Live-mirror values" below) — a ~20x difference that
means the `low_margin` rule branch (priority 3 of 5 in
`Position._apply_long_position_rules` / `_apply_short_position_rules`,
`packages/gridcore/src/gridcore/position.py:359` and the short mirror) never
fires (or fires on the wrong condition) in replay versus production. The
`increase_same_position_on_low_margin` flag is also inverted by default
(`False` in backtest/repo-example vs `true` live), so the
`_adjust_position_for_low_margin` branch (`position.py:455`) does the opposite
of live (suppress-opposite vs self-boost). This blocks risk-mgmt timing
hypothesis tests on the 2026-06-03 storm (issues #159, #160).

The `low_margin` branch fires only when BOTH `0.94 < position_ratio < 1.05`
(near-equal long/short margin, the open-interval `is_position_equal` gate at
`position.py:270`) AND `total_margin < min_total_margin` hold. Raising
`min_total_margin` from `0.15` to `3`/`2.5` makes the threshold far easier to
cross, but the near-equal gate still applies — so the round-trip test and any
hypothesis-test setup must produce near-equal margins under the threshold for
the branch to fire.

### Live-mirror values (operator-supplied — NOT repo-derived)

This repo contains **no committed live gridbot config**: the only gridbot
config present is `apps/gridbot/conf/gridbot.yaml.example`, which ships the
backtest-matching defaults `min_total_margin: 0.15` and
`increase_same_position_on_low_margin: false` (and `RULES.md` documents only
`0.15`). The operator runs a private/gitignored live config (not in this repo).
The live per-strategy values used in Phase 3 — `min_total_margin` `3` (LTC) /
`2.5` (SOL), `increase_same_position_on_low_margin: true`, `leverage: 10`,
`min_liq_ratio: 0.8`, `max_liq_ratio: 1.2` — are **operator-supplied** from
that private config and CANNOT be verified against anything in this repo. They
are treated as the source of truth for "what live runs". Phase 3 applies a
two-tier confirmation rule: illustrative commented examples in committed
`.example` files are allowed (with caveats); concrete values in gitignored
iterative yamls require per-value operator confirmation (see Phase 3).

Goal: make these five fields pass-through from `ReplayStrategyConfig` →
`BacktestStrategyConfig` with defaults that match `BacktestStrategyConfig`
(zero behaviour change for existing replay yamls), document live-mirror
values in committed `.example` templates, and add config + runner wiring
tests.

## Phase 3 — Document yaml configs (operator-local live values)

**Repo convention — iterative configs are NOT committed deliverables.**
`.gitignore:89` ignores `apps/replay/conf/iterative_*.yaml` ("per-window
iterative configs are not meant for the repo"; see also
`docs/features/0056_REVIEW.md`). Those paths must NOT appear in "Files to
change" or be committed. Live-mirror numbers belong in operator-local copies
the operator maintains outside git (or in untracked gitignored yamls).

**Two-tier rule for live-mirror numbers (resolves guard vs template).**

The live-mirror numbers (`min_total_margin` 3/2.5, flag `true`, `leverage` 10,
ratios 0.8/1.2) are operator-supplied and unverifiable from this repo (see
"Live-mirror values" in Context). They are used in two different places with
different confirmation bars:

1. **Committed `.example` templates** (`replay.yaml.example`, file #6 below):
   MAY document these values as **commented illustrative examples** with
   mandatory inline caveats ("operator-supplied — confirm against your own
   live config before use"). This is documentation, not runtime config; the
   repo ships backtest defaults (`0.15` / `false`) as the uncommented
   baseline. The operator already supplied these numbers for feature 0071 /
   issues #159/#160, so they are confirmed enough to cite as examples in
   committed docs — but every mention MUST carry the not-repo-derived caveat.

2. **Operator-local iterative yamls** (gitignored `iterative_*.yaml`, used for
   actual storm replay runs): require **per-value operator confirmation**
   before writing concrete numbers. If a value cannot be confirmed at
   implementation time, leave that field out so it falls back to the verified
   backtest default — a wrong value an operator trusts is worse than a
   documented default.

Phases 1, 2, and 4 do NOT depend on these specific numbers and can land
independently of iterative-yaml confirmation.

**Operator-local workflow (not a repo commit):** once confirmed, add the five
fields to the `strategy:` block of each operator's gitignored iterative copy
(e.g. `apps/replay/conf/iterative_v10_ltc.yaml` — create locally; zero
matches in the working tree today). Per-strategy operator-reported values:

- LTCUSDT (`iterative_v10_ltc.yaml`, `iterative_v11_ltc.yaml`):
  `min_liq_ratio: 0.8`, `max_liq_ratio: 1.2`, `min_total_margin: 3`,
  `increase_same_position_on_low_margin: true`, `leverage: 10`.
- SOLUSDT (`iterative_v10_sol.yaml`, `iterative_v11_sol.yaml`): same except
  `min_total_margin: 2.5`.

**Committed template (repo deliverable):** `replay.yaml.example` — add the five
fields (commented) with an inline note:
these default to `BacktestStrategyConfig` values (0.8 / 1.2 / 0.15 / false / 10)
when omitted; populate ALL five to mirror live risk-mgmt. Per the two-tier rule
above, include operator-reported illustrative values as comments (e.g.
`# min_total_margin: 3  # LTC example — operator-supplied, confirm against your
live config`) alongside the caveat that they are not repo-derived; the repo's
only gridbot config (`gridbot.yaml.example`) ships `0.15` / `false` as the
verified baseline. Do NOT uncomment live-mirror numbers in the example — keep
them commented documentation only.
