# 0020 — Extract `AuthCooldownManager` from `Orchestrator`

## Description

Move the auth-cooldown state machine out of `Orchestrator` into a
dedicated `AuthCooldownManager` class in its own module. Today the logic
lives in three disconnected places inside `orchestrator.py` — the two
state dicts in `__init__`, the `_on_auth_cooldown_entered` entry method
(around line 581), and the cooldown-expiry sweep at the top of
`_health_check_once` (lines 1029–1046). Co-locating all of it in one
class makes the state machine readable in one place, simplifies
`Orchestrator`, and enables narrow unit tests.

**No behaviour change.** The fail-loud main-thread guard, per-strategy
cycle counting, retry-queue clearing, and notifier alert semantics stay
byte-identical. This is a structural refactor, not a fix.

---

## Motivation

Auth cooldown is a self-contained state machine with a documented
main-thread invariant (`orchestrator.py:599–611`), yet it is currently
smeared across three locations in a 1000+ line file:

1. **State**: `_auth_cooldown_until` and `_auth_cooldown_cycles` dicts
   (line 163–164).
2. **Entry** (executor → orchestrator callback): `_on_auth_cooldown_entered`
   (line 581). Writes state, clears retry queue, alerts.
3. **Expiry** (timed sweep): embedded inside `_health_check_once`
   (line 1029–1046). Reads state, calls `executor.reset_auth_cooldown()`,
   alerts, deletes state.

The three touch overlapping dict keys but are separated by ~450 lines
of unrelated orchestrator code (WS setup, runner wiring, REST helpers,
health-check WS reconnect logic). Anyone modifying the cooldown rules
has to remember all three sites. This is a textbook candidate for
cohesive extraction.

---

## Invariants (DO NOT CHANGE)

| Guarantee | Why |
|---|---|
| Main-thread guard in entry callback | Documented in `orchestrator.py:599–611` as "fail-loud thread guard is deliberate — not a missing lock". Raises `RuntimeError` on non-main-thread callers. Must carry over verbatim. |
| Cycle counter increments on every cooldown entry (`cycles += 1`) | Used in notifier messages and by any future backoff logic; must not reset on reuse of the same `strat_id`. |
| Retry queue cleared on cooldown entry (`retry_queue.clear()`) | Prevents stale auth-failing intents from firing the same cooldown again immediately after resume. See `orchestrator.py:622–628`. |
| Expiry sweep iterates over `list(keys())` snapshot, not live dict | Allows `del self._auth_cooldown_until[strat_id]` without `RuntimeError: dict changed size during iteration`. Pattern at line 1031. |
| Expiry calls `executor.reset_auth_cooldown()` only if executor is found | Defensive — covers dynamic strategy removal. |
| Notifier alert keys: `auth_cooldown_{strat_id}` (entry), `auth_cooldown_resume_{strat_id}` (expiry) | Notifier dedupes by `error_key`; changing them would break dedupe behaviour. |
| `auth_cooldown_minutes` read from `GridbotConfig` | Runtime tuning knob. |

## Files to create

### `apps/gridbot/src/gridbot/auth_cooldown_manager.py` (new)

New module. Contents:

- `class AuthCooldownManager:`
  - Constructor dependencies (inject, do not reach back into orchestrator):
    - `strategy_executors: dict[str, IntentExecutor]`
      — the manager needs to call `executor.reset_auth_cooldown()` on
      expiry and read `executor.auth_failure_count` on entry for the
      notifier message. Pass by reference so later-added strategies
      appear automatically.
    - `retry_queues: dict[str, RetryQueue]`
      — the manager needs to call `retry_queue.clear()` on entry.
      Pass by reference for the same reason.
    - `notifier: Notifier`
    - `cooldown_minutes: int` (read once from `GridbotConfig.auth_cooldown_minutes`)
  - Instance state (migrates from `Orchestrator`):
    - `_auth_cooldown_until: dict[str, datetime]` — strat_id → expiry.
    - `_auth_cooldown_cycles: dict[str, int]` — strat_id → cumulative count.
  - Public methods:
    - `enter(strat_id: str) -> None`
      — body = current `_on_auth_cooldown_entered`, including the
      main-thread guard (verbatim `RuntimeError` with the same
      message), cycle increment, expiry computation, retry-queue
      clear, notifier alert.
    - `sweep_expired(now: datetime) -> None`
      — body = current expiry-sweep block from `_health_check_once`.
      Takes `now` as argument (not `datetime.now(UTC)` inline) so
      unit tests can advance time deterministically.
    - `is_active(strat_id: str) -> bool` (optional, nice-to-have)
      — returns `True` if `strat_id` is currently in cooldown. Used
      by potential future callers who want to check cooldown state
      without touching internals. Not strictly needed by the refactor;
      may defer.
    - `cycle_count(strat_id: str) -> int` (optional)
      — returns cumulative cycle count for a strategy. Read-only.
      May defer if no caller needs it.
  - Module logger: `logger = logging.getLogger(__name__)` (so log
    records carry the new module name).

The class holds **no reference** to `Orchestrator`. All cross-subsystem
hand-off goes through the four constructor-injected arguments. Main
testability win: a unit test can instantiate `AuthCooldownManager` with
a couple of mock executors and a fake notifier, no orchestrator needed.

## Files to change

### `apps/gridbot/src/gridbot/orchestrator.py`

- **Imports**: add `from gridbot.auth_cooldown_manager import AuthCooldownManager`.
- **`__init__` changes**:
  - Remove fields `_auth_cooldown_until` and `_auth_cooldown_cycles`.
  - Instantiate `AuthCooldownManager` eagerly in `__init__`, just after
    the `PositionFetcher` block. The constructor takes
    `_strategy_executors` and `_retry_queues` by reference; the dicts
    are declared empty here (lines 103, 105) and populated by
    `_init_strategy` later, so reads through the manager always see
    the latest entries. Mirrors the 0019 `PositionFetcher` pattern at
    `orchestrator.py:133–139`.
    ```
    self._auth_cooldown = AuthCooldownManager(
        strategy_executors=self._strategy_executors,
        retry_queues=self._retry_queues,
        notifier=self._notifier,
        cooldown_minutes=self._config.auth_cooldown_minutes,
    )
    ```
- **`start()`**: no changes needed. Because the manager is constructed
  in `__init__`, the `on_cooldown_entered` callback registered by
  `_init_strategy` resolves against the already-built
  `self._auth_cooldown`. The executor and retry-queue dicts are held
  by reference and fill in naturally as `_init_strategy` runs, so no
  ordering caveat applies.
- **Callback registration (`_init_strategy`, line 491)**:
  ```
  on_cooldown_entered=lambda sid=strat_id: self._on_auth_cooldown_entered(sid),
  ```
  becomes
  ```
  on_cooldown_entered=lambda sid=strat_id: self._auth_cooldown.enter(sid),
  ```
- **`_health_check_once` (line 1028–1046)**: replace the inline
  expiry-sweep block with a single call:
  ```
  self._auth_cooldown.sweep_expired(datetime.now(UTC))
  ```
  The outer `try/except Exception` (line 1082–1085) stays — it still
  needs to catch any `AuthCooldownManager.sweep_expired` exceptions
  alongside WS-reconnect failures. Same alert key.
- **Delete** method `_on_auth_cooldown_entered` (body moved to
  `AuthCooldownManager.enter`).
- **Keep** the fail-loud main-thread guard. It moves with the method
  body — `AuthCooldownManager.enter()` performs the check. Docstring
  in the new class repeats the reasoning verbatim (fail-loud preferred
  over lock, see 0020 motivation for why).

### `apps/gridbot/tests/test_orchestrator.py`

- Tests that currently call `orchestrator._on_auth_cooldown_entered(...)`
  or assert on `orchestrator._auth_cooldown_until` / `_auth_cooldown_cycles`
  must route through `orchestrator._auth_cooldown` instead (or be moved
  wholesale to `test_auth_cooldown_manager.py`, see below).
- Expected migration: most pure-state tests move to the new file; any
  test that covers the **integration** between cooldown expiry and
  `_health_check_once` (e.g. "does the health check actually invoke
  the sweep?") stays in `test_orchestrator.py` but as a one-liner with
  a mocked `AuthCooldownManager.sweep_expired`.

### `apps/gridbot/tests/test_auth_cooldown_manager.py` (new)

New test module covering `AuthCooldownManager` in isolation. At minimum:

- `test_enter_from_non_main_thread_raises` — spawns a `threading.Thread`
  that calls `enter()`, asserts `RuntimeError` with the documented
  message prefix. **This is the most important test** — the fail-loud
  guard is the security-property of the design.
- `test_enter_increments_cycle_counter` — first call sets cycle=1,
  second sets cycle=2 for the same strat_id.
- `test_enter_sets_expiry_based_on_cooldown_minutes` — assert expiry
  is roughly `now + timedelta(minutes=cooldown_minutes)` (tolerate
  sub-second drift).
- `test_enter_clears_retry_queue` — mock retry queue, assert
  `clear()` called.
- `test_enter_alerts_notifier_with_cycle_number` — assert alert body
  contains `"cycle {N}"`.
- `test_enter_missing_executor_falls_back_to_question_mark` — executor
  dict empty; assert alert body contains `"? consecutive auth errors"`.
- `test_sweep_expired_removes_past_expiries` — seed two entries,
  advance `now`, assert expired removed + remaining untouched.
- `test_sweep_expired_calls_reset_on_executor` — mock executor, assert
  `reset_auth_cooldown()` called exactly once per expired strat_id.
- `test_sweep_expired_missing_executor_still_deletes_entry` — dict
  should be cleared even if executor dict lost the key mid-flight.
- `test_sweep_expired_alerts_with_resume_message` — alert body contains
  `"cooldown expired"` and `"cycle {N}"`.

These tests are currently either missing or awkwardly embedded in
`test_orchestrator.py`; isolating them removes the need to instantiate
a full orchestrator just to check cooldown math.

## Algorithm / dependency-flow

Before:

```
Orchestrator
├── _auth_cooldown_until
├── _auth_cooldown_cycles
├── _on_auth_cooldown_entered          ← called from executor callback
└── _health_check_once
       └── inline expiry-sweep loop    ← reads/clears the two dicts
```

After:

```
Orchestrator
├── _auth_cooldown: AuthCooldownManager
│     ├── _auth_cooldown_until
│     ├── _auth_cooldown_cycles
│     ├── enter(strat_id)              ← called from executor callback
│     └── sweep_expired(now)           ← called from _health_check_once
│
└── _health_check_once                  ← one-line delegate for cooldown expiry
```

Executor callback flow is unchanged in shape — still a closure passed
into `IntentExecutor.__init__` — only the bound target changes.

## Non-goals / out of scope

- **No change to cooldown semantics.** `auth_cooldown_minutes`, cycle
  counter behaviour, retry-queue clear, and notifier alert keys ship
  untouched.
- **No locking added.** The fail-loud main-thread invariant stays; a
  lock would subtly change semantics and the `_on_auth_cooldown_entered`
  docstring explicitly rejects that direction.
- **No observability metrics.** Counter/histogram of cooldown cycles
  is a separate monitoring concern (see plan 0019's non-goals — no
  monitoring stack exists yet).
- **No change to `IntentExecutor`.** The executor still owns
  `auth_failure_count`, `auth_cooldown` flag, `reset_auth_cooldown()`.
  This plan only moves the coordinator that reacts to executor events.
- **No merge with `PositionFetcher` into a generic "managers" module.**
  They are independent subsystems with no shared state.
- **No scheduler extraction.** `_health_check_once` is still invoked
  via timestamp-gating in `_tick`; only the inline sweep inside it
  becomes a one-line delegate.

---

## Sequencing vs plan `0019`

Plans `0019` (PositionFetcher) and `0020` (AuthCooldownManager) are
independent — different files, different state, different callers.
Recommended order:

1. Ship `0019` first. It's larger (~150 lines + 5 methods + 4 constants)
   and surfaces the extraction pattern.
2. Ship `0020` second. It reuses the same pattern (inject dependencies,
   no back-reference to orchestrator, instantiate in `start()`) and is
   smaller (~60 lines + 2 methods), so there is less friction once
   `0019` has set the precedent.

Doing them in the same PR is possible but discouraged — two isolated
refactors review more cleanly separately.
