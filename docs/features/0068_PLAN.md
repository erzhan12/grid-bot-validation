# Feature 0068 — gridbot-health: `qty_zero` ERROR category + 110017/110072 event_coverage tracking (Issue #150)

## Context

The 2026-05-30 multi-strat incident (#149) surfaced two gaps in the `gridbot-health` analyzer:

1. 535× `ErrCode 110017 (orderQty will be truncated to zero)` on SOLUSDT fell into the generic `other` bucket of `_categorize_error`, drowning the signal (`other=270` looked like uncategorized noise when 95%+ was one classifiable failure mode).
2. The `event_coverage` rare-event ledger (shipped in #145) does not name `110017` or `110072` as first-class events, yet both are exactly the rare-but-meaningful signal the ledger captures (first ever seen 2026-05-30).

This is a **SKILL-ONLY** change to the gitignored `.claude/` tree. No feature branch, no tracked-source impact, no runtime impact. Two edits, both in `.claude/skills/gridbot-health/analyze.py`, plus `SKILL.md` doc and `test_analyze.py` tests.

**#145 status: CLOSED (shipped).** `event_coverage` + Table 13 are fully live. Therefore both Change C and Change D are implementable now in one pass — no deferral/stub needed.

Verbatim from the issue, the exact match rule for Change C:
> Match `"110017"` OR `"orderQty will be truncated to zero"` → category `qty_zero`.

And the two new event_coverage keys for Change D:
> ```jsonc
> "qty_zero":     {"count": 0, "first_seen": null, "last_seen": null},  // ErrCode 110017
> "dup_link_id": {"count": 0, "first_seen": null, "last_seen": null}    // ErrCode 110072
> ```

## Scope clarifications resolved by code research

- **`qty_zero` in `_categorize_error`** (Change C): new bucket added *before* the `other` fallback, *after* the existing `grid_state_writer` branch. This automatically flows into Table 1 row 6 (`error_categories` dict at `analyze.py:1486-1489`) and into `_build_event_coverage_window`'s ERROR mapping (`analyze.py:874-881`).
- **`dup_link_id` (110072) is NOT a `_categorize_error` case.** 110072 ("OrderLinkedID is duplicate") is currently counted separately as `err_dupe` (`analyze.py:1468-1469`, the `_make_check` 110072 counter feeding Table 1 row 3 / Table 3), NOT routed through `_categorize_error`. The issue adds `dup_link_id` ONLY as an `event_coverage` key (Change D). It must be populated in `_build_event_coverage_window` by a direct line scan mirroring `err_dupe` (`"110072" in msg or "OrderLinkedID is duplicate" in msg`), NOT via a `_categorize_error` category. Do NOT add a `dup_link_id` branch to `_categorize_error`.
- **`qty_zero` event key population** (Change D): once `_categorize_error` returns `qty_zero`, the existing ERROR-mapping loop in `_build_event_coverage_window` (`analyze.py:874-881`) maps `_categorize_error` categories to `error_*` event keys via a dict. Add a `"qty_zero": "qty_zero"` entry to that map so `qty_zero` ERRORs flow into the new event key (it is a category, unlike `dup_link_id`).

## Algorithm (Change C ordering)

`_categorize_error` is a first-match-wins ordered chain. The new branch must sit:

1. after `grid_state_writer` (line 1156),
2. before the final `return "other"` (line 1158),

so legacy categories are unchanged and only previously-`other` 110017 messages are reclassified:

```
if "110017" in msg or "orderQty will be truncated to zero" in msg:
    return "qty_zero"
return "other"
```
