# Feature 0051 — `last_cross` fill mode: transition-based aggressor detection

## Brief

Feature 0051 implements GitHub issue
[#117](https://github.com/erzhan12/grid-bot-validation/issues/117). It adds a
new replay fill mode, `LAST_CROSS`, that detects aggressor crossings of a
limit price by inspecting `last_price` transitions across consecutive ticker
snapshots, rather than the current `last_price` value or current L1
bid/ask.

The 2026-05-22 deep dive on a Bybit-mainnet LTCUSDT shadow run isolated a
+12.6s lag in replay for a SELL @ 54.20 (live filled at 11:48:58.776;
replay book_touch filled at 11:49:11). Existing modes can't reproduce the
aggressor cross encoded in the ticker stream:

- `STRICT_CROSS`: BUY when `last < limit`, SELL when `last > limit`. Misses
  at-limit prints entirely.
- `TRADE_THROUGH_AT_LIMIT`: BUY when `last ≤ limit`, SELL when `last ≥
  limit`. `last_price` is sticky between trades (Bybit only updates after a
  trade), so the same threshold check fires on many later sampled ticks
  where no new aggressor cross actually happened. Too eager.
- `BOOK_TOUCH` (current replay default, Feature 0033): BUY when
  `ask1 ≤ limit`, SELL when `bid1 ≥ limit`. Waits for the resting side to
  move past the limit; this lags the actual fill timestamp (the +12.6s in
  the deep dive).

The aggressor cross is in the **transition** of `last_price` across the
limit. The plan adds `FillMode.LAST_CROSS`, threads it through the existing
config and engine plumbing in parallel with the other three modes, and
holds the new state (`prev_last_price` per symbol) on the fill simulator
instance.

Per issue #117, this feature does not deprecate `book_touch` or
`trade_through_at_limit`, does not change the fill price (always limit
price), and does not introduce probabilistic fills. It is also intentionally
narrower than the eventual `public_trades`-driven mode (a follow-up after
the public trades recorder change tracked under #issue-for-0052).

## Last-Cross Algorithm

Per-symbol state on the simulator instance (full details in "Per-tick
state advance" below):

- `self._prev_last_price: dict[str, Decimal]` — committed prior-tick
  `last_price`, written once per tick.
- `self._tick_prev_last: dict[str, Decimal | None]` — read slot for
  the current tick; `_should_fill_last_cross` reads from here.
- Treat "no prior tick yet" as `None`. A `None` read-slot value must
  **not** fire a fill — there is no observed transition.

For each `check_fill(order, market)` call, the simulator needs the
current `last_price` and the **previous** `last_price` for the same
symbol observed on the prior tick.

Per-snapshot algorithm executed inside
`_should_fill_last_cross(side, limit_price, snapshot)`:

1. The per-tick stash/commit logic in `advance_market` has already
   run before this function is called — the order manager invokes
   `self.fill_simulator.advance_market(market)` once per tick as
   the first statement inside the `isinstance(market, TickerEvent)`
   branch of `check_fills`, before any `check_fill` call. See
   "Per-tick state advance" for the staging and the bare-`Decimal`
   short-circuit (the legacy `else`-branch of `check_fills`, and
   any direct call to `check_fill` with a bare `Decimal`, never
   sees `advance_market`).
2. Read `symbol = snapshot.symbol`, `prev_last =
   self._tick_prev_last.get(symbol)`, and `curr_last =
   snapshot.last_price`. `_should_fill_last_cross` **never** reads
   from `self._prev_last_price` directly.
3. Guard invalid data: if `curr_last <= 0` or `symbol is None`, return
   `False`. (Mirrors existing behavior in the other modes; state was
   already short-circuited upstream.)
4. Compute the fill decision:
   - `BUY`: fill when `prev_last is not None and prev_last > limit_price
     and curr_last <= limit_price`.
   - `SELL`: fill when `prev_last is not None and prev_last < limit_price
     and curr_last >= limit_price`.
   - Any other side: raise `ValueError("Invalid order side: ...")` to
     mirror `_should_fill_book_touch`.
5. Sticky last_price (`prev_last == curr_last`) **does not** fire a fill
   in either direction. The strict inequality on `prev_last` ensures
   `prev_last == limit_price` is not treated as crossing on its own.

Multi-tick crossings:

- A jump from `prev_last = 54.10` to `curr_last = 54.30` past a SELL limit
  at `54.20` must fire on the first post-cross tick: `prev_last(54.10) <
  54.20 and curr_last(54.30) >= 54.20` → `True`.
- Symmetric for BUY: `prev_last = 54.30` to `curr_last = 54.10` past a
  BUY @ 54.20: `54.30 > 54.20 and 54.10 <= 54.20` → `True`.
- The algorithm does not interpolate or attribute the fill to an
  intermediate tick. Fill price stays at the limit (issue #117 explicit
  non-goal: "Don't change fill price").

Edge cases the algorithm handles correctly by construction:

- Two consecutive equal `last_price` values straddling no cross: no fill.
- `prev_last` exactly at the limit, `curr_last` strictly past: under
  the strict algorithm chosen here, this is **not** considered a
  fillable cross. For BUY @ `L`: `prev_last == L` makes
  `prev_last > L` false → no fill on this transition. The non-strict
  alternative (`prev_last >= limit_price`) is flagged in "Open
  behavior choice" below; tests #13 and #14 pin the strict-form
  behavior so flipping the operator to non-strict will fail loudly.
  This intentionally matches the issue's `prev_last > limit_price`
  (strict) inequality. The cross will fire on the **next** snapshot
  if `curr_last` later moves back across `L`.
- Snapshots with `bid1`/`ask1` are not consulted by `LAST_CROSS`. The
  mode is intentionally last-price-transition-only.
- **Cross-back ticks (no fired-once guard in the simulator).** Consider
  a SELL @ 54.20 that fires on tick T (`prev=54.10`, `curr=54.30`) and
  then on tick T+1 the price drops back (`prev=54.30`, `curr=54.10`).
  The SELL is not re-evaluated by the simulator because the order
  manager removes it from `active_orders` on the first fill
  (`active_orders.pop(order_id)` at
  `apps/backtest/src/backtest/order_manager.py:296`). The fill
  simulator is intentionally **stateless about which orders have
  already filled** — dedup is the order manager's responsibility.
  Future maintainers MUST NOT add a "fired-once" guard inside
  `_should_fill_last_cross`; doing so would double-track state that
  the order manager already owns and would break the per-order
  independence of `check_fill`.

## Per-tick state advance

The order manager calls `check_fill(order, market)` once per active order
per tick. For N active orders on the same symbol, the simulator must not
advance `prev_last_price` N times — it would inflate `prev_last` to
`curr_last` after the first call and lose the transition signal for the
remaining orders on the same tick.

Critically, the read side (`_should_fill_last_cross`) and the write
side (`self._prev_last_price[symbol] = curr_last`) must be staged so
that the read side **never** observes the just-written `curr_last`
during the same tick. A naive "advance at the end of `_to_snapshot`"
would overwrite `_prev_last_price[symbol]` before `_should_fill` runs,
yielding `prev == curr` on every call and silently disabling fills.

### Two-slot state model

Per-symbol simulator state:

- `self._prev_last_price: dict[str, Decimal]` — the **committed**
  prior-tick `last_price` for each symbol. Written exactly once per
  tick (by `advance_market`). This is the "long-lived" slot,
  surviving across ticks.
- `self._tick_prev_last: dict[str, Decimal | None]` — the **read slot**
  for the in-flight tick. `_should_fill_last_cross` reads from here,
  never from `_prev_last_price`. Populated by `advance_market` on
  every tick from the value previously committed in
  `_prev_last_price` (or `None` if no prior tick was seen for this
  symbol).
- `self._tick_token: dict[str, tuple]` — the last advance-token per
  symbol. Used by `advance_market` to detect re-entry on the same
  tick (idempotency guard). Equality check only.

### Default plan: `advance_market` hook driven by the order manager

Default plan: **the `advance_market(market: TickerEvent)` hook is
called by `BacktestOrderManager.check_fills` once per tick, before
the per-order loop, unconditionally — regardless of whether any
orders are active for the symbol on that tick.** This is the only
correct shape, because the read-only "advance inside `check_fill`"
variant misses the `T → T+1` transition whenever no order is checked
on tick T (grid fully filled one side, gaps between cascades, etc.).
On such ticks `check_fill` is never invoked for the symbol,
`_prev_last_price[symbol]` would never advance, and an order placed
at T+1 would see a stale `prev_last` from before T (or `None`).

`advance_market` lifecycle on every tick:

1. `BacktestOrderManager.check_fills(market)` calls
   `self.fill_simulator.advance_market(market)` as the **first
   statement inside the `isinstance(market, TickerEvent)` branch
   of `check_fills`** — i.e., after the existing input
   normalization (`market is None` → `current_price` coercion at
   `order_manager.py:266-272`) and inside the same branch that
   sets `fill_timestamp = timestamp or market.exchange_ts` and
   `symbol_filter = market.symbol` (`order_manager.py:274-276`),
   placed before the `fills: list[ExecutionEvent] = []`
   accumulator and the active-orders loop. The legacy bare-`Decimal`
   `else`-branch (`order_manager.py:277-283`) never reaches this
   call.
2. `advance_market` normalizes `market` (or accepts a `TickerEvent`
   directly) and extracts `symbol = market.symbol`,
   `curr_last = Decimal(market.last_price)`, and
   `token = (symbol, market.exchange_ts, market.local_ts)`.
3. Guard: if `curr_last <= 0` or `symbol is None`, return
   immediately — no state mutation. Invalid market data is a no-op
   for state.
4. Compare `token` to `self._tick_token.get(symbol)`:
   - **Token mismatch** (first call for this tick):
     a. Stash the prior committed value into the read slot:
        `self._tick_prev_last[symbol] =
        self._prev_last_price.get(symbol)`. This may be `None` if no
        prior tick was seen for this symbol.
     b. Commit the new value: `self._prev_last_price[symbol] =
        curr_last`.
     c. Update the token: `self._tick_token[symbol] = token`.
   - **Token match** (re-entry for the same tick): no-op. The
     idempotency guard makes `advance_market` safe to call more
     than once per tick (defensive; the order manager calls it
     exactly once).
5. The per-order loop then runs as today, calling
   `check_fill(order, market)` once per active order. Inside,
   `_should_fill_last_cross(side, limit_price, snapshot)` reads
   `prev_last = self._tick_prev_last.get(snapshot.symbol)` and
   `curr_last = snapshot.last_price`, then evaluates the strict
   inequality. `_should_fill_last_cross` **never writes** to any
   state slot.

This staging guarantees:

- Every tick the order manager observes advances the simulator,
  including ticks with zero active orders for the symbol — the
  `T → T+1` transition is never lost.
- The first call on a tick stashes the original `prev_last` into the
  read slot **before** committing the new value. The read slot
  retains the cross-tick boundary for the entire per-order loop.
- All N orders on the same tick observe the same
  `(prev_last, curr_last)` pair because they read the same stashed
  read slot.
- Across ticks, the next tick's token mismatches, the read slot is
  re-stashed from the freshly committed `_prev_last_price`, and the
  cycle repeats.

`local_ts` is included in the token as a tiebreaker because Bybit
ticker streams can emit two distinct snapshots sharing the same
`exchange_ts` (the recorder stamps `local_ts` for ordering). Using
`(exchange_ts, local_ts)` avoids dropping the transition signal in
that case and is resilient to non-strict `exchange_ts` monotonicity
(clock skew on the source, recorder merges, or recorder bugs that
emit a regressed `exchange_ts`). The token comparison is
equality-only — we never compare ordering, so monotonicity is not
required.

### Alternative: advance inside `check_fill` (rejected for production replay)

A read-only alternative — perform the stash/commit at the top of
`check_fill` itself, keyed by the same per-tick token — was
considered. It localizes the change to `fill_simulator.py` and
leaves `order_manager.py` untouched. **It is incorrect for
production replay** because `BacktestOrderManager.check_fills` only
calls `check_fill` once per active order (`order_manager.py:292`).
On a tick with zero active orders for the symbol, `check_fill` is
never called, the per-tick token never advances, and the next valid
`check_fill` observes a stale `prev_last` from two-or-more ticks
ago (or `None`). This silently mis-detects or misses fills on grids
that briefly drain one side or between trade cascades.

This alternative may be acceptable for **read-only unit tests** that
drive the simulator directly with a synthetic order sequence, but it
must not be used as the production replay path. The plan keeps it
documented only as a rejected option so reviewers don't propose it
again.

### Mode-gating (optional, low priority)

Optional low-priority optimization: gate the `advance_market` body
on `if self._mode is FillMode.LAST_CROSS: ...` and early-return for
the other three modes. Functionally safe either way because
`STRICT_CROSS`, `TRADE_THROUGH_AT_LIMIT`, and `BOOK_TOUCH` never
read `_tick_prev_last`, but the gate avoids three dict writes
(`_tick_prev_last`, `_prev_last_price`, `_tick_token`) per tick
when the mode is not `LAST_CROSS`. Implement only if the v7
benchmark on `last_cross` shows measurable cost; otherwise defer.

### State-advance preconditions

The stash, the commit, and the token write inside `advance_market`
happen **only** when `curr_last > 0` and `symbol is not None`. A
zero or otherwise non-positive `last_price` is treated as invalid
market data and is a no-op for state and for the fill check. This
guarantees that a stray zero-`last_price` tick cannot poison either
the committed `_prev_last_price` slot or the read-side
`_tick_prev_last` slot for a symbol, and a subsequent valid tick
still sees the original `prev_last`. The ordering inside
`advance_market` is: (1) normalize, (2) check `curr_last > 0` and
`symbol is not None`, (3) compare per-tick token, (4) on mismatch,
stash read slot then commit. The legacy bare-`Decimal` path of
`check_fill` never reaches `advance_market` (the order manager only
invokes `advance_market` for `TickerEvent` inputs); see "Symbol
resolution and legacy bare `Decimal`" below.

## Symbol resolution and legacy bare `Decimal`

The legacy bare-`Decimal` path on `check_fill` (covered by existing tests
and `test_book_touch_falls_back_to_trade_through_on_bare_decimal_*`) has
no symbol and no `exchange_ts`. `LAST_CROSS` needs both.

Required behavior for legacy bare `Decimal` input on `LAST_CROSS`:

- Return `False` (no fill) and do not mutate state. Document this in the
  mode's docstring and the matrix-test row.
- Explicit no-state-mutation contract: when `market` is a bare `Decimal`,
  `_to_snapshot` returns `MarketSnapshot(last_price=market, symbol=None)`
  and `check_fill` does not invoke any state advance — `advance_market`
  is only called by `BacktestOrderManager.check_fills` on
  `TickerEvent` inputs, so the bare-`Decimal` path (which originates
  from the standalone backtest engine and unit tests that drive
  `check_fill` directly with a `Decimal`) never reaches the per-tick
  advance. The simulator MUST NOT update `_prev_last_price` for any
  symbol on the bare-`Decimal` path, MUST NOT update the per-tick
  `(symbol, exchange_ts, local_ts)` token, and MUST NOT read or write
  any entry in `_prev_last_price`. This preserves per-symbol history
  captured from earlier `TickerEvent` calls so that mixed-input call
  sequences (a legal but unusual pattern) cannot corrupt state.
- Rationale: shadow replay and Phase 4 always feed `TickerEvent`; bare
  `Decimal` is the legacy backtest path and is not the target of this
  mode. Falling back to `TRADE_THROUGH_AT_LIMIT` here would re-introduce
  the sticky `last_price` over-fire that motivated this mode in the
  first place.

This means `LAST_CROSS` will behave differently from `BOOK_TOUCH` in the
legacy bare-`Decimal` matrix row, which is an intentional and documented
divergence.

## Tests

Simulator-direct transition tests live in
`apps/backtest/tests/test_fill_simulator.py` (#1-#7, #9-#14). Order-manager
integration tests live in `apps/backtest/tests/test_order_manager.py`
(#8, #8b, #15) because they drive through
`BacktestOrderManager.check_fills` to exercise the `advance_market` hook
end-to-end. Use the existing `_order` and `_ticker` helpers in
test_fill_simulator.py and the order-manager fixtures in
test_order_manager.py. Add a `_ticker_for(symbol, last, ...)` variant
if symbol-aware tests are needed (the existing `_ticker` hard-codes
`"LTCUSDT"`, which is fine for most cases).

### Test driver pattern

**Fixture timestamp discipline (CRITICAL).** Every snapshot in a two-tick
(or N-tick) test MUST carry monotonically distinct `(exchange_ts,
local_ts)` values. The per-tick idempotency guard in `advance_market`
keys on `(symbol, exchange_ts, local_ts)`; two snapshots sharing the
same token cause the second `advance_market` call to no-op, the stash
never runs, and the test silently asserts `False` on every cross. The
existing `_ticker` helper at
`apps/backtest/tests/test_fill_simulator.py:27-43` hard-codes one
`datetime(2026, 5, 11, 4, 49, 58)` for both `exchange_ts` and
`local_ts` — **this is the trap**. Two ways to discipline the
fixtures:

1. Extend the existing `_ticker` helper to accept a
   `tick_index: int = 0` parameter that offsets `exchange_ts` and
   `local_ts` by `timedelta(milliseconds=tick_index)`. Callers in
   two-tick tests pass `tick_index=0` for `ticker_prev` and
   `tick_index=1` for `ticker_curr`.
2. Or have callers pass explicit `exchange_ts` / `local_ts` per
   snapshot.

`_ticker_for(symbol, last, ...)` must follow the same discipline
(accept a `tick_index` or explicit timestamps). Without this,
**every two-tick test (#2-#7, #8, #9, #10, #13, #14, #15) silently
fails**: the second `advance_market` token-matches the first,
`_tick_prev_last[symbol]` stays `None`, and
`_should_fill_last_cross` returns `False` on every cross.

Production replay is protected from this collision by the DB unique
constraint on `(symbol, exchange_ts)` at
`shared/db/src/grid_db/models.py:265-267` plus cursor pagination in
`HistoricalDataProvider`. Tests do not go through that path, so the
discipline is the test author's responsibility.

For `LAST_CROSS`, the prior-tick state lives in the simulator's
`_tick_prev_last` read slot, which is populated by
`simulator.advance_market(ticker)`. **A direct call to
`simulator.check_fill(order, ticker_curr)` that never invokes
`advance_market` leaves `_tick_prev_last[symbol] = None` and always
returns `False`** — that contradicts the pinned expectation in every
transition scenario below.

The default test pattern for the simulator-direct two-snapshot
`(prev_last, curr_last)` transition tests (#2-#7, #9, #10, #12,
#13, #14) is:

```python
simulator.advance_market(ticker_prev)
simulator.advance_market(ticker_curr)
result = simulator.check_fill(order, ticker_curr)
```

#### Order-manager integration driver

Tests #8, #8b, and #15 drive through
`BacktestOrderManager.check_fills(...)` directly, not via
`simulator.advance_market(...)`. Test #8's driver shape is:
construct two active BUY orders on the same symbol, call
`order_manager.check_fills(ticker_prev)` then
`order_manager.check_fills(ticker_curr)`, then assert both orders
fill on the second call. This makes the architectural intent
explicit: the order manager dispatches per-order `check_fill`
under one `advance_market` call per tick. Test #8b extends this
driver by also calling `simulator.advance_market(market)` directly
a second time on the same `TickerEvent` to exercise the
idempotency guard. Test #15 uses the same driver to feed
`check_fills` on ticks with empty `active_orders`.

A convenience helper is recommended to keep tests concise:

```python
def _drive_two_ticks(
    simulator,
    order,
    prev_last,
    curr_last,
    *,
    symbol: str = "LTCUSDT",
):
    ticker_prev = _ticker_for(symbol, prev_last, tick_index=0)
    ticker_curr = _ticker_for(symbol, curr_last, tick_index=1)
    simulator.advance_market(ticker_prev)
    simulator.advance_market(ticker_curr)
    return simulator.check_fill(order, ticker_curr)
```

Test #1 (no prior tick) uses a single-snapshot driver:
`simulator.advance_market(ticker_curr)` then
`simulator.check_fill(order, ticker_curr)`. This mirrors the
order-manager path on a first-ever tick — `_prev_last_price[symbol]`
is empty before the call, `advance_market` stashes `None` into
`_tick_prev_last[symbol]`, and `_should_fill_last_cross` returns
`False` because `prev_last is None`. Applying the default
two-snapshot pattern to test #1 would give it a prior tick (the
first `advance_market(ticker_prev)` would commit `ticker_prev.last`
into `_prev_last_price`, so the second `advance_market` stashes a
real `Decimal` value into the read slot) and defeat the
`prev_last is None` assertion that is the entire point of the
test.

Tests that exercise the order-manager integration explicitly (tests
#8, #8b, and #15) instead drive through
`BacktestOrderManager.check_fills(ticker_prev)` then
`BacktestOrderManager.check_fills(ticker_curr)` so that
`advance_market` fires naturally from the gated call inside
`check_fills`. These two patterns — direct
`simulator.advance_market(...)` driving and order-manager
`check_fills(...)` driving — are the only valid ways to populate
the read slot. Tests #11 (legacy bare `Decimal`) and the
single-snapshot matrix row are the exceptions: they deliberately
do **not** call `advance_market`, since the legacy path is
contractually no-advance.

Required tests (issue #117 lists 6+; this plan calls out the specific
shapes):

1. **No fill without prior tick.** First `check_fill` on a symbol returns
   `False` even if `curr_last` is past the limit. Asserts the
   `prev_last is None` guard.
2. **Sticky last_price does not trigger fill.** Two consecutive ticks
   with `prev_last == curr_last == limit` and `prev_last == curr_last`
   above or below the limit. BUY and SELL cases. Asserts the
   `prev != curr` requirement implicit in the strict inequality.
3. **Single-tick BUY cross.** `prev_last = 54.30`, `curr_last = 54.20`,
   BUY @ 54.20 → fill on the second snapshot.
4. **Single-tick SELL cross.** `prev_last = 54.10`, `curr_last = 54.20`,
   SELL @ 54.20 → fill on the second snapshot.
5. **Multi-tick BUY cross (gap-down).** `prev_last = 54.30`, `curr_last =
   54.10`, BUY @ 54.20 → fill on the first tick post-cross.
6. **Multi-tick SELL cross (gap-up).** `prev_last = 54.10`, `curr_last =
   54.30`, SELL @ 54.20 → fill on the first tick post-cross.
7. **Wrong-direction transition does not fill.** BUY @ 54.20 when
   `prev_last = 54.10` and `curr_last = 54.30` (price moving away from a
   BUY) → no fill. Symmetric for SELL with price moving down.
8. **Two orders on the same symbol/tick both see the cross.** Drive
   the order manager's `check_fills` path with two BUY limits at
   54.20 active on the same `TickerEvent`
   (`prev_last=54.30 -> curr_last=54.10`); both fill. Asserts the
   "advance once per tick" guarantee under the two-slot model —
   `advance_market` runs once inside the
   `isinstance(market, TickerEvent)` branch of `check_fills`,
   `_should_fill_last_cross` never writes, so both orders read the
   same `(prev=54.30, curr=54.10)` pair.
   **8b. `advance_market` is idempotent for the same tick.** After
   the two BUY fills above, call `simulator.advance_market(market)`
   directly a second time with the same `TickerEvent`. Assert the
   token guard rejects re-stash: both slots stay consistent with the
   two-slot model after the redundant call —
   `simulator._prev_last_price[symbol] == Decimal("54.10")`
   (committed slot advanced exactly once to `curr_last`) and
   `simulator._tick_prev_last[symbol] == Decimal("54.30")` (read
   slot still holds the original `prev_last`). Then call
   `check_fill` for a third BUY @ 54.20 on the same `market` and
   assert it also fills (still observes the cross).
9. **Per-symbol isolation.** Two consecutive ticks for `LTCUSDT` followed
   by a first-ever tick for `BTCUSDT`. The `BTCUSDT` check has
   `prev_last is None` and must not fill regardless of price.
10. **Invalid `last_price` (zero) does not advance state.** A tick with
    `curr_last = 0` must not overwrite the previous valid `prev_last`.
    The next valid tick must still see the original `prev_last`. This
    test should sit alongside the existing
    `test_non_positive_last_price_never_fills`.
11. **Legacy bare `Decimal` input returns `False` for `LAST_CROSS`.**
    Documented divergence from `BOOK_TOUCH` and `TRADE_THROUGH_AT_LIMIT`.
12. **Invalid side raises.** Mirror
    `test_book_touch_rejects_invalid_side` for `LAST_CROSS`.
13. **`prev_last == limit, curr_last < limit` (BUY) does not fill on
    the boundary tick.** BUY @ 54.20 with `prev_last = 54.20` and
    `curr_last = 54.10` → no fill. Pins the strict inequality on the
    `prev_last > limit_price` form chosen in "Open behavior choice".
    Then advance one more tick where `curr_last = 54.30` (price moves
    back above): also no fill (no cross-down from prev=54.10 to
    curr=54.30). This prevents silent regression if the open-behavior
    choice flips to non-strict.
14. **`prev_last == limit, curr_last > limit` (SELL) does not fill on
    the boundary tick.** SELL @ 54.20 with `prev_last = 54.20` and
    `curr_last = 54.30` → no fill. Symmetric pin for the SELL strict
    inequality `prev_last < limit_price`. Same follow-up tick assertion
    as test #13.
15. **State advances on ticks with no active orders.** Drive the
    order manager's `check_fills` path with three `TickerEvent`s
    for `LTCUSDT` (`last = 54.30`, `54.10`, `54.10`) while
    `active_orders` is empty. Then add a SELL @ 54.20 to
    `active_orders` and feed a fourth `TickerEvent` with
    `last = 54.30`. Assert the SELL fills on the fourth tick —
    `prev_last` at the moment of evaluation is `54.10` (the
    committed value from the third no-order tick), and
    `curr_last = 54.30`, so the strict-cross SELL fires. This pins
    the F-4-1 fix: `advance_market` runs unconditionally inside
    the `isinstance(market, TickerEvent)` branch of `check_fills`,
    even when no orders are active for the symbol, so the
    `T → T+1` transition is never lost.

The existing `test_fill_mode_matrix` parametrize across
`list(FillMode)` adds an explicit `LAST_CROSS: False` entry per
matrix row (since single-tick snapshots without a prior price never
fill for `LAST_CROSS`). This documents the behavior, keeps the
matrix exhaustive, and is consistent with the test_fill_simulator.py
required-changes bullet above. The dedicated LAST_CROSS tests #1-#15
above are additive — they cover the stateful scenarios the static
single-snapshot matrix can't represent (no-prior-tick guard, two-tick
transitions, multi-tick gaps, legacy bare-Decimal contract, and
order-manager integration).

## Open behavior choice

Issue #117's pseudocode uses `prev_last > limit_price` (strict). A tick
where `prev_last == limit_price` and `curr_last` then crosses past the
limit is therefore not treated as a cross on that snapshot. The plan
follows the issue verbatim. If reviewers prefer the non-strict form
(`prev_last >= limit_price and curr_last < limit_price` for BUY, and
symmetric for SELL), the change is one operator per check and one row in
each of the single-tick tests; flag it during PR review rather than
forking the algorithm here.
