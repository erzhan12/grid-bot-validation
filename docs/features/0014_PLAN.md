# 0014: `--save-events` Flag for Gridbot Live Data Capture

## Description

Add a `--save-events` CLI flag to the gridbot that starts an embedded `EventSaver` alongside the trading bot. This enables simultaneous live trading and data capture in a single process, using the same account credentials and database. The captured data (tickers, executions, orders, positions, wallet) can then be fed to the replay engine for shadow validation — comparing what the bot actually did vs what the backtest engine would have done given the same market data.

## Motivation

Currently, running live data capture requires starting the `event_saver` as a separate process with its own config and credentials. This creates operational complexity and risks credential duplication. The gridbot orchestrator already has access to all the necessary account info and WebSocket streams.

## Files to Change

### 1. `apps/gridbot/src/gridbot/config.py`
- Add `enable_event_saver: bool = False` field to `GridbotConfig`
- This is the config-file equivalent of the CLI flag

### 2. `apps/gridbot/src/gridbot/main.py`
- Add `--save-events` CLI argument to `cli()` parser
- When flag is set, override `config.enable_event_saver = True`
- Pass flag through to `main()` → orchestrator

### 3. `apps/gridbot/src/gridbot/orchestrator.py`
- Import `EventSaver` from `event_saver.main` and `AccountContext` from `event_saver.collectors`
- In `__init__()`: store `enable_event_saver` flag, add `self._event_saver: Optional[EventSaver] = None`
- In `start()`, after account/strategy init and before WebSocket connect:
  - If `enable_event_saver` is True:
    - Create `EventSaverConfig` with:
      - `symbols` = union of all strategy symbols
      - `testnet` = from first account's testnet flag
      - `database_url` = from `GridbotConfig.database_url`
    - Create `EventSaver(config, db)`
    - For each account in `self._config.accounts`, call `event_saver.add_account(AccountContext(...))` mapping:
      - `account_id` → generate or derive UUID from account name
      - `user_id` → generate or derive UUID
      - `run_id` → from `self._run_ids` (created in `_create_run_records()`)
      - `api_key`, `api_secret` → from `AccountConfig`
      - `environment` → `"testnet"` or `"mainnet"` based on `AccountConfig.testnet`
      - `symbols` → strategies for this account
    - Call `await self._event_saver.start()`
- In `stop()`:
  - If `self._event_saver`: call `await self._event_saver.stop()`

### 4. `apps/gridbot/tests/test_orchestrator.py`
- Add test: `test_event_saver_started_when_flag_enabled`
  - Verify `EventSaver.start()` is called when `enable_event_saver=True`
  - Verify `add_account()` is called for each account
- Add test: `test_event_saver_not_started_when_flag_disabled`
  - Verify `EventSaver` is not created when flag is False (default)
- Add test: `test_event_saver_stopped_on_shutdown`
  - Verify `EventSaver.stop()` is called during orchestrator shutdown

## Algorithm

Startup sequence with `--save-events`:

1. Load gridbot config (accounts, strategies, database_url)
2. Init REST clients, executors, reconcilers per account
3. Init strategy runners
4. **Create EventSaver** from gridbot config
5. **Add accounts** to EventSaver with credentials from gridbot config
6. **Start EventSaver** (starts public + private collectors, writers)
7. Perform startup reconciliation
8. Connect gridbot WebSockets (separate from event_saver's own WS connections)
9. Create Run records in DB
10. Start background loops

Shutdown:
1. Stop background loops
2. Stop retry queues
3. Disconnect gridbot WebSockets
4. **Stop EventSaver** (flushes writers, disconnects its WS connections)
5. Update Run records

## Notes

- The `EventSaver` maintains its own WebSocket connections (separate from the orchestrator's). This is intentional — the event_saver needs raw Bybit messages for persistence, while the orchestrator normalizes events for trading logic.
- `run_id` linkage: The orchestrator's `_create_run_records()` currently logs a placeholder. For full persistence, this needs to actually create `Run` rows and pass the UUIDs to `AccountContext.run_id`. This is existing tech debt (not introduced by this feature).
- The `--save-events` flag defaults to off, so existing gridbot behavior is unchanged.
