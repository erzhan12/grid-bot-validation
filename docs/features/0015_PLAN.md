# 0015: Auth Error Cooldown — Per-Strategy Retry Cycles

## Context

When the gridbot hits Bybit auth/permission errors (e.g. invalid API key, wrong permissions), the retry queue keeps hammering the API indefinitely. The bot should detect these non-retryable errors, count consecutive failures per-strategy, and after 5 failures enter a configurable cooldown (default 30 minutes). During cooldown the strategy keeps processing market data but skips order execution. After cooldown, it retries 5 more times. This cycle repeats forever until the error resolves or the bot is manually stopped. A Telegram alert is sent on every cycle. The cooldown duration is configured globally via `auth_cooldown_minutes` in the YAML config.

## Bybit Auth Error Codes

Codes to classify as auth errors:

- `10003` — Invalid API key
- `10004` — Sign error / signature mismatch
- `10005` — Permission denied
- `33004` — API key expired

Source: error strings from `rest_client._check_response()` follow the format `[retCode] retMsg` (e.g. `[10005] Permission denied...`).

## Files and Functions to Change

### `apps/gridbot/src/gridbot/config.py`

- Add `auth_cooldown_minutes: int = 30` field to `GridbotConfig`.
- YAML example:
  ```yaml
  auth_cooldown_minutes: 30  # optional, defaults to 30
  ```

### `apps/gridbot/src/gridbot/executor.py`

- Define `AUTH_ERROR_CODES = {10003, 10004, 10005, 33004}` at module level.
- Add a helper `_is_auth_error(error: str) -> bool` that parses the error string for `[code]` pattern via regex and checks against `AUTH_ERROR_CODES`.
- Add `_auth_failure_count: int = 0` and `_max_auth_failures: int = 5` to `__init__`.
- Add `_auth_cooldown: bool = False` property (read by runner to skip order execution).
- In `execute_place()` and `execute_cancel()` catch blocks: if `_is_auth_error(str(e))`, increment `_auth_failure_count`. If count >= `_max_auth_failures`, set `_auth_cooldown = True` and return `OrderResult(success=False, error=...)` (don't raise — the strategy stays alive).
- On any **successful** call, reset `_auth_failure_count = 0` and `_auth_cooldown = False`.
- Add `reset_auth_cooldown()` method — called by orchestrator after 30-min cooldown expires. Resets `_auth_failure_count = 0` and `_auth_cooldown = False`.

### `apps/gridbot/src/gridbot/runner.py`

- In `_execute_intents()`: before executing, check `self._executor.auth_cooldown`. If `True`, skip execution with a debug log and return early.
- This ensures the strategy keeps processing tickers/positions (grid state stays current) but doesn't attempt any orders during cooldown.

### `apps/gridbot/src/gridbot/orchestrator.py`

- Track cooldown state per strategy: `_auth_cooldown_until: dict[str, datetime] = {}` mapping `strat_id` to when cooldown expires.
- In `_on_ticker()`: after calling `runner.on_ticker()`, check if the runner's executor just entered cooldown (`executor.auth_cooldown` went from `False` to `True`). If so:
  - Set `_auth_cooldown_until[strat_id] = now + config.auth_cooldown_minutes`
  - Send Telegram alert: `"Strategy {strat_id}: {count} consecutive auth errors, entering 30-min cooldown (cycle N)"`
  - Log the cooldown start
- Add cooldown check in the existing health check loop or a dedicated check: when `datetime.now() >= _auth_cooldown_until[strat_id]`, call `executor.reset_auth_cooldown()` and remove from dict. Log: `"Strategy {strat_id}: cooldown expired, resuming order execution"`.

### `apps/gridbot/src/gridbot/retry_queue.py`

- In `process_due()`: check `is_paused` callable before attempting retry. If paused, skip the entire batch.
- Re-check `is_paused` before each individual retry (cooldown may activate mid-batch).
- Accept `is_paused` callable via `__init__` that the orchestrator sets to `lambda: executor.auth_cooldown`.

### Retry queue clearing on cooldown entry

When cooldown activates, the orchestrator clears the strategy's retry queue. This is a deliberate design choice:

- Auth-error retries would fail with the same error and are not useful work.
- After 30+ minutes of cooldown, queued intents have stale prices — executing them is worse than regenerating.
- With `max_elapsed_seconds=30`, non-auth transient retries would expire on their own before any cooldown window matters.
- Post-cooldown recovery relies on the **ticker path** (engine regenerates placements for missing grid levels) and **order-sync loop** (reconciler detects and fixes order state via REST). Both are authoritative and independent of the retry queue.

## Algorithm

1. REST call fails in `executor.execute_place()` or `execute_cancel()`
2. Executor parses error string for `[NNNNN]` pattern via regex
3. If code is in `AUTH_ERROR_CODES`, increment `_auth_failure_count`
4. If `_auth_failure_count >= 5`, set `_auth_cooldown = True`, fire `on_cooldown_entered` callback, return failed result
5. If code is NOT an auth error, return normal `OrderResult(success=False)` (existing behavior)
6. On any successful REST call, reset `_auth_failure_count = 0` and `_auth_cooldown = False`
7. Runner checks `executor.auth_cooldown` before executing intents — skips if `True` (also re-checks mid-batch)
8. Orchestrator receives `on_cooldown_entered` callback, clears retry queue, sets cooldown timer, sends Telegram alert
9. During cooldown: tickers/positions keep flowing (grid state stays current), but no orders are placed or retried
10. After cooldown expires (health check loop), orchestrator calls `executor.reset_auth_cooldown()`
11. Next ticker generates fresh intents at current prices; order-sync loop reconciles any stale state
12. If those 5 attempts also fail, re-enters cooldown (step 4). Cycle repeats forever until success or manual stop.
13. Telegram alert sent on every cycle with cumulative cycle count.
