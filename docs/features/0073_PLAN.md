# 0073 — Real CI workflow (`make test` + `make lint`)

## Context

GitHub issue #176 (P0). The repo has **no CI that gates merge health**.
Existing workflows (`.github/workflows/claude.yml`,
`claude-code-review.yml`, `risk-tier-monitor.yml`) only run Claude /
scheduled jobs — none run `make test` or `make lint` on push/PR.

**Goal:** every PR must pass tests and lint before merge. Add a
`ci.yml` workflow that checks out the repo, installs deps via `uv`,
runs `make test` and `make lint`, and fails the job on any failure.

This plan adds the workflow and documents CI as the source of truth.
It does **not** fix the underlying red tests/lint — those are tracked
in sibling issues #177–#180. The central design decision (Phase 0)
is how the workflow behaves while the repo is still red.

## Phase 0 — Design decision: how CI handles the currently-red repo

The acceptance criteria say the job must **fail on any failure**, but
`make lint` (170 errors) and possibly `make test` are red today.
A hard-required CI added now would block **every** PR, including the
fix PRs for #177–#180. The plan resolves this with a **two-step
rollout**, and the implementer must pick the step that matches reality
at implementation time:

- **Step A (recommended default) — ship CI now with real failing
  checks, not required yet.** Add `ci.yml` with two jobs (`test`,
  `lint`). Both jobs run `make test` / `make lint` and **fail the
  workflow** on non-zero exit — no `continue-on-error`. The workflow
  is simply **not** marked as a required status check in branch
  protection. Failed jobs show as red/annotated on the PR but do not
  block merge until #177–#180 land and the team enables the required
  check. This satisfies the goal (lines 10–12): every job genuinely
  fails on failure; only merge gating is deferred.
- **Step A′ (opt-in only) — advisory green CI via
  `continue-on-error`.** Use **only** if the team explicitly wants the
  overall workflow to stay green while lint (or another job) is red.
  `continue-on-error: true` makes the job **not** fail the workflow,
  so it no longer satisfies the hard-fail goal and must **not** be used
  as the basis for later required-check setup. If used at all, restrict
  it to the `lint` job only (never `test`) and add a `# TODO(#180):
  remove once ruff is clean` comment. Treat as temporary scaffolding,
  not the default rollout.
- **Step B — ship CI as a hard gate after #177–#180 land.** If the
  fixes for #177–#180 have already merged and `make test` + `make lint`
  are both green locally, add `ci.yml` with **no** `continue-on-error`
  and request branch protection so it is required to merge.

The implementer MUST run `make lint` and `make test` locally first.
If green → Step B. If red → Step A (preferred): real failing jobs,
branch protection off. Do **not** put `continue-on-error: true` on the
`test` job under any circumstance — a passing workflow with failing
tests directly undermines "every PR must pass tests before merge." Only
consider Step A′ for `lint` if the team explicitly wants advisory
green CI; otherwise keep lint red and visible via Step A.
Branch-protection (marking the check "required") is a GitHub repo
setting, not a file — flag it to the user as a manual follow-up; the
workflow file alone cannot enforce it.

## Phase 1 — `ci.yml` workflow

Create `.github/workflows/ci.yml` with:

- `name: CI`
- **Triggers** (acceptance: `pull_request` and `push` to `main`):
  ```
  on:
    pull_request:
    push:
      branches: [main]
  ```
- **Concurrency** — cancel superseded runs on the same ref to save CI
  minutes (justified for a P0 "make green" loop with frequent pushes):
  ```
  concurrency:
    group: ci-${{ github.ref }}
    cancel-in-progress: true
  ```
- **Permissions** — minimal: `contents: read` (no secrets needed; tests
  are hermetic and do not hit Bybit).
- **Two jobs**, both `runs-on: ubuntu-latest`, run in parallel:
  1. `test` — steps:
     - `actions/checkout@v4`
     - `astral-sh/setup-uv@v4` (enable built-in cache:
       `with: enable-cache: true` — caches the uv download/global
       cache keyed on `uv.lock`, the minimal-but-correct caching choice;
       no separate `actions/cache` step needed)
     - `uv python install 3.12`
     - `uv sync`
     - `make test`
  2. `lint` — steps: checkout, setup-uv (cache), `uv python install
     3.12`, `uv sync`, `make lint`.
     - Step A (default): no `continue-on-error`; lint failures are real
       red jobs. Merge is unblocked only because the check is not
       required in branch protection.
     - Step A′ (opt-in only): add `continue-on-error: true` to this
       job only, with `# TODO(#180): remove once ruff is clean`. Never
       apply `continue-on-error` to `test`.
- **Single Python version (3.12), no matrix.** Justification: the repo
  pins `3.12` in `.python-version`; the live bot and all dev run on
  3.12. A matrix adds cost/flakiness with no payoff for a P0 "make
  green" gate. (`requires-python >=3.11` is a floor, not a test target.)
- Job failure propagates from `make`'s non-zero exit, satisfying "fails
  the job on any failure".

### Why two jobs instead of one
Splitting `test` and `lint` gives independent pass/fail signals: under
Step A, a red lint job does not hide a green or red `test` job (and
vice-versa). Under Step A′, `continue-on-error` on `lint` only must not
be applied to `test`, so test failures always fail the workflow. Both
jobs can run concurrently, so wall-clock is not worse.

## What could break / risks

- **Network/secrets:** none required — tests are hermetic; CI uses no
  Bybit keys. If any test in the `make test` packages reaches the
  network in CI, mark it skip/`pytest.mark.skipif` (note as a follow-up;
  not expected based on research).
- **uv cache key drift:** `setup-uv` cache keys on `uv.lock`; stale
  cache is auto-invalidated on lock change.
- **`make test` runtime:** runs ~9 package suites + integration; if it
  exceeds the default 6h GitHub timeout (it won't), add a
  `timeout-minutes`. Recommend `timeout-minutes: 30` on the `test` job
  as a cheap safety bound.
- **Branch protection not enforceable from the file** — must be set in
  GitHub repo settings; flag to user.
