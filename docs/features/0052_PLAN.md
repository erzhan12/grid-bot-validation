# 0052 â€” Fix DB-backed grid_state_snapshot seed in replay (cross-run lookup)

## Context

Feature 0047 persisted gridbot's grid state to the `grid_state_snapshots`
table and added `load_grid_state_from_snapshots()` in replay so seeded
runs could pick up the grid state from the DB instead of the legacy
JSON file at `seed.grid_state_path`.

The loader filters the lookup by `run_id`. That assumption is wrong:

- The **recorder** creates a NEW `run_id` (`run_type='recording'`) on
  each start.
- **Gridbot** creates its OWN `run_id` (`run_type='live'`) at process
  start, completely independent of the recorder. Gridbot's
  `GridStateWriter` writes `grid_state_snapshots` under its own live
  `run_id`.
- Replay receives the recorder's `run_id` (via `seed.at_ts` window /
  the recording it is replaying) and uses it to query
  `GridStateSnapshotRepository.get_at_or_before(run_id, ...)`. The
  WHERE predicate `run_id == <recorder_run_id>` matches **zero** rows
  because gridbot wrote under a different `run_id`.
- The engine sees `grid_seed is None`, falls back to
  `seed.grid_state_path` JSON (which has been empty/stale since
  2026-05-20), then to a fresh blank build. All historical replays
  show `grid seed source=file`, confirming the DB path has never
  actually fired in production â€” v8's 100% match rate held only
  because the JSON file was manually refreshed before each run.

**Fix strategy (chosen):** drop `run_id` from the snapshot lookup
query. The desired semantics is "what was gridbot's grid state at this
wall-clock time for this `(account_id, strat_id)`?" â€” independent of
which process wrote it.

The writer side (gridbot `GridStateWriter`) and the schema stay
unchanged. The `run_id` column itself stays on the table and on the
ORM model; only the **lookup** predicate changes.

## Algorithm â€” post-fix lookup

For each replay seed step:

1. `engine._load_seed` opens a DB session and calls
   `load_grid_state_from_snapshots(db_session, account_id,
   strat_id, symbol, at_ts, expected_step, expected_count)` â€”
   no `run_id`, with `symbol=config.symbol`.
2. Loader checks `GridStateSnapshot.__tablename__` exists (pre-0047
   DB guard â€” unchanged).
3. Loader calls
   `GridStateSnapshotRepository(db_session).get_at_or_before(
   account_id, strat_id, symbol, at_ts)`.
4. Repo executes:
   `SELECT * FROM grid_state_snapshots
    WHERE account_id = :a AND strat_id = :s AND symbol = :sym
      AND exchange_ts <= :t
    ORDER BY exchange_ts DESC, id DESC LIMIT 1`.
5. If `row is None` â†’ INFO `"no grid snapshot at-or-before <ts>"`,
   return `None` (engine falls back to file path â†’ fresh).
6. If `row.grid_count != expected_count` or
   `Decimal(str(row.grid_step)) != Decimal(str(expected_step))` â†’
   INFO with the mismatch, return `None` (file fallback). **No
   success-path log fires here.**
7. Else (success path only, after both mismatch guards): emit new
   INFO
   `"<strat>: grid snapshot loaded from run_id=<row.run_id>
    exchange_ts=<row.exchange_ts>"` and return
   `GridStateSeed(strat_id, row.grid_json, float(row.grid_step),
   int(row.grid_count))`.

`GridStateSnapshotRepository.get_latest` is **not** touched by this
fix: it remains per-`run_id` because the gridbot writer's dedupe
gate and the orchestrator bootstrap probe (issue #108) both rely
on per-run scoping.

## Deployability

- Recorder and gridbot keep running. No schema / writer changes.
- Replay is a separate process invoked offline against the recorded
  DB; updating it does not affect either live process.
- Backwards compatibility:
  - Old DBs without the `grid_state_snapshots` table â†’ loader returns
    `None` via the existing `sa_inspect(...).has_table(...)` guard;
    engine falls back to `seed.grid_state_path`.
  - DBs with snapshots written under any `run_id`(s) â†’ first
    in-window row by `exchange_ts DESC, id DESC` is returned.
