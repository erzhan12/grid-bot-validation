# 0074 — Fix integration test for `time_in_force=GTC` (issue #177, P0)

## Context

Feature 0066 (issue #159) added maker-only (`PostOnly`) support to the
executor. `IntentExecutor.execute_place` in
`apps/gridbot/src/gridbot/executor.py` now unconditionally threads a
`time_in_force` kwarg into `BybitRestClient.place_order`:

```
time_in_force="PostOnly" if intent.post_only else "GTC"
```

(`executor.py:273-285`, the `time_in_force=...` line is `executor.py:284`.)

The integration test
`tests/integration/test_engine_to_executor.py::TestExecutorRESTPayloadMapping::test_place_intent_maps_to_rest_params`
asserts the **full** `place_order` kwargs dict with exact equality
(`assert call_kwargs == { ... }` at `tests/integration/test_engine_to_executor.py:207-216`).
That expected dict omits `time_in_force`, so the actual call — which now
includes `"time_in_force": "GTC"` — no longer matches and the test fails.

This is a P0 "make repository green" task (red CI on `main` per issue #177).
The fix is confined to the one integration test file; production code in
`executor.py` is correct and must not change.

## Reproduction

```
uv run pytest tests/integration/test_engine_to_executor.py::TestExecutorRESTPayloadMapping::test_place_intent_maps_to_rest_params -q
```

Currently fails on the dict-equality assertion (missing `time_in_force` key).

## Files and changes

### `tests/integration/test_engine_to_executor.py` (only file changed)

`test_place_intent_maps_to_rest_params` (method at `:192`, expected dict at
`:207-216`):

- The intent used is `place_intents[0]` from `_build_intents` (`:178-190`),
  built by `GridEngine.on_event` with `post_only` unset (default `False`),
  so the live `time_in_force` is `"GTC"`.
- Add the missing key to the asserted `call_kwargs` dict so exact equality
  holds:
  `"time_in_force": "GTC"` — default (non-post-only), feature 0066.
- Keep every existing key (`symbol`, `side`, `order_type`, `qty`, `price`,
  `reduce_only`, `position_idx`, `order_link_id`) and the
  `order_link_id`-suffix assertions (`:217-221`) unchanged.

This mirrors the already-green unit-test expectations in
`apps/gridbot/tests/test_executor.py::TestExecutorPlaceOrder::test_place_order_success`
(`:238-247`, which asserts `"time_in_force": "GTC"`) and
`test_place_order_short_direction` (`:272-281`).

### Optional companion assertion (post_only -> PostOnly)

The issue suggests an optional companion assertion for
`post_only=True -> time_in_force="PostOnly"`. This mapping is **already**
covered at the unit level by
`apps/gridbot/tests/test_executor.py::TestExecutorPlaceOrder::test_post_only_intent_passes_postonly_tif`
(`:286-295`) and the GTC default by `test_default_intent_passes_gtc_tif`
(`:297-301`).

If an integration-level companion is added, it belongs in
`TestExecutorRESTPayloadMapping` as a new method that constructs a
`PlaceLimitIntent` with `post_only=True`, runs it through a non-shadow
`IntentExecutor` with a `MagicMock` `rest_client`, and asserts
`mock_rest.place_order.call_args.kwargs["time_in_force"] == "PostOnly"`.
Note `_build_intents` produces only default (non-post-only) grid intents, so
such a test must build the intent directly via `PlaceLimitIntent.create(...,
post_only=True)` rather than reuse the helper. This is optional and may be
skipped since the unit suite already proves the mapping.
