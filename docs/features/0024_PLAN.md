# 0024_PLAN — Active WS reconnect with socket-level health probe

## Description

Today the bot relies entirely on pybit's internal reconnect logic for WebSocket recovery. Our wrapper (`packages/bybit_adapter/src/bybit_adapter/ws_client.py`) only **passively detects** disconnection: a heartbeat thread fires a `WARNING` after 30 s of message silence and emits an `on_disconnect` callback (`ws_client.py:237-255` for public, `488-506` for private). The wrapper never calls `exit() + connect()` itself, never inspects the underlying socket's TCP state, and never explicitly resubscribes. `Detected reconnection` is also passive — it logs only when the **next** message arrives after a detected gap (`ws_client.py:267-272` / `518-524`), so the printed gap is "time until next push," not "time pybit took to reconnect."

Live evidence on **mainnet** (`/tmp/gridbot.log` 2026-05-03 01:58 — 02:19): three disconnect / reconnection cycles within ~21 minutes, with 6–15 minute gaps between `Detected disconnection` WARN and `Detected reconnection` INFO. On a liquid mainnet symbol (LTCUSDT pushes ticker 5–20× per second), 30 s of silence is a real fault — either a dead TCP socket pybit hasn't noticed, a Bybit-side rate-limit on our connect ID, or a frozen NAT/firewall.

User selections from prior conversation (verbatim):
- "это mainnet" — fix targets mainnet behavior, not testnet quiet-market noise.
- bbu2 reference is preferred ("посмотри как сделано на bbu2") — copy the **`_ensure_*_connection`** pattern from `bbu_reference/bbu2-master/bybit_api_usdt.py:179-219`: rate-limited socket-level poll + active `exit() + connect()` reset.
- Hybrid model: keep our existing 30 s message-gap detector as a **secondary** signal for the "socket alive, server silent" failure mode that bbu2's TCP-level check cannot catch.

## Context

- Disconnect detection today (passive WARN only): `packages/bybit_adapter/src/bybit_adapter/ws_client.py:220-255` (public `_heartbeat_loop`), `:471-506` (private same).
- Reconnection log today (passive, fires on next message): `:257-280` (public `_update_last_message_ts`), `:508-531` (private same).
- Wrapper's `is_connected()` is **state-flag based**, not socket-level: `:178-181` (public), `:411-414` (private). Returns True until something flips `_state.is_connected` to False — which only happens on explicit `disconnect()`. A dead TCP socket leaves the flag stuck at True.
- pybit's native socket-level check: `.venv/lib/python3.12/site-packages/pybit/_websocket_stream.py:103-110` — `self.ws.sock.connected` (TCP-level boolean).
- pybit reconnect loop: `_websocket_stream.py:112-186` (`_connect`). Default `retries=10`; after 10 failures pybit raises `WebSocketTimeoutException` and stops — `retries=0` selects `infinitely_reconnect=True` instead (`:142-145`).
- pybit ping cadence: defaults `ping_interval=20`, `ping_timeout=10` (`:163-164`). We don't override.
- Orchestrator main-loop tick (where periodic checks live, bbu2 timestamp-gating pattern): `apps/gridbot/src/gridbot/orchestrator.py:292-424` (`_tick`); existing schedulers `_next_health_check`, `_next_position_check` etc.
- `on_disconnect` / `on_reconnect` callback hooks already accepted by the WS clients (`:90-91` public, `:318-319` private). Currently unwired in orchestrator — no consumer.

## Reference

- **bbu2 active-reconnect pattern**: `bbu_reference/bbu2-master/bybit_api_usdt.py:179-219` — `_reset_ws_public`, `_reset_ws_private`, `_ensure_public_connection`, `_ensure_private_connection`. Rate-limited (`ENSURE_SOCKET_INTERVAL = 10` s); calls pybit's native `is_connected()`; on failure does `ws.exit() + connect_ws_*()` in a `while True` with `time.sleep(1)`.
- **bbu2 hot-path call sites**: `bybit_api_usdt.py:222` (every `get_last_close`), `:230` (every `get_last_filled_order`), `:420` (every `__get_position_status`). Ensures the socket is healthy on every read.
- **pybit `retries=0` semantics**: `_websocket_stream.py:142-145` — `if retries == 0: infinitely_reconnect = True`. Eliminates the "give up after 10 attempts" failure mode.
- **Existing tests for the wrapper**: `packages/bybit_adapter/tests/test_ws_client.py` — `TestPublicWebSocketClientDisconnectDetection`, mirror class for private. Mocks `pybit.unified_trading.WebSocket` via `@patch("bybit_adapter.ws_client.WebSocket")`.

## Algorithm

The fix is the bbu2 pattern, adapted to our wrapper architecture (per-account WS clients held by orchestrator) and combined with the existing message-gap detector as a fallback.

### Step 1 — expose pybit's socket-level health on our wrapper

Add `is_socket_alive()` to both `PublicWebSocketClient` and `PrivateWebSocketClient`:
- Returns `False` if `self._ws is None`.
- Otherwise returns `self._ws.is_connected()` — pybit's native check (`self.ws.sock.connected` boolean).
- Holds `self._lock` for the duration to avoid racing with `connect()` / `_disconnect_internal()`.

This is **distinct** from the existing `is_connected()`, which we keep but rename in docstring as "logical connected flag" (whether `connect()` was called and `disconnect()` hasn't been). New name `is_socket_alive()` makes the TCP-level intent explicit.

### Step 2 — `reset()` method

Add `reset()` to both clients (public & private):
1. Stop heartbeat watchdog (`_stop_heartbeat_watchdog()`), to prevent a stale `_detected_disconnect=True` flag from racing with the new connection.
2. Acquire `self._lock` and call `_disconnect_internal()` — closes existing socket via `self._ws.exit()` (already wraps exception per `:151-161`).
3. Release lock, call `connect()` — re-establishes socket and re-subscribes to all topics. `connect()` already re-subscribes from `self.symbols` (public) or from the truthy `self.on_*` callbacks (private), so subscription is automatically restored.
4. Log `INFO` "WebSocket {public|private} reset (reconnect attempt)" with the account name where applicable.

The order — stop heartbeat → disconnect → connect (which restarts heartbeat) — matches `disconnect()` semantics today and avoids deadlock with the heartbeat thread holding `self._lock`.

### Step 3 — pybit kwargs tuning

In both `connect()` methods, pass `retries=0` to the `pybit.unified_trading.WebSocket(...)` constructor. By the table at `_websocket_stream.py:142-145`, this makes pybit's internal reconnect infinite — eliminating the 10-attempt cliff at which pybit raises `WebSocketTimeoutException` and stops trying. Optionally also set `ping_interval=10` and `ping_timeout=5` for faster pybit-side dead-socket detection — see "Out of scope" for why we're conservative here.

### Step 4 — orchestrator-side periodic health probe (primary signal)

In `Orchestrator.__init__`, add a new scheduler timestamp `_next_ws_health_check: float = 0.0` and a constant `_WS_HEALTH_CHECK_INTERVAL = 10.0` (matches bbu2's `ENSURE_SOCKET_INTERVAL`).

In `Orchestrator._tick()` periodic-checks block (after `_next_health_check`, before `_next_order_sync` — line 386 area in current `orchestrator.py`), add:

1. If `now < self._next_ws_health_check`: skip.
2. Else: advance `self._next_ws_health_check = now + _WS_HEALTH_CHECK_INTERVAL` BEFORE the call (matches existing pattern of advancing the timestamp before invocation so a hung call cannot retry every tick).
3. Iterate `self._public_ws.values()` and `self._private_ws.values()`. For each client:
   - If `client.is_socket_alive()` is True: continue.
   - Else: log `WARNING` "WS socket dead for {account_name}/{public|private}; resetting", then call `client.reset()`.
4. Wrap the per-client check in try/except so one stuck reset does not abort the iteration over remaining accounts. Use the same `_notifier.alert_exception("ws_health_check", e, error_key=...)` pattern as the existing periodic checks.

### Step 5 — wire the message-gap detector to also trigger reset (secondary signal)

The existing 30 s message-gap detector raises `on_disconnect` callback. Today no consumer wires this in the orchestrator. Add an orchestrator method `_on_ws_disconnect(account_name: str, kind: str, disconnected_at: datetime)` that:
1. Logs `WARNING` "WS message gap detected for {account_name}/{kind}; resetting".
2. Calls `client.reset()` for the corresponding client.

Wire the callback at WS-client construction sites (search `Orchestrator._init_account` for `PublicWebSocketClient(...)` and `PrivateWebSocketClient(...)` constructors and pass `on_disconnect=lambda ts: self._on_ws_disconnect(account_name, "public", ts)` / `..."private"...`).

`on_disconnect` runs on the heartbeat thread (per `ws_client.py:251-255`). `reset()` does its own locking. Calling `reset()` from a callback that itself runs inside the heartbeat thread requires `_stop_heartbeat_watchdog()` to be safe to call from inside the heartbeat thread — verify by inspection that `threading.Event.set()` plus `Thread.join(timeout=2.0)` does not deadlock when self-joining; if it does, dispatch the reset to a one-shot worker thread instead. This is a known sharp edge — flag in implementation review.

### Step 6 — relationship between primary and secondary signals

Both signals call the same `reset()`. Race conditions:
- Periodic check sees `is_socket_alive() == False`, calls `reset()`.
- Concurrently the heartbeat thread fires `on_disconnect` and calls `reset()` again.

`reset()` is idempotent only if it tolerates being called when `_ws is None` (already handled by `_disconnect_internal()` at `:152-153`). The `connect()` step already handles "already connected" by calling `_disconnect_internal()` first (`:107-109`). So back-to-back resets degrade gracefully into a single re-establishment plus a no-op.

To dampen storms, the primary check is rate-limited to once per 10 s; the secondary fires at most once per 30 s of silence (heartbeat threshold). Worst case is two resets within a single second around a real disconnect — acceptable.

### Step 7 — keep `is_connected()` semantics (do not break existing callers)

The existing `is_connected()` (`:178-181` / `:411-414`) is referenced by the orchestrator and by tests. Do NOT change its return value. Add `is_socket_alive()` as a new method. Test cases that previously asserted `is_connected()` true after `connect()` still hold; new tests assert `is_socket_alive()` directly when verifying the new reset path.

## Interaction with existing logic

- **`Detected disconnection` WARN line stays** — it remains the secondary signal. Its semantics shift slightly: today it's "we'll passively wait"; after this change, it's "and we'll actively reset right now." The log line itself doesn't need to change.
- **`Detected reconnection` INFO line stays** — fires when the new socket actually delivers a message. After active reset its meaning becomes accurate ("pybit reconnected and traffic resumed"), not "we got the next push from a possibly-still-broken connection."
- **`reconnect_count` counter** at `_state.reconnect_count` (`:50`, `:521`) keeps incrementing on each detected reconnection regardless of whether reset was active or passive. No change in counter semantics.
- **`on_disconnect` / `on_reconnect` callbacks** become wired in the orchestrator. Tests that previously left these as `None` don't break — the new wiring is at construction time inside `_init_account`, not in the WS-client class itself.
- **`pybit retries=0`**: the only behavioral change is that pybit no longer raises `WebSocketTimeoutException` after 10 attempts. Existing pybit-internal reconnect logic still runs; we just don't let it give up. Combined with our Step 4 active reset every 10 s, this means: even if pybit's internal reconnect is broken, we override it 10 s later from main-loop side.
- **Feature 0023 (WS position drain)**: orthogonal. The new health check runs as another periodic gate alongside `_next_position_check` / `_next_health_check`. Position-drain logic does not interact with WS-reset path.

## Out of scope

- **`ping_interval=10, ping_timeout=5` overrides** for pybit. Considered, but pybit defaults (20/10) are tuned for Bybit's published `ping/pong` policy; tightening them risks false-positive reconnects. Revisit only if Step 4's 10 s socket polling proves insufficient.
- **`disconnect_threshold` reduction** below 30 s. Today's 30 s is a deliberate safety margin. Reducing it would make the secondary signal noisier without proportional benefit, since the primary signal (10 s polling) is already faster.
- **Telegram/notifier alert escalation** for "disconnect >60 s without reconnect". Worth doing eventually, but separable. Out of scope here so the patch stays small.
- **Per-account WS-health rotation** like `_fetch_positions_rotation_tick` does for positions. Not needed: the WS-health check is a `is_socket_alive()` boolean, not a REST round-trip; checking N accounts in one tick is microseconds.
- **Mutex / leader-election so primary and secondary signals don't both call `reset()` near-simultaneously.** Step 6 explains why back-to-back resets degrade gracefully; explicit coordination is unnecessary complexity.
- **Dropping the existing message-gap detector entirely.** It catches the "socket alive, server silent" failure mode that pybit's TCP check cannot. Keep both.
