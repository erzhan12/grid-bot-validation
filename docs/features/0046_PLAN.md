# Feature 0046 — Throttle `Same-order error active` WARNING flood during SAME ORDER soft-block

## Context

When the SAME ORDER soft-block latches with a non-auto-clearing verdict (typically `REAL_DUPLICATE`, or `UNKNOWN` when REST cross-check failed), `GridBotRunner.on_ticker` emits a `WARNING gridbot.runner: <strat>: Same-order error active, skipping order placement` line on every engine tick that enters the placement-gate. Ticker cadence is ~100–400 ms.

Incident 2026-05-15: soft-block latched at 22:25:30 after `REAL_DUPLICATE`; over the next ~1h 6min the bot emitted ~12,040 such WARNINGs (`gridbot-health` window 22:14→23:15 showed 9,158 lines vs steady-state ~600/h). This floods `/tmp/gridbot.log`, hides real fills/reconciler messages, and skews log-volume-based alerting.

Feature 0031 added the soft-block + REST-verified auto-clear machinery (`WS_GLITCH_SUSPECTED` clears, `REAL_DUPLICATE` stays latched). This feature is operational hygiene layered on top: keep the first WARNING loud, then throttle.

Issue reference: GitHub #94.

## Behavioural requirements

1. **First emission after entering blocked state**: WARNING fires unchanged — same level, same message body. Alert-worthy.
2. **Subsequent placement-gate hits within a 60 s window**: do not log; increment an in-memory suppressed-count counter.
3. **Heartbeat re-emit**: while still blocked, re-emit at most every 60 s. The re-emitted message includes the suppressed count since the previous emission, e.g. `Same-order error active, skipping order placement (suppressed N since last)`. Resets the counter on emission.
4. **Block-cleared transition (`_same_order_error: True → False`)**: emit one `INFO` line summarising the suppression window, e.g. `Same-order error cleared (suppressed N WARNINGs since)`, **only when at least one WARNING was emitted during the latched period** (i.e. `_same_order_warn_last_ts is not None`). When the block latched and cleared without `on_ticker` ever reaching the placement gate, no WARNINGs were emitted, so the recovery INFO is also skipped — the recovery INFO closes out a warning flood, not a state transition. Counter and last-emit timestamp reset on transition either way.
5. **Re-entry after a cleared block**: the next True transition restarts at step 1 (loud first WARNING again).

Throttle window constant: **60 seconds**. Defined as a module-level constant (sibling to existing `_SAME_ORDER_TIME_WINDOW_SEC`, `_SAME_ORDER_DEDUP_TTL_SEC`) so tests can monkeypatch it.

## Files & functions to change

### `apps/gridbot/src/gridbot/runner.py`

- **Module-level constants** (near existing `_SAME_ORDER_TIME_WINDOW_SEC` / `_SAME_ORDER_DEDUP_TTL_SEC`): add `_SAME_ORDER_WARN_THROTTLE_SEC = 60.0`.
- **`GridBotRunner.__init__`** (around lines 220–243, beside `self._same_order_error`):
  - `self._same_order_warn_last_ts: Optional[datetime] = None` — timestamp of the most recent emitted WARNING for the current latched block; `None` while not blocked or right after clear.
  - `self._same_order_warn_suppressed: int = 0` — count of suppressed warnings since the last emission within the current block.
- **`GridBotRunner.on_ticker`** (lines 371–381): replace the unconditional `logger.warning(...)` inside the `if self._same_order_error:` branch with the throttle decision:
  - If `_same_order_warn_last_ts is None` → first occurrence: emit WARNING unchanged, set `_same_order_warn_last_ts = datetime.now(UTC)`, leave suppressed counter at 0.
  - Else, compute elapsed = `now - _same_order_warn_last_ts`.
    - `elapsed >= _SAME_ORDER_WARN_THROTTLE_SEC` → emit WARNING with suffix `(suppressed N since last)` where N = current `_same_order_warn_suppressed`; reset suppressed counter to 0; update `_same_order_warn_last_ts = now`.
    - Else → increment `_same_order_warn_suppressed`; do not log.
  - Return path is unchanged (still `return intents`).
- **Clear transition handling**: the soft-block is set in two places (`_check_same_orders_side` line 1274 via dedup re-establish, line 1308 first-trigger path) and cleared in three contexts:
  - **A.** External: `reset_same_order_error` (line 274–281), called from WebSocket reconnect / operator paths.
  - **B.** Clean-fill auto-clear: inside `_check_same_orders` (line 1155), where `self._same_order_error = False` is set unconditionally, then per-side checks may re-set it. The True→False net transition happens when no duplicate was re-established. This is the `test_same_order_error_auto_clears_on_clean_execution` path (test line 2009) and the dominant in-flight clear path during normal operation — it **must** emit the recovery INFO and reset throttle state, otherwise the next latch is misclassified as a continuation and the loud-first WARNING is suppressed (P1).
  - **C.** REST verdict auto-clear: `_diagnostic_rest_check_executions` line 1421–1425 (`WS_GLITCH_SUSPECTED`), which already logs its own verdict-bearing INFO immediately before calling `reset_same_order_error()`.

  One small private helper plus a parameterised public reset, keeping all "clear everything" side-effects centralised in one place:

  1. `_emit_clear_recovery_if_needed(self) -> None` — emits one `logger.info(f"{self.strat_id}: Same-order error cleared (suppressed N WARNINGs since)")` only when `_same_order_warn_last_ts is not None`, then resets `_same_order_warn_last_ts = None` and `_same_order_warn_suppressed = 0`. **Does not touch `_same_order_error` or the execution buffers.** Idempotent on subsequent calls (no-ops when state already reset). Used as the canonical recovery-INFO formatter so the message body is defined in exactly one location.

  Wiring:

  - **`reset_same_order_error(emit_recovery_info: bool = True) -> None`** (line 274–281, context A): becomes the single owner of "clear flag + clear execution buffers + clear throttle state". Behaviour: snapshot `was_set = self._same_order_error`; set `_same_order_error = False`; `_recent_executions_long.clear()`; `_recent_executions_short.clear()`; if `was_set and emit_recovery_info`, call `_emit_clear_recovery_if_needed()`; otherwise reset throttle state inline (`_same_order_warn_last_ts = None`, `_same_order_warn_suppressed = 0`). External callers (e.g. WebSocket reconnect path) keep the default `emit_recovery_info=True` and continue to work unchanged. The parameter exists so the REST verdict path (context C) can suppress the helper's INFO and emit its own combined verdict+suppressed INFO instead — without duplicating reset side-effects.
  - **`_check_same_orders`** (line 1091, context B): the existing line 1155 reset is **not** a full clear — it intentionally leaves the execution buffers intact so the per-side checks can re-establish the flag on the same call. It cannot route through `reset_same_order_error` (which clears buffers). Instead: snapshot `was_set = self._same_order_error` at the top of the method, before line 1155's unconditional reset. After both side-checks have run (current line 1159), if `was_set and not self._same_order_error` call `_emit_clear_recovery_if_needed()`. Line 1155 stays a direct assignment. This addresses P1; the helper's `_same_order_warn_last_ts is not None` guard inside `_emit_clear_recovery_if_needed` means silent latch + silent clear (block latched and cleared before `on_ticker` ever entered the placement gate) emits no INFO, per the relaxed acceptance criterion above.
  - **`_diagnostic_rest_check_executions`** (line 1421–1425, context C): to avoid emitting two recovery INFOs on the WS-glitch auto-clear path (P3), merge them into one and reuse `reset_same_order_error` for state cleanup. Edits:
    - Snapshot the suppressed counter into a local and check whether throttle state is populated **before** the reset.
    - Amend the existing INFO format to append `; suppressed N WARNINGs since` when throttle state was populated; leave the line unchanged otherwise.
    - Replace `self.reset_same_order_error()` at line 1425 with `self.reset_same_order_error(emit_recovery_info=False)`. The reset now performs flag + buffer + throttle-state cleanup, but the helper INFO is suppressed because this path has already emitted the combined verdict+suppressed INFO above. Net: REST-auto-clear path emits exactly one INFO that carries both verdict and suppressed-count context, and reset side-effects remain centralised in `reset_same_order_error`.

## Algorithm — throttle decision in `on_ticker`

```
on_ticker entry, _same_order_error == True:
  now = datetime.now(UTC)
  if _same_order_warn_last_ts is None:
    log WARNING (loud, no suffix)
    _same_order_warn_last_ts = now
    # _same_order_warn_suppressed stays 0
  else:
    elapsed_sec = (now - _same_order_warn_last_ts).total_seconds()
    if elapsed_sec >= _SAME_ORDER_WARN_THROTTLE_SEC:
      log WARNING with suffix "(suppressed {_same_order_warn_suppressed} since last)"
      _same_order_warn_last_ts = now
      _same_order_warn_suppressed = 0
    else:
      _same_order_warn_suppressed += 1
      # no log
  return intents
```

Recovery emission helper (`_emit_clear_recovery_if_needed` — flag-untouching, buffer-untouching):

```
if _same_order_warn_last_ts is not None:
  log INFO "Same-order error cleared (suppressed {_same_order_warn_suppressed} WARNINGs since)"
  _same_order_warn_last_ts = None
  _same_order_warn_suppressed = 0
# else: idempotent no-op (silent latch + silent clear → no recovery INFO)
```

Parameterised reset (`reset_same_order_error(emit_recovery_info=True)` — context A; also reused by context C):

```
was_set = self._same_order_error
self._same_order_error = False
_recent_executions_long.clear()
_recent_executions_short.clear()
if was_set and emit_recovery_info:
  _emit_clear_recovery_if_needed()
else:
  _same_order_warn_last_ts = None
  _same_order_warn_suppressed = 0
```

Clean-fill auto-clear inside `_check_same_orders` (context B — does NOT clear buffers):

```
was_set = self._same_order_error            # snapshot before line 1155 reset
self._same_order_error = False              # existing line 1155 (unchanged)
_check_same_orders_side(long)
if self._same_order_error: return           # existing early-return
_check_same_orders_side(short)
if was_set and not self._same_order_error:  # True→False net transition
  _emit_clear_recovery_if_needed()
```

REST verdict auto-clear inside `_diagnostic_rest_check_executions` (context C, `WS_GLITCH_SUSPECTED` branch):

```
suffix = ""
if _same_order_warn_last_ts is not None:
  suffix = f"; suppressed {_same_order_warn_suppressed} WARNINGs since"
log INFO f"{strat_id}: SAME ORDER soft-block auto-cleared "
         f"(WS glitch confirmed by REST cross-check; verdict={verdict}){suffix}"
self.reset_same_order_error(emit_recovery_info=False)
# Single combined INFO above; reset handles flag + buffers + throttle state.
```

## Out of scope

- Verdict classification changes (covered by issue #93).
- Throttling other SAME ORDER log lines (the loud `SAME ORDER ERROR` first-trigger ERROR is already dedup-gated via `_same_order_dedup_cache` per feature 0031).
- Changing the soft-block escape semantics — `REAL_DUPLICATE` still stays latched until external reset.
- Persisting the throttle counters across process restarts (instance-only state, by design).

## Acceptance constraints

- A soft-block spanning exactly 1 hour from latch emits **≤ 61** WARNING lines from this path — at most 1 loud first WARNING plus ≤ 60 heartbeat re-emissions (vs current ~10,000+). Equivalently: ≤ 60 additional heartbeat WARNINGs per hour after the first.
- First WARNING after entering blocked state is unchanged (still loud, no suffix). This holds across **all three** clear paths (external reset, clean-fill auto-clear in `_check_same_orders`, REST verdict auto-clear) — re-entry after any of them restarts at the loud-first state.
- Recovery from soft-block emits exactly **one** INFO line when at least one WARNING was emitted during the latched period. For the WS-glitch REST path, that single INFO carries both verdict context and the suppressed count.
- When the block latched and cleared without any WARNING ever firing (no `on_ticker` placement-gate entry while latched), no recovery INFO is emitted — the recovery INFO closes out a warning flood, not a state transition. Reset side-effects (flag, buffers where applicable, throttle counters) still run.
- The recovery INFO is not emitted on False→False non-transitions (e.g. clean fills while the block was never latched).
- Reset side-effects (`_same_order_error`, execution buffers, throttle state) are owned by exactly one method (`reset_same_order_error`) so future state additions only need to touch one site.
- Existing `test_runner.py` tests still pass; new regression tests cover at-most-one WARNING per N-second window when the block stays latched, and the recovery INFO + throttle reset for all three clear contexts.
