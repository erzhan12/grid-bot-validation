# Feature 0062 ‚Äî Exclude ended gridbot runs from the replay grid-state seed lookup

## 1. Title & problem statement

`GridStateSnapshotRepository.get_at_or_before` (`shared/db/src/grid_db/repositories.py:1289-1326`) is the cross-run replay seed loader introduced in feature 0052. It selects the latest `grid_state_snapshots` row matching `(account_id, strat_id, symbol, exchange_ts <= at_ts)` ordered by `exchange_ts DESC, id DESC`, with **no check that the writing run was active at `at_ts`**.

After a **graceful** gridbot restart, the replay window's `seed.at_ts` (a recorder-clock timestamp) frequently lands **after** the previous live run's `end_ts` but **before** the new live run wrote its bootstrap snapshot. In that interval the loader returns the OLD run's stale snapshot, so replay validates against the WRONG grid. Because a (stale) row exists, feature 0054's fail-loud policy (`seed.enabled=True`) does **not** raise ‚Äî 0054 only raises `SeedDataQualityError` when *no* row is found. The bug is therefore silent: replay produces confidently wrong validation deltas. (Memory observation 4163: a sibling silent-fallback bug from 0052 masked a defect for 3 days; same class of failure.)

**Unclean shutdown (out of v1 scope unless mitigated ‚Äî see ¬ß4.6):** `end_ts` is stamped only on the normal stop path (`orchestrator.py:1399-1400`, `_update_run_records_stopped`). After crash/kill/SIGTERM without that path, the old live/shadow run can remain `status="running"`, `end_ts=NULL`. The planned `or_(Run.end_ts.is_(None), Run.end_ts >= at_ts)` predicate still treats that run as active at `at_ts`, so its stale snapshot can remain eligible after a new process starts and creates a fresh run ‚Äî the same wrong-grid window, but **without** a stamped `end_ts`. Startup does not close orphaned runs before `_create_run_records` inserts the new run (`orchestrator.py:1265-1275`).

A remote (UNMERGED) branch `cursor/critical-bug-investigation-633a` (head `f9d53a4`, open PR #138) attempted this fix but landed with a **failing test** (`apps/replay/tests/test_engine_seed.py::test_load_seed_prefers_db_over_file`). **This plan is a standalone implementation from current `main` ‚Äî it does NOT assume PR #138 is merged.** Verified on current `main`: `repositories.py:5` is `from sqlalchemy import func, tuple_` (no `or_`), and `get_at_or_before` has NO `.join(Run...)` and no run predicates. All of Steps 1-3 are therefore **mandatory new production edits**, not a "reconcile with #138". PR #138 is referenced only as a prior reference implementation; nothing in this plan depends on it existing in the tree.

## 2. Goal / non-goals

**Goal**
- `get_at_or_before` returns only snapshots written by a run that was **active at `at_ts`** (`start_ts <= at_ts AND (end_ts IS NULL OR end_ts >= at_ts)`) AND whose `run_type` is a grid-writing type (`live` / `shadow`).
- **Primary fix:** stale snapshots from runs whose **`end_ts` was stamped before `at_ts`** (graceful stop) are excluded ‚Üí loader returns `None` ‚Üí engine applies the 0054 `seed.enabled` policy (raise when enabled, file/blank fallback otherwise).
- **Unclean restart:** not fully solved by the predicate alone (¬ß4.6); v1 documents the gap and recommends a follow-up unless ¬ß8 #4 mitigation is adopted in this PR.
- Production seed loads continue to succeed (the active live/shadow run is selected).
- The FULL `apps/replay` and `shared/db` test suites pass after the prescribed test updates.

**Non-goals**
- No change to `get_latest` (`repositories.py:1253-1287`, the live gridbot path) ‚Äî live trading is unaffected, no bot restart needed.
- No schema migration in v1 by default (¬ß4.7). Do **not** claim `ix_grid_state_snapshots_lookup` "covers" the new query shape ‚Äî it leads with `run_id` and omits `symbol` (`models.py:454-456`), while `get_at_or_before` filters `(account_id, strat_id, symbol, exchange_ts <= at_ts)` with **no** `run_id` predicate. Acceptable for v1 (loader runs once per replay window, not hot-path) only after honest perf notes in ¬ß4.7 / ¬ß7.
- No change to the engine fail-loud policy (0054) or the file-fallback ordering in `engine._load_seed`.

## 4. Design

### 4.1 New predicate
INNER JOIN `runs` on `Run.run_id == GridStateSnapshot.run_id`; require the writer run active at `at_ts`:
```python
Run.start_ts <= at_ts,
or_(Run.end_ts.is_(None), Run.end_ts >= at_ts),
```
Boundaries **inclusive** on both ends. Inclusive `end_ts` is correct: a run that ended exactly at `at_ts` still legitimately owns the grid state at that instant; excluding it could spuriously return `None`.

### 4.2 run_type predicate ‚Äî DESIGN DECISION
**KEEP `Run.run_type.in_(("live","shadow"))`** and **fix the tests that seed under a recording run**. Rationale:
1. **Production invariant.** Grid snapshots are written ONLY by live/shadow runs (`orchestrator.py:1266`; RULES.md:349 "gridbot-owned"). The recorder never writes them. The predicate encodes that invariant and is defense-in-depth against a future writer mis-stamping a row.
2. **Necessary-but-not-sufficient.** The active-run window alone fixes the reported bug, but pairing it with the role predicate is strictly more precise and matches the data model. The fixtures that reuse the *recording* `"seed-run"` for grid snapshots are a test convenience, not production-representative, and should be corrected.
3. **Mirrors existing docs** (`repositories.py:1300-1302`, `snapshot_loader.py:297-303`): writer is a `live` run; recorder is the independent `recording` run that only contributes `at_ts`. Step 3 updates both docstrings after the repository change.

Cost = updating a small, enumerated test set (Step 5a, Step 6a/6b/6d). That is the right fix, not a reason to weaken the predicate.

### 4.3 Resulting query
```python
from sqlalchemy import func, or_, tuple_   # add or_ (line 5)

def get_at_or_before(self, account_id, strat_id, symbol, at_ts):
    return (
        self.session.query(GridStateSnapshot)
        .join(Run, Run.run_id == GridStateSnapshot.run_id)
        .filter(
            GridStateSnapshot.account_id == account_id,
            GridStateSnapshot.strat_id == strat_id,
            GridStateSnapshot.symbol == symbol,
            GridStateSnapshot.exchange_ts <= at_ts,
            Run.run_type.in_(("live", "shadow")),
            Run.start_ts <= at_ts,
            or_(Run.end_ts.is_(None), Run.end_ts >= at_ts),
        )
        .order_by(GridStateSnapshot.exchange_ts.desc(), GridStateSnapshot.id.desc())
        .first()
    )
```
This is a NEW production edit on current `main` (where `get_at_or_before` has no JOIN and `or_` is unimported). It coincides with what PR #138 intended, but the plan does not rely on #138 being merged.

### 4.4 Error handling
No new exceptions. Returning `None` is the contract; `snapshot_loader.py:344` handles it and `engine._load_seed` applies the 0054 policy. The fix simply makes `None` occur in the stale-run case ‚Üí the desired raise under `seed.enabled=True`.

### 4.5 Tie-breaks & overlap
- **INNER JOIN safety:** The INNER JOIN correctly EXCLUDES any orphan snapshot (no matching `runs` row) ‚Äî that is the desired behaviour (no run = cannot verify the writer was active). In **production Postgres**, `GridStateSnapshot.run_id` is `nullable=False` + FK `ondelete="CASCADE"` (`models.py:437-440`), so orphans cannot exist in the first place. In **tests (SQLite in-memory)** FK enforcement is OFF by default (no `PRAGMA foreign_keys=ON`), so an orphan COULD be inserted ‚Äî but the INNER JOIN still safely drops it. Either way the result is correct; we do not rely on FK enforcement in tests.
- **Overlapping live+shadow on same key:** if both are active at `at_ts` and both wrote, multiple rows match; the unchanged `ORDER BY exchange_ts DESC, id DESC` deterministically picks the most recent. No `run_type` tie-break added ‚Äî in production one orchestrator process runs one `run_type` per strat at a time (`orchestrator.py:1266`).

### 4.6 Known limitation ‚Äî unclean shutdown (`end_ts IS NULL`)

The `end_ts` window predicate fixes the **reported** production path: graceful stop stamps `end_ts`, restart, replay `seed.at_ts` lands in the gap before bootstrap.

It does **not** reliably fix **unclean** restarts:

| State after crash/kill | Old run at `at_ts` | New run after restart |
|------------------------|--------------------|------------------------|
| `end_ts` NULL, `status="running"` | Passes `end_ts IS NULL` branch | Also passes (new run active) |
| Stale snapshot `exchange_ts <= at_ts` | Still matches JOIN | Bootstrap row may not exist yet |

`ORDER BY exchange_ts DESC` can still return the **old** run's last snapshot ‚Äî same silent wrong-grid failure as ¬ß1.

**v1 default (recommended):** Ship the `end_ts` + `run_type` guard only; document this gap in RULES.md (Step 4) and ¬ß7; track follow-up **gridbot startup hygiene** ‚Äî before creating a new live/shadow run for `(account_id, strategy_id)`, mark any prior open runs for that strat `status="completed"`, `end_ts=now()` (or a dedicated `status="abandoned"`). That closes the DB window without changing replay tie-break semantics.

**Optional in-PR mitigation (¬ß8 #4 ‚Äî discouraged):** Pick the winning writer run **from `runs` first**, then match snapshots for that run only. **Wrong (does not fix crash):** `max(start_ts)` computed over the snapshot JOIN result ‚Äî the new run has **no** snapshot rows yet, so only the old run appears and its stale row still wins. **Correct (two-phase):**
1. Subquery on `runs` only: among `live`/`shadow` rows for the same `(account_id, strategy_id)` identity active at `at_ts`, take the single `run_id` with greatest `start_ts` (tie-break `run_id` if needed).
2. `INNER JOIN grid_state_snapshots` on that `run_id` + `(account_id, strat_id, symbol, exchange_ts <= at_ts)`, then `ORDER BY exchange_ts DESC, id DESC`.

Pre-bootstrap the new run exists in step 1 but step 2 returns no rows ‚Üí `None` ‚Üí 0054 raises. **Trade-off:** ¬ß6.1 #6 (`test_get_at_or_before_two_active_runs_latest_wins`) no longer models production ‚Äî rewrite for newest-run-wins. Prefer **startup orphan cleanup** (v1 default) over this query change: simpler, fixes the root cause, no replay tie-break churn.

### 4.7 Performance / index honesty

- **JOIN:** `runs.run_id` PK lookup per candidate snapshot row is cheap; the snapshot side dominates.
- **`ix_grid_state_snapshots_lookup`:** `(run_id, account_id, strat_id, exchange_ts, id)` ‚Äî does **not** match the replay filter `(account_id, strat_id, symbol, exchange_ts)` with no `run_id`. Postgres may scan/filter on `account_id`/`strat_id` then join, or use a less selective plan as the table grows. **Do not document this index as "covering" the lookup.**
- **v1 acceptance:** Run `EXPLAIN (ANALYZE, BUFFERS)` on production-like Postgres for a representative `get_at_or_before` call after Step 2; record plan type (index scan vs seq scan) and row estimates in the PR notes. If seq scan on `grid_state_snapshots` at expected row counts, defer new index to follow-up.
- **Follow-up index (if EXPLAIN warrants):** migration adding e.g. `(account_id, strat_id, symbol, exchange_ts DESC, id DESC)` on `grid_state_snapshots` ‚Äî separate from this feature unless latency is unacceptable.

## 7. Risks & mitigations
- **INNER JOIN drops snapshots with no `runs` row** ‚Äî this is intended (excluding orphans is correct: no run = window unverifiable). In production Postgres orphans cannot exist (`run_id` `nullable=False` + FK CASCADE, `models.py:437-440`). Tests run on SQLite with FK enforcement OFF by default, so an orphan could be inserted ‚Äî the INNER JOIN still excludes it safely. We do not depend on SQLite FK enforcement.
- **Timezone-aware vs naive** ‚Äî `exchange_ts`/`start_ts`/`end_ts` are all `DateTime(timezone=True)`; `at_ts` is tz-aware (test fixtures + the pre-existing `exchange_ts <= at_ts` already depend on it). We add no new tz semantics. New repo tests use tz-aware UTC to lock the contract. The `engine._seed_pre_check` `_strip_tz` path is a SEPARATE concern and is not touched.
- **Performance** ‚Äî see ¬ß4.7. `runs.run_id` PK supports the JOIN; `ix_grid_state_snapshots_lookup` is **not** aligned with the cross-run filter (run_id-first, no symbol). One-shot loader: acceptable if EXPLAIN on realistic data is acceptable; otherwise add `(account_id, strat_id, symbol, exchange_ts, id)` index in a follow-up.
- **Unclean shutdown** ‚Äî see ¬ß4.6. Predicate alone does not exclude stale open live/shadow runs after crash; v1 documents + recommends gridbot orphan-run cleanup on startup. ¬ß8 #4 two-phase max-`start_ts` is optional and discouraged (easy to implement incorrectly).
- **Duplicate run_id in `test_engine_seed.py`** ‚Äî the literal `"gridbot-live-run"` is ALREADY used by `test_load_seed_prefers_db_snapshot_from_other_run_id` on `main`. The plan therefore does NOT add it to the shared fixture; each new live run (Steps 6a/6b/6d/6e) uses a distinct `run_id`. Run the full file to confirm no `UNIQUE constraint failed: runs.run_id`.
- **Standalone from `main` (NOT on top of PR #138)** ‚Äî `main` does not contain #138's production change, RULES.md edit, the reproducer unit test, or the engine reproducer. ALL of Steps 1-8 are new edits. PR #138 is a reference only; do not assume any of it is present in the tree.
