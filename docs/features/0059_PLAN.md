# Feature 0059 — Per-snapshot deltas for upnl / cur_realised / cum_realised / position_value (parity with 0058 log line)

## Context

Feature 0058 emits a per-position-update DEBUG line in
`packages/gridcore/src/gridcore/position.py:277-292` carrying four
USDT-denominated values: `upnl_usdt`, `realized_usdt` (cur cycle),
`cum_realized_usdt`, `pos_value_usdt`. The same code path runs live
(gridbot) and backtest (replay), so an operator sees these four numbers
in both contexts.

The replay validation comparator currently does **not** expose all four
at the granularity the log line emits:

- `unrealised_pnl_mean_abs_delta` / `unrealised_pnl_max_abs_delta` — per-snapshot, EXISTS.
- `cur_realised_pnl_final_delta` — final value only (0056). No per-snapshot delta.
- `cum_realised_pnl_final_delta` — final value only (0034). No per-snapshot delta.
- `position_value` — NOT captured in the DB schema at all; no delta.

Goal: surface per-snapshot timeseries deltas for all four fields plus a
`position_value` final delta, so replay validation reports parity at the
same granularity the 0058 log line produces.

Pattern to follow: feature 0056 added `cur_realised_pnl`
(`scripts/migrate_0056_cur_realised_pnl.py`, writer wiring, loader probe,
comparator delta, reporter rows). `position_value` mirrors that pipeline.

## Constraints / decisions baked into this plan

- **Additive only.** Keep existing metric fields
  (`cur_realised_pnl_final_delta`, `cum_realised_pnl_final_delta`,
  `unrealised_pnl_*_delta`) unchanged — they are referenced by existing
  tests (`apps/comparator/tests/test_position_metrics.py`,
  `test_reporter.py`) and downstream CSV consumers. The new per-snapshot
  aggregates are added alongside, NOT as renames. The issue's "rename
  optional" for `upnl_usdt_*` is declined to avoid churn; the existing
  `unrealised_pnl_*` names already cover that field per-snapshot.
- **NULL-safe.** Pre-migration `position_value` rows are NULL; the loader
  probe and per-field delta logic must skip (None) rather than error,
  exactly like the 0056 `cur_realised_pnl` path. NULL `position_value`
  delta must NOT set `has_missing_telemetry` (same exclusion 0056 made
  for `cur_realised_pnl`, see `position_metrics.py:138-146`).
- **Per-symbol breakout** is already structurally satisfied: replay runs
  one symbol per invocation (`engine.py` loads snapshots filtered by
  `config.symbol`, then writes its own `validation_metrics.csv`). No
  multi-symbol aggregation code exists in the comparator. This plan does
  NOT add cross-symbol grouping; LTC and SOL each get their own report
  from their own replay run. (Confirm with operator if a combined report
  is wanted — out of current scope.)

## Phase 4 — Comparator (deltas + aggregates)

### 4.1 Per-pair deltas
`apps/comparator/src/comparator/position_metrics.py`.

`PositionComparisonPair` dataclass (line 56-92): add **two** new
`Optional[Decimal]` delta fields (default None). The 0058 log line exposes
four USDT deltas; cur/cum are already on the pair as `cur_realised_pnl_delta`
and `cum_realised_pnl_delta` — do not duplicate them. The bullets below
map all four log-line fields to pair sources; only the first and last bullets
are new dataclass members:
- `upnl_usdt_delta` — per-snapshot raw `unrealised_pnl` delta
  (`bt.unrealised_pnl - live.unrealised_pnl`). Distinct from the existing
  `unrealised_pnl_delta`, which is RECOMPUTED against `live.mark_price`.
  This new field compares the stored USDT values 1:1, matching the 0058
  log line semantics (`position.unrealized_pnl` as-emitted). Keep both.
- `cur_realised_usdt_delta` — alias semantics of existing
  `cur_realised_pnl_delta`; reuse `cur_realised_pnl_delta` rather than
  duplicating (see 4.2). [No new field — documented here so the reader
  knows the aggregate sources from `cur_realised_pnl_delta`.]
- `cum_realised_usdt_delta` — reuse existing `cum_realised_pnl_delta`.
- `pos_value_delta` — `_safe_sub(bt.position_value, live.position_value)`.

Net new per-pair fields: `upnl_usdt_delta`, `pos_value_delta`.

`_build_pair` (line 107-147): compute the two new deltas via `_safe_sub`.
Do NOT add `pos_value_delta is None` / `upnl_usdt_delta is None` to the
`has_missing_telemetry` block (line 138-146) — pre-migration NULLs would
universally trip the flag (same rationale as the 0056 `cur_realised_pnl`
exclusion noted at line 76-80).

`_unmatched_bt_marker` (line 336-360): set
`pair.upnl_usdt_delta = None` and `pair.pos_value_delta = None`.

### 4.2 Aggregate metrics
`apps/comparator/src/comparator/metrics.py`, `ValidationMetrics`
(dataclass, telemetry block at line 90-116). Add nine new fields
(`Decimal`, default `Decimal("0")`):
- `upnl_usdt_mean_abs_delta`, `upnl_usdt_max_abs_delta`
- `cur_realised_usdt_mean_abs_delta`, `cur_realised_usdt_max_abs_delta`
- `cum_realised_usdt_mean_abs_delta`, `cum_realised_usdt_max_abs_delta`
- `pos_value_usdt_mean_abs_delta`, `pos_value_usdt_max_abs_delta`
- `pos_value_final_delta` (final value, mirroring the cur/cum final deltas)

(9 new fields total; existing `*_final_delta` and recomputed-unrealised
aggregates retained unchanged.)

`apps/comparator/src/comparator/position_metrics.py`, `fold_metrics_into`
(line 252-333). The existing `_agg(field_name)` helper (line 275-291)
already does mean-abs / max-abs over matched non-None per-pair values.
Reuse it:
- `upnl_usdt_mean/max = _agg("upnl_usdt_delta")`
- `cur_realised_usdt_mean/max = _agg("cur_realised_pnl_delta")`
- `cum_realised_usdt_mean/max = _agg("cum_realised_pnl_delta")`
- `pos_value_usdt_mean/max = _agg("pos_value_delta")`

For `pos_value_final_delta`: add a per-side last-pair aggregation block
mirroring the `cum_realised_pnl_final_delta` block (line 312-319) — iterate
matched pairs, keep last non-None `pos_value_delta` per side, sum across
sides. This keeps hedge-mode long/short both represented. **Reporting
tradeoff (intentional):** unlike realised PnL, equal-and-opposite long/short
position-value drift can cancel in this scalar sum; per-snapshot
`pos_value_usdt_mean/max_abs_delta` still surfaces drift across all matched
pairs, so operators should rely on those for hedge-mode diagnosis, not
`pos_value_final_delta` alone.

Also update the `fold_metrics_into` docstring (line 257), which currently
reads "Mutate ``metrics`` with the 12 new aggregate fields (idempotent)."
— bump the count to **21** (12 existing + 9 new). Likewise update the
`ValidationMetrics` class-level telemetry comment (`metrics.py`, line
90-116) so the documented field count reflects the nine additions.

### 4.3 Aggregation note
The new per-snapshot cur/cum aggregates sum |delta| across ALL matched
pairs (via `_agg`), whereas the existing final-delta fields capture only
the last per-side value. Both are retained because the issue explicitly
wants per-snapshot accuracy (intermediate drift that cancels out is
invisible to a final-only comparison). Document this in the metrics
docstring so the two families are not confused.
