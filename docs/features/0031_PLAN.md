# Feature 0031 — SAME ORDER detector dedup + soft-block auto-recovery on WS_GLITCH_SUSPECTED

## Brief

Two distinct defects in the live SAME ORDER detector were exposed by the mainnet `mainnet_live` LTCUSDT incident on 2026-05-09 → 2026-05-10:

1. **No dedup.** The same phantom (same `(current_order_id, previous_order_id)` pair, same price, same side) re-triggered SAME ORDER ERROR + ALERT on the runner 3 h 24 min after the first trigger, although nothing on the exchange had changed (no new fills for that pair). Each retrigger emits 5 ERROR-line bursts and a Telegram ALERT, polluting the on-call channel.
2. **Soft-block does not auto-clear when REST cross-check rules out a real duplicate.** The diagnostic REST cross-check ran, returned `verdict=WS_GLITCH_SUSPECTED` (REST sees only one of the two order_ids in recent executions → the second "fill" never existed), yet `_same_order_error=True` stayed latched and order placement remained `BLOCKED` for ≈8 hours until the bot's WS reconnected and `reset_same_order_error()` fired indirectly.

This feature makes the detector idempotent over the dedup window and makes `WS_GLITCH_SUSPECTED` a self-healing signal that releases the soft-block automatically.

Non-goals:

- Reworking the SAME ORDER detector itself (`_check_same_orders_side` time-window guard, buffer classification, hedge-mode reduce_only logic).
- Identifying the root cause of WS execution-stream replays. The recorder feature 0029 already captures private-WS payloads and is the right tool to chase that separately.
- Migrating `_diagnosed_same_order_pairs` semantics to a different cross-check policy.

## Phase 1 (Critical) — Dedup at SAME ORDER detection

### Goal

When the detector observes a SAME ORDER pair that was already seen and recorded within `_SAME_ORDER_DEDUP_TTL_SEC`, the dedup gate must take responsibility for the entire response — replacing the noisy ERROR + ALERT path with a single DEBUG line — while preserving the soft-block state implied by the cached verdict.

Universal contract on an adjudicated cache hit (`WS_GLITCH_SUSPECTED` or `REAL_DUPLICATE`):

- Do NOT emit the `logger.error("... SAME ORDER ERROR ...")` line.
- Do NOT call `notifier.alert(...)`.
- Do NOT run the REST cross-check (the first occurrence already produced or attempted a verdict).
- DO emit a single `logger.debug` line ("duplicate SAME ORDER pair suppressed within TTL", with pair, price, side, age-since-first-seen, and the cached verdict), so post-incident greps still surface the suppressed event.

Verdict-specific exception for `_same_order_error` (the only place verdicts diverge):

- **WS_GLITCH_SUSPECTED**: do not set `_same_order_error = True`. The Phase 2 auto-clear has already released the block; the dedup gate must not silently re-arm it.
- **REAL_DUPLICATE**: **must set `_same_order_error = True`**. `_check_same_orders` unconditionally resets the flag to False at the top of every call (`runner.py:1016`); without an explicit re-set here, the dedup gate would silently lift a block that the first occurrence correctly latched.
- **UNKNOWN** (cross-check earlier failed, no verdict yet): treat as cache miss and fall through to the full first-trigger path (fresh ERROR + ALERT + REST attempt). The "no notifier, no REST" universal contract above does NOT apply to UNKNOWN — UNKNOWN is the explicit "we never adjudicated this, retry" state.

### State changes

In `StrategyRunner.__init__` (around lines 191–195):

- Replace `self._diagnosed_same_order_pairs: set[frozenset[str]]` with a verdict-aware dict whose key is the order-id pair (`frozenset` of two strings) and whose value is a record holding `first_seen_ts` (UTC datetime), `last_seen_ts` (UTC datetime), and `verdict` (one of `"WS_GLITCH_SUSPECTED"`, `"REAL_DUPLICATE"`, or `"UNKNOWN"`). The new cache supersedes both prior responsibilities of `_diagnosed_same_order_pairs` (REST-recheck rate-limit) and the missing dedup behavior, and adds verdict-awareness so retriggers can preserve a real-duplicate block (see Phase 1 algorithm point 3).
- Field name: `_same_order_dedup_cache`. The name is deliberately neutral: the cache stores **all** SAME ORDER pair adjudications (`WS_GLITCH_SUSPECTED`, `REAL_DUPLICATE`, and `UNKNOWN`), not only phantom ones. A "phantom"-flavored name would invite a future reader to skip non-phantom entries as an "optimization" — the exact bug this feature exists to prevent. Type of the value may be a small private dataclass or a tuple — implementation detail, not constrained by the plan.

Add a module-level constant near `_SAME_ORDER_TIME_WINDOW_SEC` (line 61):

- `_SAME_ORDER_DEDUP_TTL_SEC` set to `21600.0` (6 hours). The known incident retrigger gap is 3 h 24 min; 6 h covers it with ≈75% headroom, and is short enough that a stale cached pair cannot persist across an operator shift. Lower bound is the observed gap; we deliberately **do not** accept a TTL that fails the very incident this feature is built to prevent. Upper bound is informally capped by "should not survive a normal operator shift" so a truly novel duplicate after that window is loud, not silent.

### Dedup-key choice (decision recorded in plan)

Two candidates considered:

- **(a) `exec_id` of the phantom execution.** `ExecutionEvent.exec_id` is populated from Bybit `execId` and is on every WS payload. *Rejected as primary key:* the WS-replay scenario producing the retrigger emits the same `execId` (or a fresh one — both behaviors have been observed), so `exec_id` is neither necessary nor sufficient. Furthermore the SAME ORDER detector compares **two** executions (current + previous in the buffer); a single execId is the wrong granularity.
- **(b) `frozenset((current.order_id, previous.order_id))`.** Chosen. Order-independent, deterministic, exchange order_ids are unique per the broker so collision risk is zero. Matches the granularity at which SAME ORDER triggers — once we know "this pair is a known phantom" we want to suppress *every* re-emission of that pair for the TTL window. Mirrors the pre-existing `_diagnosed_same_order_pairs` key, so we are extending semantics, not introducing a new key shape.

Price + side are deliberately omitted from the key: the same `frozenset` of order_ids deterministically implies the same price + side (both tracked orders / fills referencing the same exchange order_id always carry the same price and side).

### Algorithm changes inside `_check_same_orders_side` (current lines 1054–1135)

The current control flow (after the partial-fill / time-window guards) is:

1. Resolve current/previous `TrackedOrder` and apply the time-window guard.
2. Compute the reduce-only fields for the diagnostic dump.
3. Emit the SAME ORDER ERROR log line.
4. Set `_same_order_error = True`.
5. Call `notifier.alert(...)`.
6. Compute the order-id pair key.
7. If the pair is not in `_diagnosed_same_order_pairs`, register it and run `_diagnostic_rest_check_executions(...)`.
8. Return.

After this feature, the control flow becomes:

1. Same as today (time-window guard unchanged).
2. Compute the pair key early, **before** the ERROR log path.
3. **Dedup gate (new).** Look up the pair in `_same_order_dedup_cache` after first dropping any entry whose `last_seen_ts` is older than the TTL (lazy expiry, bounds memory). If the pair is in the cache:
   - **WS_GLITCH_SUSPECTED hit** — emit a single DEBUG line ("duplicate SAME ORDER pair suppressed within TTL", with pair, price, side, age in seconds, and verdict). Do not touch `_same_order_error` (the Phase-2 auto-clear has already released it; leaving it at the value `_check_same_orders` reset to is correct). Do not call the notifier. Do not call REST cross-check. Update `last_seen_ts` to now. Return.
   - **REAL_DUPLICATE hit** — emit the DEBUG line (same content, but with verdict=REAL_DUPLICATE so on-call greps still see it). Set `_same_order_error = True` to **re-establish** the soft-block that the upstream `_check_same_orders` reset to False at the start of the same call. Do not call the notifier (the original ALERT is still authoritative; re-emitting an ERROR-level "ALERT" line is the noise this feature exists to remove). Do not call REST cross-check. Update `last_seen_ts`. Return.
   - **UNKNOWN hit** — REST cross-check failed on the first occurrence and never produced a verdict. Conservative path: fall through to the full first-trigger handling below so the operator gets a fresh, loud ALERT and a fresh cross-check attempt; this is treated as if the cache had no entry (do not return early). Optional refinement out of scope: rate-limit "UNKNOWN" reattempts to once per minute via the same `last_seen_ts`.
4. **First-trigger path (cache miss or UNKNOWN fall-through).** Compute the reduce-only fields. Emit the existing SAME ORDER ERROR log line. Set `_same_order_error = True`. Call `notifier.alert(...)` exactly as today.
5. **Record the pair** in `_same_order_dedup_cache` with `first_seen_ts = last_seen_ts = now` and `verdict = "UNKNOWN"`. The cache entry exists *before* the cross-check runs so a same-event burst (5 ERROR repeats inside 1 second observed in the incident) is suppressed by the dedup gate on the second through fifth events.
6. **REST cross-check.** Run `_diagnostic_rest_check_executions(current, previous)`. Phase 2 inside that function will (a) write the resolved verdict back into the cache entry and (b) on `WS_GLITCH_SUSPECTED`, release the soft-block.
7. Return.

The single-cache contract replaces the prior split between `_diagnosed_same_order_pairs` (REST rate-limit) and the absent dedup behavior. Both responsibilities now share one keyed lookup with explicit verdict semantics.

### Re-arm after TTL expires

When the buffer still holds the phantom pair and the same WS replay arrives after the TTL window (≥ 6 h), the cache entry has been dropped and the detector re-fires as if first-seen. This is the intended fail-loud behavior: a phantom that recurs after 6 hours of silence is operationally indistinguishable from a fresh issue and deserves a fresh alert + cross-check. There is no acceptance of "one extra alert per hour" — TTL is sized so that a single observation of the documented incident produces zero retriggers, not one.

### Notifier behavior

No changes to `Notifier.alert`. The dedup gate is upstream of the notifier, so suppressed retriggers never reach it. The 60-second `_DEFAULT_THROTTLE_SECONDS` Telegram throttle stays in place as defense-in-depth for any genuine ERROR-level alert burst.

## Phase 2 (Major) — Auto-clear soft-block when REST verdict says WS glitch

### Goal

When `_diagnostic_rest_check_executions` concludes `verdict == "WS_GLITCH_SUSPECTED"`, the soft-block is by definition a false positive — REST's authoritative execution list is missing one of the two "duplicate" fills, so there is no real duplicate. The runner must:

- Write the resolved verdict back into the matching `_same_order_dedup_cache` entry (the entry was inserted by Phase 1 step 5 with `verdict = "UNKNOWN"`).
- Emit a single INFO log line stating that the SAME ORDER soft-block was auto-cleared because REST cross-check confirmed a WS glitch, including the strat_id and verdict.
- Call `self.reset_same_order_error()` to flip `_same_order_error → False` and clear both per-side execution buffers, so the lingering bad pair cannot immediately re-trigger via stale buffer state.
- **Do NOT call the notifier.** `Notifier.alert(...)` always emits at ERROR level (`notifier.py:67`); a successful recovery emitted at ERROR level is a contradiction. The INFO log is sufficient — operators reading the gridbot log will see the unblock, and the absence of a fresh `ALERT: SAME ORDER ERROR` line going forward (suppressed by the dedup gate) is itself the observable signal that the bot recovered.

For `verdict == "REAL_DUPLICATE"`, write the verdict back into the cache entry and otherwise leave behavior identical to today (block latched, ERROR log already emitted on the first-trigger path). The Phase 1 dedup gate uses this cache value to re-establish `_same_order_error = True` on subsequent occurrences of the same pair within the TTL.

For the REST exception path (line 1177–1181, no verdict produced), leave the cache entry as `verdict = "UNKNOWN"`; the block stays. Phase 2 must not auto-clear when the cross-check itself failed — that would be a worse failure mode than the bug this feature fixes.

### Changes inside `_diagnostic_rest_check_executions` (current lines 1137–1181)

After the verdict is computed and the existing verdict log line is emitted, add — on the success path only — two responsibilities, in order:

1. **Cache write-back.** Update the `_same_order_dedup_cache` entry for the current `(cur_id, prev_id)` pair (which Phase 1 step 5 has already inserted with `verdict = "UNKNOWN"`): set `verdict` to the computed value (`"WS_GLITCH_SUSPECTED"` or `"REAL_DUPLICATE"`). `last_seen_ts` does not change here — it was set by Phase 1 just before this call. If, defensively, the entry is missing (race or mis-ordered call), insert one with `first_seen_ts = last_seen_ts = now` and the resolved verdict.
2. **Auto-clear branch.** If the verdict equals `"WS_GLITCH_SUSPECTED"`: emit the INFO log line described above and call `self.reset_same_order_error()`. Do not invoke the notifier. The `"REAL_DUPLICATE"` branch performs no action beyond step 1.

The exception branch is unchanged — it logs at ERROR ("...cross-check failed (diagnostic only, ignoring): ...") and returns. The cache entry inserted by Phase 1 keeps `verdict = "UNKNOWN"` and the soft-block remains latched.

### Effect on Phase 1 cache interaction

`reset_same_order_error()` does **not** touch `_same_order_dedup_cache`. Intentional: after the soft-block is released, the same pair must remain dedup-suppressed for the rest of the TTL so a follow-up WS replay does not re-arm the block on a phantom that has already been adjudicated. The cache is the persistent "we already adjudicated this pair, here is the verdict" memory; `reset_same_order_error()` is the operational "release the block now" action; they are orthogonal and must remain so.

### Re-arm semantics

- **Same pair, WS_GLITCH_SUSPECTED, within TTL.** Phase 1 dedup gate suppresses the retrigger; flag stays at whatever `_check_same_orders` last reset it to (False). Block stays released.
- **Same pair, REAL_DUPLICATE, within TTL.** Phase 1 dedup gate re-arms `_same_order_error = True`. Block stays latched. No new ALERT emitted.
- **Same pair, UNKNOWN (cross-check earlier failed), within TTL.** Phase 1 falls through to first-trigger path. Fresh ERROR + ALERT + cross-check.
- **New pair, any verdict.** Cache miss; first-trigger path runs end-to-end. If REST returns `WS_GLITCH_SUSPECTED`, Phase 2 auto-clears; if `REAL_DUPLICATE`, block stays.
- **Same pair after TTL expires.** Cache entry dropped on lazy-expiry sweep. First-trigger path runs end-to-end.

## RULES.md update (Step 6)

After implementation, update `RULES.md` § "SAME ORDER detection" (current line 256) to record:

- `_same_order_dedup_cache` is the single dedup mechanism, keyed by `frozenset` of the two order_ids in the SAME ORDER pair, valued by a record carrying `first_seen_ts`, `last_seen_ts`, and `verdict ∈ {WS_GLITCH_SUSPECTED, REAL_DUPLICATE, UNKNOWN}`. TTL `_SAME_ORDER_DEDUP_TTL_SEC = 21600` (6 h). Don't reintroduce a separate `_diagnosed_same_order_pairs` rate-limit set — that responsibility is unified with dedup on this key.
- The dedup gate is verdict-aware: WS_GLITCH retriggers are silently suppressed; REAL_DUPLICATE retriggers are also silenced but **must re-set `_same_order_error = True`** to defeat the unconditional reset that `_check_same_orders` does at the top of every call; UNKNOWN retriggers fall through to the full first-trigger path.
- `verdict == "WS_GLITCH_SUSPECTED"` from `_diagnostic_rest_check_executions` auto-clears the soft-block via `reset_same_order_error()`. `REAL_DUPLICATE` and exception paths leave the block latched. `reset_same_order_error()` does not clear the dedup cache, by design.
- The auto-clear path emits an INFO log only — **no `notifier.alert`** — because `Notifier.alert` always emits at ERROR level (`notifier.py:67`) and a successful recovery should not produce an ERROR-level log line.
- `Notifier.alert(error_key=...)` only throttles Telegram delivery (`_DEFAULT_THROTTLE_SECONDS = 60` at `notifier.py:18`); `logger.error("ALERT: ...")` always fires. To suppress the log line itself, dedup must happen upstream of the notifier, not via `error_key`.
