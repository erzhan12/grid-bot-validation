# Feature 0042 - UTA account wallet fields for liquidation parity

## Goal

Feature 0034 exposed that backtest liquidation prices are using the wrong
balance input. The Bybit/bbu2 cross-margin liquidation formula is correct,
but replay currently seeds `BacktestSession.current_balance` from
`wallet_snapshots.wallet_balance`, which is the per-coin USDT `coin[].walletBalance`.

For UTA cross margin, Bybit's available balance input is the account-level
`totalAvailableBalance` field, in USD, from the wallet account object outside
the `coin[]` list. Feature 0042 records those account-level wallet metrics and
feeds `total_available_balance` into replay/backtest liquidation calculations.

Expected effect: short-side `liq_price_max_abs_delta` should move down from
about 972 USDT by at least one order of magnitude. Buy-side `bt_liq` remains
0043 scope.

## Confirmed Context

- `BacktestRunner._estimate_liquidation_price` already matches Bybit/bbu2.
- Current sinks are `runner.py` callers that pass `self._session.current_balance`
  into liquidation and risk state construction.
- `BacktestSession.current_balance` starts as `initial_balance` and is refreshed
  in `update_equity`; replay currently sets that initial value from the legacy
  per-coin wallet seed.
- `WalletWriter._messages_to_models` and recorder REST snapshots both loop over
  `coin[]` and store one row per coin, but they ignore account-level fields on
  the surrounding wallet object.
- Bybit official REST and WS docs place the needed fields above `coin[]`;
  `coin[].walletBalance` remains per coin.
- Existing production CSV confirms the direction: for shorts, increasing the
  available-balance input raises backtest liquidation price toward live; for
  longs, increasing available balance can push liq to zero.

## Non-Goals

- Do not alter `_estimate_liquidation_price` math.
- Do not implement hedge-credit / buy-side `liqPrice=0` behavior; defer to 0043.
- Do not change existing per-coin `wallet_balance` or `available_balance`.
- Do not modify comparator pairing, metrics, or other writers.
- Do not add retry/DLQ behavior to wallet writer.
- Do not adjust seed-time unrealized PnL unless residual deltas remain material.
- Do not commit, push, or open a PR without explicit operator instruction.

## Implementation Plan

1. Schema: add nullable `Numeric(20, 8)` columns to `WalletSnapshot`:
   `total_equity`, `total_available_balance`, `total_margin_balance`,
   `account_im_rate`, `account_mm_rate`. Store account rates as raw Bybit
   decimal ratios, not percentages. Keep wallet indexes unchanged.

2. Migration: add `scripts/migrate_0042_uta_wallet_fields.py`, mirroring the
   0034 migration with idempotent `ALTER TABLE wallet_snapshots ADD COLUMN`.
   It has no dependency on prior 0029/0034 columns, avoiding the 0034
   migration regression on DBs missing `run_id`. No backfill in v1;
   historical rows remain NULL.

3. Decimal helper: extract the wallet coercion helper to
   `shared/db/src/grid_db/_decimal.py` as `decimal_or_zero`. Import it from
   both recorder and event_saver so `None` and `""` handling stays identical.

4. WebSocket wallet writer: parse account fields once per `wallet_data`, before
   the `coin[]` loop, and stamp the same five values onto every emitted coin
   row. Set `raw_json={**coin_data, "_account": account_subset}` so existing
   coin keys remain top-level while account context is retained. The subset is:
   `accountType`, `marginMode`, `totalEquity`, `totalAvailableBalance`,
   `totalMarginBalance`, `accountIMRate`, `accountMMRate`.

5. REST initial snapshot: parse the same account fields from each account
   object before looping over `coin[]`, using `decimal_or_zero` semantics for
   `None` / `""`, and stamp them on each per-coin row.

6. Repository: extend `WalletSnapshotRepository.bulk_insert` with the five new
   columns. Existing reads can return ORM rows; no extra repository method is
   required.

7. Seed loader: add frozen `WalletSeed(coin_balance, total_available_balance,
   total_equity, total_margin_balance, account_im_rate, account_mm_rate)` plus
   `load_wallet_seed_full(...) -> Optional[WalletSeed]`. Return `None` when no
   row exists or `total_available_balance` is NULL, so legacy migrated DBs fall
   back to config `initial_balance`. Keep `load_wallet_snapshot` indefinitely
   as the legacy per-coin helper and export both helpers.

8. Replay engine: switch `_load_seed` to call `load_wallet_seed_full`, return
   `Optional[WalletSeed]`, and initialize `BacktestSession` from
   `wallet_seed.total_available_balance` when present. Log both the per-coin
   `WalletSeed.coin_balance` and `WalletSeed.total_available_balance`.

9. Backtest semantics: document that in replay with 0042 seed data,
   `initial_balance/current_balance` represent UTA `totalAvailableBalance`
   plus simulated PnL/fees/funding, not per-coin USDT wallet balance. This can
   double-count seed unrealized PnL if the seed has net UPL; in the 15h CSV the
   first paired long/short net UPL is about +0.77 USDT, which shifts short liq
   by roughly `0.77 / 2.2 = 0.35` USDT, far below the ~435 USDT balance-input
   error. The derivative comes from the short-side closed form; no formula
   change in 0042.

10. Tests: cover repository round-trip, WS stamping and empty-string coercion,
    REST stamping, `load_wallet_seed_full` success, loader NULL fallback, replay
    engine fallback to config `initial_balance` when 0042 columns are NULL,
    replay session initialization from `total_available_balance`, and runner
    liquidation scenarios with `wallet_balance` interpreted as UTA available.

11. Documentation: update `RULES.md` with wallet columns and the balance
    semantic shift; add a small 0034 Risks footnote saying 0042 addresses the
    account-level balance input and buy-side hedge credit remains 0043.
