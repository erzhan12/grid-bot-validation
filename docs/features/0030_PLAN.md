# 0030_PLAN — orderLinkId millisecond-suffix hotfix and prefix-normalization cascade

## Description

Bybit caches `orderLinkId` for ~1–2h after an order's terminal state
(cancel/fill). Our `PlaceLimitIntent.client_order_id` is a deterministic
16-char SHA256 hex digest of `(symbol, side, price, direction)`, so any
re-placement of the same logical intent within the cache window collides
with the prior order on the wire and is rejected as ErrCode 110072
"OrderLinkedID is duplicate". A live incident on 2026-05-08 produced
~12 000 duplicate-rejected placement attempts and 0 successful orders
across a 2 h window before the fix landed.

The fix has two halves:

1. **On the wire**: append `-{int(now_utc_ms)}` to every `orderLinkId`
   sent at placement time, so each placement attempt carries a unique
   value that doesn't collide with Bybit's cache. The deterministic SHA
   prefix is preserved as the substring before the first `-`.

2. **On every read site that uses `orderLinkId` as a matching key**:
   strip the suffix back to the prefix. The internal contract that
   `client_order_id` is the deterministic SHA prefix is unchanged —
   only the wire form is now suffixed; persistence stores the suffixed
   value verbatim for forensics, but lookup keys, join keys, and seed
   keys all live in prefix space.

A single shared helper, `gridcore.intents.extract_client_order_prefix`,
lives next to the SHA generator (`PlaceLimitIntent.create`) so wire-
format knowledge has exactly one home. Three application-layer call
sites (gridbot runner, comparator loader, replay seed loader) call it
on read.

This was landed retroactively as two stacked PRs: PR #69 (gridbot +
gridcore + replay + docs) and PR #70 (comparator). Both are now in
main as of `1db2934` (#70 merge) on top of `8cbc860` (#69 merge).

## Phase 1 — Wire format change and shared helper (the data layer)

### `apps/gridbot/src/gridbot/executor.py`

`IntentExecutor.execute_place` (around lines 167–189): in the live
branch (not shadow mode), build a unique `orderLinkId` by appending
`-{int(datetime.now(UTC).timestamp() * 1000)}` to
`intent.client_order_id`, then pass that to
`BybitRestClient.place_order(order_link_id=...)`. The suffix is
purely numeric, monotonically increasing across milliseconds, and the
deterministic prefix is guaranteed to never contain `-` because
`hashlib.sha256().hexdigest()[:16]` returns only `0-9a-f`. A HOTFIX
comment block above the construction documents the rationale and the
2026-05-08 incident.

Tests: `apps/gridbot/tests/test_executor.py`. The three existing tests
that asserted `order_link_id == intent.client_order_id` verbatim
(`test_place_order_success`, `test_place_order_short_direction`,
`test_execute_place_passes_orderLinkId_from_client_order_id`) must
assert prefix-and-suffix shape instead: kwargs match expected on all
fields except `order_link_id`, which has shape `f"{prefix}-{digits}"`
where the suffix is non-empty and `isdigit()`.

### `packages/gridcore/src/gridcore/intents.py`

Add a module-level function `extract_client_order_prefix(order_link_id:
str | None) -> str | None` near `PlaceLimitIntent.create`. Algorithm:

1. If `order_link_id is None` → return `None`.
2. Otherwise `prefix, _, _ = order_link_id.partition("-")`.
3. Return `prefix or None` — collapses the empty-string case so callers
   using `prefix or fallback_id` cleanly fall back.

Backward compat: a value with no `-` partitions to `(value, "", "")`,
so the helper returns the value unchanged. This covers pre-hotfix
recorder/event-saver rows whose `orderLinkId` had no suffix.

### `packages/gridcore/src/gridcore/__init__.py`

Re-export `extract_client_order_prefix` from the package root next to
`PlaceLimitIntent` and `CancelIntent`. Add it to `__all__`.

### `packages/gridcore/tests/test_intents.py` (new file)

A single parametrized test exercising the contract: `None` → `None`,
empty string → `None`, no-hyphen input → unchanged, hyphenated input →
prefix only, edge cases (leading hyphen, trailing hyphen, multi-hyphen).

## Phase 2A — gridbot runner read-side normalization

### `apps/gridbot/src/gridbot/runner.py`

Import `extract_client_order_prefix` alongside other gridcore imports.
Two call sites:

1. `_find_tracked_order(order_link_id, order_id)` (around lines
   693–707). Currently does `if order_link_id and order_link_id in
   self._tracked_orders` for the fast path, then linear-scans by
   `order_id`. After the wire-format change, `order_link_id` from
   inbound REST/WS events is suffixed; the dict is keyed by the
   unsuffixed `intent.client_order_id`. Fix: extract prefix first, do
   the dict lookup on the prefix; keep the linear-scan order_id
   fallback unchanged.

2. `inject_open_orders(orders)` (around lines 814–907). Currently does
   `client_id = order_link_id or order_id` where `order_link_id` is the
   raw exchange-side string. After the wire-format change this would
   key injected entries by the suffixed string while self-placed
   entries (via `_execute_place_intent`) key by the unsuffixed prefix —
   on a bot restart the same logical order would end up tracked under
   two different keys. Fix: extract prefix first; `client_id =
   link_prefix or order_id`. The existing collision guard on `client_id
   in self._tracked_orders` then correctly catches restart-collision
   cases.

The two other `_find_tracked_order` call sites in `runner.py` (the
reconciler diff loop) automatically benefit; no callsite-side change.

### `apps/gridbot/tests/test_runner.py`

Six new tests in the existing `TestStrategyRunnerOrderTracking` and
`TestStrategyRunnerOrderUpdate` classes:

- Suffix-stripping fast-path lookup (force `order_id=None` to bypass
  the linear scan).
- Pre-hotfix unsuffixed lookup (backward compat).
- `inject_open_orders` with a suffixed `orderLinkId` keys by prefix.
- Restart-collision: place an intent, then inject the same logical
  order under its suffixed `orderLinkId`; assert no duplicate entries.
- Empty-string `orderLinkId` collapses to `order_id` fallback (codifies
  the contract that `""` and missing key are treated identically).
- End-to-end through `on_order_update`: simulate a `Filled` event with
  a suffixed `orderLinkId`, assert the tracked order's status flips.

### `apps/gridbot/src/gridbot/reconciler.py`, `config.py` (stale-comment cleanup)

Two pre-existing comment blocks claimed "we no longer send orderLinkId
to Bybit" — stale since feature 0029 wired it through. Update both to
reflect that orderLinkId is sent (with suffix); update
`validate_no_shared_symbol`'s docstring and ValueError text to ground
the (account, symbol)-uniqueness invariant in the deterministic-prefix
collision (two strategies on the same pair compute the same SHA prefix
for the same logical order), not in orderLinkId absence. Validator
logic itself is unchanged.

## Phase 2B — comparator loader read-side normalization

### `apps/comparator/src/comparator/loader.py`

`LiveTradeLoader.load` (lines around 116–122): grouping live executions
into trades currently keys by `client_id = ex.order_link_id or
ex.order_id`. Post-hotfix, `private_executions.order_link_id` carries
the suffix verbatim; replay/backtest emit the deterministic prefix as
`client_order_id`. Without normalization, comparator parity reports
~100% divergence on every post-hotfix execution.

Fix: import `extract_client_order_prefix` from `gridcore.intents`;
inside the grouping loop, compute `link_prefix =
extract_client_order_prefix(ex.order_link_id)`; use `client_id =
link_prefix or ex.order_id`. The fallback log on `link_prefix is None`
preserves the pre-existing fallback semantics for NULL `order_link_id`.

### `apps/comparator/tests/test_loader.py`

Five new tests in `TestLiveTradeLoader`:

- Pre-hotfix unsuffixed `order_link_id` is preserved unchanged.
- Single suffixed `order_link_id` strips to prefix.
- NULL `order_link_id` triggers fallback log (asserted via `caplog`).
- Two suffixed rows with the same prefix and different `order_id`
  produce two trades with the same `client_order_id` and sequential
  `occurrence` indices (the ID-reuse path under the hotfix).
- Two suffixed rows with the same suffix and same `order_id` aggregate
  into one trade (partial-fill path under the hotfix).

One pre-existing test fixture (`test_loader_groups_by_order_link_id_when_present`)
must change `"LX-1"` → `"LX1"` because the hyphen was incidental and
would now be stripped. The new behaviour is covered explicitly by
`test_suffixed_order_link_id_strips_to_prefix`.

### `docs/features/0029_PLAN.md`

Append a one-paragraph contract note to the comparator-matching
section: post-2026-05-08 `orderLinkId` carries `-{millis}` on the wire
and in `private_executions`; `LiveTradeLoader` strips the first `-`
and everything after it via `extract_client_order_prefix`; replay
emits the deterministic prefix unchanged; both sides rejoin on the
prefix.

## Phase 2C — replay seed loader read-side normalization

### `apps/replay/src/replay/snapshot_loader.py`

`load_active_orders` (around line 390): currently builds
`client_id = row.order_link_id or row.order_id` where `row` is an
`Order` table row. After the hotfix, `Order.order_link_id` carries the
suffixed wire form. The seed `client_id` is later compared against
intents that replay's engine produces (which use the deterministic
prefix from `PlaceLimitIntent.create`); without normalization, seeded
replays of post-hotfix snapshots misalign on every active order.

Fix: import `extract_client_order_prefix` from `gridcore.intents`;
change line 390 to `client_id = extract_client_order_prefix(row.order_link_id)
or row.order_id`. The triple-comment above the assignment can be
shortened to point at the helper.

### `apps/replay/tests/test_snapshot_loader.py`

One new test `test_strips_orderlinkid_suffix_post_hotfix` in
`TestLoadActiveOrders`: insert an `Order` row with a suffixed
`order_link_id`; assert the resulting `ActiveOrderSeed.client_id` is
the prefix only.

One pre-existing fixture (`test_happy_path_with_order_link_id_and_fallback`)
must change `"LX-1"` → `"LX1"` for the same reason as the comparator
fixture above.
