# Feature 0032 â€” Preserve `orderLinkId` across place retries

## Context

After the 2026-05-08 hotfix (PR #69, RULES.md item #7), `IntentExecutor.execute_place()` appends a fresh millisecond suffix to `orderLinkId` on **every** call:

```
order_link_id = f"{intent.client_order_id}-{int(now_ms)}"
```

This avoided Bybit `ErrCode 110072` ("OrderLinkedID is duplicate") on the cached deterministic prefix, but it broke retry idempotency: if Bybit accepts an order and the HTTP response then times out (or any other ambiguous failure), the bot reattempts with a **new** suffix and Bybit accepts a **second real order** at the same grid level (different wire ids â†’ different orders to the exchange).

This feature moves wire-id assignment to the placement lifecycle (one id per logical placement, reused across retries) so that an ambiguous-failure reattempt is rejected by Bybit's id cache as intended. It supersedes the duplicate Cursor-bot PRs **#71** and **#72**.

## Goals

1. **Lifecycle invariant:** the same assigned wire `orderLinkId` is carried by the placement intent used for tracking and retries, instead of being regenerated on each `execute_place` call. This wire id is reused on:
   - immediate runner reattempts after a tracked-order failure,
   - `RetryQueue.process_due()` retries, and
   - fresh engine re-emissions of the same logical placement (same `client_order_id`) after a failed tracked placement.
2. `client_order_id` (deterministic 16-hex prefix) remains the strategy-level identity (dict keys, comparator/replay joins). It is NOT mutated by this feature.
3. Executor-side suffix generation kept as a fallback for direct callers that did not pre-assign the wire id (back-compat).
4. Order-sync (`Runner.inject_open_orders`) can upgrade a pending/failed tracked order with the same prefix when Bybit later reports the accepted order in open orders â€” closing the ambiguous-failure window â€” AND cancels any pending retry for that intent.
5. `OrderResult` exposes the wire `order_link_id` for observability (logs, forensics), in both live and shadow paths.

## Non-goals

- No changes to the deterministic-prefix algorithm or the wire format `{16-hex}-{ms}`.
- No changes to the comparator/replay strip-suffix-on-read invariants in RULES.md item #7.
- Not addressing other retry-related concerns (backoff curve, max attempts, etc.).

## Files to change

### Data layer

- `packages/gridcore/src/gridcore/intents.py`
  - `PlaceLimitIntent` is `@dataclass(frozen=True)` â€” DO NOT unfreeze.
  - Add field:
    `order_link_id: str | None = field(default=None, compare=False)`
    `compare=False` is mandatory: this wire id must NOT participate in dataclass equality, otherwise `RetryQueue.remove()` and any set-based tracking break (two intents that are identical placements but acquired different wire ids would compare unequal).
  - Document field in dataclass docstring: `None` until the runner assigns it for this placement lifecycle via `dataclasses.replace(intent, order_link_id=...)`; executor falls back to inline generation if still `None`.
  - **Do not** add it to `_IDENTITY_PARAMS` â€” it must not affect `client_order_id` derivation.

- `apps/gridbot/src/gridbot/order_link_id.py` (NEW small module)
  - `make_order_link_id(client_order_id: str, *, now_ms: Callable[[], int] = â€¦) -> str`
    Returns `f"{client_order_id}-{now_ms()}"`.
  - Single source of truth used by both runner (primary path) and executor (fallback). Replaces the inline string built in `execute_place()`. (PR #71/#72 introduced a similar helper named `make_unique_order_link_id`; we reuse the spirit but keep one helper, not two.)
  - Time source pluggable for tests.

- `apps/gridbot/src/gridbot/executor.py`

  - `IntentExecutor.execute_place()`:

    - Else â†’ call `make_order_link_id(intent.client_order_id)` (fallback for direct callers that bypassed runner assignment).
    - Populate `OrderResult.order_link_id` with whatever was actually sent.

### Runner integration

- `apps/gridbot/src/gridbot/runner.py`
  - New helper (private), pure: returns a NEW intent via `dataclasses.replace`, does not mutate (frozen). Accepts an optional reusable wire id so re-emissions of the same logical placement reuse the previous failed wire id instead of minting a fresh suffix:
    ```
    def _assign_wire_link_id(
        intent: PlaceLimitIntent,
        *,
        existing_order_link_id: str | None = None,
    ) -> PlaceLimitIntent:
        if intent.order_link_id is not None:
            return intent
        wire_id = existing_order_link_id or make_order_link_id(intent.client_order_id)
        return dataclasses.replace(intent, order_link_id=wire_id)
    ```
  - In `_execute_place_intent(intent)`:
    1. **Reuse-on-reattempt lookup (new):** look up `previous = self._tracked_orders.get(intent.client_order_id)`. If `previous` exists with `previous.status == "failed"` and `previous.intent is not None` and `previous.intent.order_link_id is not None`, treat that wire id as reusable.
    2. `assigned = self._assign_wire_link_id(intent, existing_order_link_id=<reusable or None>)`.
    3. Use `assigned` everywhere downstream â€” pass to `executor.execute_place(assigned)`, store in `TrackedOrder`, hand to `_on_intent_failed(assigned, error)` (which routes into `RetryQueue.add`).
    The original engine-emitted `intent` stays untouched (frozen guarantees this; tests will assert).
  - **Lifecycle invariant (precise wording):** the wire `order_link_id` of a logical placement (keyed by `client_order_id`) is minted exactly once and is reused by every reattempt â€” whether the reattempt is a runner-immediate retry, a `RetryQueue.process_due()` retry, OR a fresh engine re-emission of the same intent after the previous attempt failed. After `replace()`, intent object identity changes; what matters is the wire id traveling with whatever intent is active for that `client_order_id`.
  - `inject_open_orders()` / `_find_tracked_order()`:
    - Already strip suffix via `extract_client_order_prefix()` (RULES.md #7).
    - **Upgrade path** when an open order's prefix matches a tracked order in `pending`/`failed` state:
      - If `existing.intent is None` (defensive â€” should not happen for failed/pending placements created by runner): log WARNING `"open-order upgrade skipped: tracked order has no intent (prefix=â€¦)"` and bail; do NOT attempt to reconstruct an intent from the open-order payload (it lacks `direction`, `grid_level`, etc.).
      - Otherwise: `upgraded_intent = dataclasses.replace(existing.intent, order_link_id=exchange_link_id)` â€” preserves original placement context (qty, grid_level, direction); we change ONLY the wire id to match what the exchange reports.
      - Move tracked order to `placed` with `order_id = exchange_order_id` and `intent = upgraded_intent`.
      - Invoke retry-cancellation callback if present: `removed = self._on_retry_cancel_for_prefix(prefix) if self._on_retry_cancel_for_prefix else 0` (see Orchestrator section).
      - Log at INFO: `tracked order upgraded from <prev_state> via reconcile (prefix=<â€¦>, order_id=<â€¦>, link_id=<â€¦>, retry_cancelled=<removed>)`.
    - DO NOT replace `existing.intent` with a freshly reconstructed intent built from the open-order payload â€” we keep the original placement context and only patch the wire id.

### Retry queue

- `apps/gridbot/src/gridbot/retry_queue.py`
  - **Storage:** `RetryItem` keeps holding whatever intent was passed to `add()`. Since runner now passes the assigned intent (the result of `replace`), the wire id automatically rides along into the queue.
  - **New method `cancel_for_prefix(prefix: str) -> int`:**
    - Match policy (in this order):
      1. Primary: `intent.client_order_id == prefix` â€” this is the strategy identity and is always set; simplest and most robust match.
      2. Fallback (defensive, for items that somehow got assigned a wire id but lost client_order_id correspondence): `extract_client_order_prefix(intent.order_link_id) == prefix`.
    - Removes any matching `PlaceLimitIntent` items. Ignores `CancelIntent` items (no concept of placement prefix).
    - Returns count of removed items (for logging).
    - Called by orchestrator on the `on_retry_cancel_for_prefix` callback the runner invokes (see Orchestrator section).
    - Idempotent and safe when nothing matches.
  - **Defensive WARN** in `add()`:
    - Only for `PlaceLimitIntent` items (CancelIntent has no `order_link_id` field â€” do not warn).
    - If `intent.order_link_id is None`, log WARNING `"PlaceLimitIntent enqueued without assigned order_link_id (prefix=â€¦); executor will fall back to fresh suffix"`. Indicates a direct-caller path bypassed runner assignment.

### Orchestrator wiring

- The orchestrator (or wherever `RetryQueue` and `StrategyRunner` are constructed and connected today) gains one extra wire alongside the existing `on_intent_failed` callback:
  - **`on_retry_cancel_for_prefix: Callable[[str], int] | None = None`** â€” runner-side callable that the orchestrator implements as `lambda prefix: retry_queue.cancel_for_prefix(prefix)`.
  - **Default is `None`** so existing test fixtures that construct `StrategyRunner(...)` with positional/keyword args do NOT need to be updated. Runner internally treats `None` as a no-op (or stores `lambda _prefix: 0`); reconcile-upgrade path simply gets a `removed_count = 0` and proceeds.
  - Runner stores this callable in `__init__` and calls it from the `inject_open_orders` upgrade path. This keeps the runner decoupled from the `RetryQueue` class.
- Concrete file: whichever module currently passes `on_intent_failed` to `StrategyRunner.__init__` (`apps/gridbot/src/gridbot/orchestrator.py` or equivalent â€” implementation step will locate exact site).

## Algorithm â€” placement lifecycle (after change)

1. Engine emits a `PlaceLimitIntent` with deterministic `client_order_id` (16-hex prefix) and `order_link_id is None` (frozen).
2. `StrategyRunner._execute_place_intent(intent)`:
   - **Reuse lookup:** `previous = self._tracked_orders.get(intent.client_order_id)`. If `previous` is in `failed` state and has a non-`None` `previous.intent.order_link_id`, that wire id is reusable.
   - `assigned = self._assign_wire_link_id(intent, existing_order_link_id=<reusable or None>)`. If reusable was provided â†’ no fresh suffix; otherwise â†’ `make_order_link_id(client_order_id)`.
   - `result = executor.execute_place(assigned)`.
   - `assigned` is the intent stored in `TrackedOrder` and routed through any failure handling.
3. `executor.execute_place(assigned)`:
   - Uses `assigned.order_link_id` as wire id (or, if still `None` for direct callers, calls `make_order_link_id(...)` as fallback).
   - Returns `OrderResult(success, order_id, error, timestamp, order_link_id=<wire_id_used>)`. Same in shadow mode.
4. **Success branch:** tracked order moves to `placed`, prefixâ†’tracked dict updated.
5. **Failure branch â€” three reattempt paths, all carry the same wire id:**
   - Immediate runner reattempt (if applicable): runner reuses the current assigned intent or, if it must reassign (e.g., refreshed from `_tracked_orders`), step 2's reuse lookup pulls the same wire id from the failed tracked state.
   - Engine re-emission on a later tick: the engine produces a fresh frozen `PlaceLimitIntent` with `order_link_id is None` and the same `client_order_id`. Step 2's reuse lookup finds the previous failed tracked intent and assigns the **same wire id** to the new `assigned` intent. No new suffix is minted.
   - `RetryQueue.process_due()`: queue item already holds the assigned intent (with wire id) put there by `retry_queue.add(assigned, error)` â†’ executor sees the same wire id.
   - In all three: Bybit's id cache rejects the duplicate if the original was actually accepted.
6. **Reconcile-upgrade path** (covers ambiguous-failure that was actually accepted):
   - Reconciler fetches open orders from REST.
   - `runner.inject_open_orders(open_orders)`:
     - For each open order, strip suffix â†’ `prefix`.
     - If `prefix` matches a tracked order in `pending` / `failed`:
       - If `existing.intent is None` â†’ WARN + skip upgrade.
       - Otherwise: `upgraded_intent = dataclasses.replace(existing.intent, order_link_id=exchange_link_id)` (preserves original placement context, patches only wire id).
       - Move tracked order to `placed` with `exchange_order_id` and `upgraded_intent`.
       - If `self._on_retry_cancel_for_prefix` is set, `removed_count = self._on_retry_cancel_for_prefix(prefix)`; otherwise `removed_count = 0`. Explicit cancellation when wired; no-op when callback is absent (e.g., bare-bones tests).
       - INFO log including upgraded fields and `retry_cancelled=<removed_count>`.
     - Otherwise, behave as today.
