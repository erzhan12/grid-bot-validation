# Feature 0044 ‚Äî State-consistency filter in PositionComparator

## Goal

Feature 0043's Phase 4 validation on recorder run `e0723754-‚Ä¶` was
dominated by 2 outlier pairs (live=14.5/17.8 USDT vs bt=0) caused by
manual long-side interventions on the live account. The backtest replay
‚Äî which only knows about grid orders ‚Äî could not reproduce those
manual fills, so `bt.size = 3.1` while `live.size = 4.8` at the same
exchange_ts. The liq formula then operates on legitimately different
states; the resulting "delta" is a state-divergence artifact, not a
formula bug.

Feature 0044 makes the comparator robust against this class of noise so
any future recorder run produces a meaningful
`liq_price_max_abs_delta` metric without requiring the operator to
manually curate "clean" recording windows.

## Confirmed Context

- `PositionComparator.pair_and_compare`
  (`apps/comparator/src/comparator/position_metrics.py:127`) emits one
  `PositionComparisonPair` per backtest snapshot. Pairs are
  `(live, bt)` matched within ¬±`pair_tolerance_s` (default 5s).
- `fold_metrics_into` (line 177) walks matched pairs and produces
  mean/max abs deltas for `liq_price`, `position_im`, `position_mm`,
  `unrealised_pnl`. Unmatched-bt pairs are counted separately
  (`position_pairs_unmatched_bt`).
- There is **no current guard** that the paired snapshots represent
  the same position state. As long as the timestamps are within
  tolerance, the pair is folded into the metrics regardless of how
  far apart `size` / `entry_price` have drifted between live and bt.
- `ValidationMetrics`
  (`apps/comparator/src/comparator/metrics.py:32`) holds the 12+
  position telemetry fields produced by `fold_metrics_into`.

## Non-Goals

- Do not change pair timestamp tolerance behavior (still ¬±5s).
- Do not change the existing unmatched-bt path (when no live row falls
  in the time window).
- Do not modify ``ValidationMetrics`` semantics for matched pairs that
  are state-consistent ‚Äî the new bucket is purely additive.
- Do not commit, push, or open a PR without explicit operator
  instruction.

## Implementation Plan

1. **New tolerance pair on `PositionComparator.__init__`**:
   ```
   size_tolerance: Decimal = Decimal("0.001")    # absolute units
   entry_rel_tolerance: Decimal = Decimal("0.001")  # 0.1% of entry
   ```
   Defaults sized to cover float / tier-rounding noise (LTCUSDT
   qty_step = 0.1 ‚Üí 0.001 is well below one step; entry drift inside
   a fill quantum is ‚â§0.05%).

2. **Inside `_pair_side`**, after the timestamp-tolerance match and
   before `_build_pair`, run a state-consistency check:
   - `live.size > 0` AND `bt.size > 0` (skip both-zero ‚Äî they always
     trivially match).
   - `|live.size - bt.size| > size_tolerance` ‚Üí mark **state_diverged**.
   - `live.entry_price > 0` AND
     `|live.entry_price - bt.entry_price| / live.entry_price > entry_rel_tolerance`
     ‚Üí mark **state_diverged**.
   State-diverged pairs are emitted with a fresh sentinel field on
   `PositionComparisonPair` (`state_diverged: bool = False`); the live
   row is still claimed (does not leak into next bt's match).

3. **New field on `ValidationMetrics`**:
   `position_pairs_state_diverged: int = 0`. Documented as "pairs
   matched by exchange_ts but where backtest state has drifted from
   live (e.g. operator manual fills not in the grid replay)".

4. **`fold_metrics_into`**:
   - Treat `state_diverged` pairs as "matched but excluded" ‚Äî they
     do NOT enter the mean/max abs delta calculations for
     `liq_price`, `position_im`, `position_mm`, `unrealised_pnl`,
     `cum_realised_pnl`.
   - Increment `position_pairs_state_diverged`.
   - Keep `position_pairs_compared` as the count of state-consistent
     pairs (so the existing "Pairs compared" header in the reporter
     matches the actual aggregate denominator).

5. **`ComparatorReporter`** (apps/comparator/src/comparator/reporter.py):
   - Add a line in the position-telemetry summary block:
     `State diverged: N` (between `Unmatched (bt)` and `Missing telemetry`).
   - In `position_comparison.csv`, add a `state_diverged` column so
     downstream readers can grep / filter.

6. **Tests** (`apps/comparator/tests/test_position_metrics.py`):
   - Happy path: pair with matching state ‚Üí counted.
   - Size diverged: live.size=4.8, bt.size=3.1 ‚Üí state_diverged=True,
     not counted in liq aggregate.
   - Entry diverged: live.entry=57.45, bt.entry=58.00 ‚Üí diverged.
   - Boundary: |Œîsize| == size_tolerance ‚Üí counted (inclusive).
   - Both zero size (closed positions): not flagged as diverged.
   - One zero, one non-zero size: state diverged (size delta > tol).

7. **Re-run Phase 4 validation** on `data/recorder_ltcusdt_phase4.db`
   run_id `e0723754-‚Ä¶` and confirm:
   - `position_pairs_state_diverged` >= 2 (the manual-intervention
     pairs).
   - `liq_price_max_abs_delta` drops to ‚â§ 1 USDT (most likely 0, since
     all remaining matched pairs were already qualitative no-liq).

8. **Documentation**:
   - `RULES.md` rule 23 ‚Äî pair-and-compare semantics with the new
     state-consistency bucket.
   - Cross-link from `docs/features/0043_PLAN.md` "Known Limitations"
     section (the manual-intervention artifact is now filterable).
