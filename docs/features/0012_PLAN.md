# 0012: Tiered MMR from Risk Limits Cache

Replace the flat `maintenance_margin_rate` (0.5%) used for backtest liquidation price estimation with Bybit's actual tiered maintenance-margin schedule loaded from `conf/risk_limits_cache.json`. Currently the backtest uses a single MMR for all position sizes, which underestimates liq risk for large positions and overestimates it for small ones. Tiered MMR makes the risk multiplier calculations match live behavior more closely.

## Problem: Flat vs Tiered

Current formula (flat):
- Long: `liq = entry * (1 - 1/leverage + mmr)` where `mmr = 0.005` always
- Short: `liq = entry * (1 + 1/leverage - mmr)`

This ignores that Bybit uses **tiered** MMR — larger positions have higher MMR (0.5% → 60%), which produces tighter liquidation prices. The risk multiplier system checks `liq_price / current_price`, so inaccurate liq prices produce wrong multipliers.

Tiered formula:
- Look up tier by `position_value`: find first tier where `max_value >= position_value`
- `mm_amount = position_value * mmr_rate - deduction`
- `mmr_effective = mm_amount / position_value`
- Use `mmr_effective` in the existing liq price formula

## Algorithm

### On runner init:

1. Load tiers for the strategy's symbol:
   - Try `conf/risk_limits_cache.json` (symbol key)
   - Fall back to hardcoded `MM_TIERS` defaults in `gridcore.pnl`
   - Store as `self._mm_tiers: MMTiers`

### When estimating liquidation price (`_estimate_liquidation_price`):

2. Accept `position_value` as additional parameter
3. Call `calc_maintenance_margin(position_value, symbol, tiers=self._mm_tiers)` → `(mm_amount, mmr_rate)`
4. Use the returned `mmr_rate` (tier-specific) instead of flat `self._mmr`
5. Apply same formula: `liq = entry * (1 - 1/leverage + mmr_rate)` for long

### Caller update (`_build_position_state`):

6. Pass `position_value` to `_estimate_liquidation_price()` (already computed on line 316)

## Design Decisions

1. **Cache file only (no API fetch at init)**: The backtest runner loads tiers from the already-committed `conf/risk_limits_cache.json`. No network calls during backtest. The `RiskLimitProvider` with API fetch + locking infrastructure from `feature/dynamic-risk-limit-tiers` is deferred — it's needed for the live bot and cache refresh, not for backtest init.

2. **Keep `maintenance_margin_rate` as fallback**: If no tiers are available for the symbol, fall back to the flat rate. This preserves backward compatibility.

3. **Port tier logic to `gridcore.pnl`**: The tier lookup, `MMTiers` type, and `calc_maintenance_margin()` belong in the shared `gridcore` package since both live bot and backtest will use them.

4. **Don't port the full `RiskLimitProvider` yet**: The cache locking, validation, and API-fetch modules (`risk_limit_info.py`, `cache_lock.py`, `cache_validation.py`, `tier_serialization.py`) are substantial. They belong in a follow-up feature (0013) for live cache refresh. For 0012, a simple JSON file read at init is sufficient.
